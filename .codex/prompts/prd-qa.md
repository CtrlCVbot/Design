# /prd-qa - PRD 기능 QA → a사 디자인 QA → 전용 에이전트 4개(a·b·t사 + t사 디자이너) 제안

PRD 기능 요구사항을 `/qa` 와 동일한 브라우저로 **잔존 0이 될 때까지** 검증·수정하고, a사 스태프 디자이너+FE 엔지니어 관점으로 화면 반영을 2회차 검증·수정한 뒤, **전용 에이전트 4개(a사
디자인·b사 관점·t사 돌파·t사 디자인)를 병렬 호출**해 각 에이전트가 **Notion 페이지의 자기 블록만** 작성·갱신합니다.

> a사=업계 최고 디자인, b사=목표 역산·선제 제품 사고, t사=불가능을 되게 하는 돌파력, t사 디자이너=극단적 단순화·감정 디테일의 UX/UI(a사와 비겹침). (회사 실명은 출력물에 쓰지 않는다.)
> 브라우저 QA 메커니즘(로그인/포트/스냅샷 전용/스크린샷 금지)은 `/qa` 와 **100% 동일**. 단일 출처: `.claude/commands/qa.md` +
`.claude/skills/qa-browser.md`.

---

## 사용법

```
/prd-qa {PRD 경로 또는 링크}                    # PRD에서 대상 화면 추출
/prd-qa {PRD 경로 또는 링크} {대상 화면 URL}     # 대상 화면 명시
```

- **첫 번째 인자(필수)**: PRD 문서 — 로컬 `.md` 경로 또는 Notion 링크. **기능 요구사항의 출처**.
- **두 번째 인자(선택)**: QA 대상 화면 URL. 생략 시 PRD에서 추출하고, 불명확하면 사용자에게 확인.

---

## 적용 원칙

- **검사 방법은 `/qa` 와 동일**, **통과 기준만 다름**: 무작위 탐색적 엣지케이스가 아니라 **"PRD 기능 요구사항 충족 여부"** 가 합격선.
- 각 라운드는 **검사 → 발견 → 즉시 수정**. 수정은 TDD 워크플로우(`.claude/skills/tdd-workflow.md`)를 따른다.
- **기능 QA는 잔존 0이 될 때까지 반복**한다 — 백엔드/도메인을 건드려야 하는 항목도 끝까지 수정. 코드로 해결 불가한 항목(제품 결정·외부 의존)만 사용자에게 **에스컬레이션**.
- 디자인 QA(화면 반영)는 2회차로 고정.
- **관점 제안(a/b/t사)은 한 블록에서 직접 쓰지 않는다.** 전용 에이전트 3개를 병렬 호출하고 각자 Notion 블록을 갱신한다.

## 상세 단계

### STAGE 0: 준비

1. **PRD 로드**: 로컬 경로면 Read, Notion 링크면 `/qa` 와 동일한 MCP 브라우저로 열어 파악.
2. **기능 요구사항 체크리스트화**: 검증 가능한 요구사항을 항목으로 분해, ID 부여(REQ-1, REQ-2 …), 모호하면 "검증 기준" 한 줄 명시.
3. **대상 화면 확정**: 두 번째 인자 우선, 없으면 PRD에서 추출, 불명확하면 사용자에게 확인.
4. **로그인**: `.claude/commands/qa.md` 의 로그인 절차(포트/계정)를 **그대로** 수행.

**완료 조건**: 요구사항 체크리스트 확정 + 대상 화면 로그인 완료.

### STAGE 1: 기능 QA (PRD 기능 요구사항만) — 잔존 0까지

`.claude/skills/qa-browser.md` 의 수행 규칙(속도·스냅샷 전용·스크린샷 금지)을 따르되, **합격 판정은 REQ-* 충족 여부** 로만.

**반복 루프 (위반이 0이 될 때까지)**

1. REQ-1..N을 브라우저에서 직접 수행하며 충족 판정.
2. 미충족 발견 시 어떤 REQ 위반인지 기록.
3. **즉시 수정**(TDD). FE 단독으로 안 되면 도메인·쿼리·타입·스토어까지 **끝까지** 고친다.
4. 재검사 → 위반 남으면 1로. **모든 REQ 충족 = 잔존 0** 까지 반복.

**에스컬레이션**: 코드로 해결 불가한 항목(제품 결정·외부 의존)만 사용자에게 보고하고 멈춘다.

**완료 조건**: 전 REQ 충족 + 관련 테스트 green + tsc 0.

### STAGE 2: a사 디자인 QA (화면 반영) — 2회차

**역할**: 너는 애플(a사)에서 가장 유망한 스태프 디자이너 겸 FE 엔지니어다. "디자인은 어떻게 작동하느냐다." PRD 의도가 화면에 디테일까지 반영됐는지 본다(기능 동작 아님).

**검사 렌즈 (필수 8)**: 시각적 위계 / 여백·리듬(8pt) / 상태 디자인(empty·loading·error·disabled·hover·focus) / 모션(200~300ms) / 일관성(토큰·재사용) /
접근성(WCAG AA·포커스·터치 ≥44px) / 카피·마이크로카피 / 디테일(1px·잘림·반응형).

**1차 검사 → 수정 → 2차 검사 → 수정** (메인 에이전트가 직접 수정). 스타일은 `.claude/skills/tailwind.md`, 컴포넌트는 `.claude/skills/shadcn-radix.md`.

### STAGE 3: Notion 페이지 구조 준비

관점 제안을 받을 Notion 페이지를 미리 구조화한다 (메인 에이전트가 1회). 스크립트: `scripts/notion-prd-qa-report.mjs` (`.env` 의 `NOTION_API_KEY`).

1. **페이지 확보**: PRD가 Screening DB 페이지면 그 `--page`. 아니면 `--title "{기능명}"` 으로 검색 → 0건이면 새 페이지(상태=QA) 생성. 페이지 ID 확보.
2. **헤더 작성**: 제목 heading + `검증` 스탬프(전 요구사항 충족(잔존 0)·테스트·tsc·콘솔·라이브) 코드블록.
3. **빈 섹션 heading 4개**를 헤더 바로 아래에 **순서대로** 만든다(코드블록 비움):
   `디자인제안(a)`, `디자인 제안(t)`, `관점2(b)`, `관점(t)`.
    - 디자인 두 블록(a·t사 디자이너)을 **나란히** 둔다 — `디자인 제안(t)` 는 `디자인제안(a)` 바로 뒤.
    - 이미 있으면 재사용. (사용자가 페이지에 적어둔 다른 heading 은 건드리지 않는다.)

### STAGE 4: 관점별 제안 — 전용 에이전트 4개 병렬

**Agent 도구로 전용 에이전트 4개를 한 메시지에서 병렬 호출**한다. 각 에이전트가 자기 섹션 블록만 갱신한다.

| 에이전트    | 페르소나                                                    | 산출 → Notion 섹션            | 분량 |
|---------|---------------------------------------------------------|---------------------------|----|
| a사      | 애플내의 최고 스태프 디자이너+FE                                     | `디자인제안(a)` (ROI 내림차순)     | ≥6 |
| b사      | `bolta프롬프트용/` 하위디렉토리 전부 읽고 사고방식                         | `관점2(b)` (현황+제안)          | ≥6 |
| t사      | 토스에 재직중인 불가능을 되게 하는 최고 기술책임자                            | `관점(t)` (돌파안)             | ≥5 |
| t사 디자이너 | 토스에서 재직중인, 없으면 회사가 안 돌아가는 UX/UI 최고 디자이너(극단적 단순화·감정 디테일) | `디자인 제안(t)` (**임팩트 높은순**) | ≥4 |

> **t사 디자이너는 a사와 겹치면 안 된다**: 먼저 `section-a.md`(a사 디자인 리뷰)를 읽고, 겹치는 주제는 제외한 새로운 고임팩트 각도(단순화·금액 타이포·온보딩·완료감·카피·빈 상태·진척감 등)
> 만 낸다.

각 에이전트 프롬프트에 공통으로 준다: **프로젝트 맥락(아래)**, **기능/대상 화면 요약**, **STAGE 1에서 고친 내용 요약**, **핵심 컴포넌트 경로**(b사는 `bolta프롬프트용/` 도), **Notion 페이지 ID**, **자기
섹션 이름**, 페르소나·출력 형식, **회사 실명 금지(a/b/t사 · 'HIG' 금지 · UI '토스트'는 '알림')**, **용어 규칙(아래)**.

**프로젝트 맥락 (각 에이전트에 필수 전달)**

- 이 제품은 **B2B 물류 TMS**(운송 관리 시스템) — 주선사·화주·차주가 화물 배차/운송/정산을 처리하는 업무 시스템이다.
- 지금은 **막 시장에 진입해 영업 중인 단계**. 도입 결정을 좌우하는 첫인상·신뢰·완성도가 특히 중요하고, 초기 고객은 운영 규모(데이터)가 작을 수 있음을 전제로 제안한다.

**용어 규칙 (독자 = 기획자/PM — 필수)**

- 본문은 **기획자가 읽는다**는 전제로 쓴다 — 쉬운 한국어, 화면·사용자·업무 관점.
- 함수명·변수명·CSS 클래스·파일 경로 같은 **코드 식별자를 본문(현황·문제·제안·진단)에 노출하지 않는다.** 꼭 필요하면 각 항목 맨 끝 `적용:`(개발 참고) 줄에만 짧게.
- 영어 약어·전문용어는 풀어 쓴다. "사용자가 화면에서 무엇을 겪고, 무엇이 어떻게 달라지는지"로 설명한다. (예: `sticky` 미적용 → "긴 목록에서 합계 바가 화면에 안 붙고 밀려 내려감")

각 에이전트가 하는 일:

1. 필요한 파일(컴포넌트/`bolta프롬프트용/`)을 직접 읽어 분석한다(코드 근거로 날카롭게).
2. 제안을 `dev/active/prd-qa-{대상}/section-{a|b|t|td}.md` 에 저장(**섹션 제목 줄은 생략** — Notion heading 이 이미 존재).
3. 실행: `node scripts/notion-prd-qa-report.mjs {그 파일} --page {ID} --section "{섹션 이름}"` → 자기 블록만 in-place 갱신.
4. 작성한 제안 전체를 반환.

> `--section` 은 그 섹션 heading 아래 코드블록만 갱신/생성하므로 **네 에이전트가 병렬로 서로 다른 블록을 안전하게** 쓴다. 재실행 시 각자 자기 블록을 in-place 교체.

**각 관점 — 두 갈래로 나눠 쓴다 (필수)**

각 에이전트는 자기 섹션을 두 부분으로 명확히 나눠 작성한다:

1. **[추가 제안]** — 아직 화면에 없는 것. 아래 각 페르소나 형식대로 (이미 구현된 건 여기선 빼고 새 것만).
2. **[기존 의심]** — 이미 만들어진 것을 비판적으로 다시 본다. **"이게 정말 사용자의 문제를 해결하는 좋은 관점·구현인가?"** 를 의심한다. 전제·방향·우선순위·UX 패턴이 어긋났다고 보면 근거와 함께 짚고 더 나은 대안을 낸다. 이미 구현됐다는 이유로 그냥 넘기지 말 것 — 잘된 점은 한 줄로만 인정하고, 핵심은 날카로운 재검토다.

두 부분 모두 자기 페르소나의 눈(a사=디자인, b사=제품, t사=돌파, t사 디자이너=단순화·감정)으로 쓴다.

**각 관점 형식 (위 `[추가 제안]` 섹션용)**

- **a사**: `[n] 제목 · 임팩트/비용/ROI` → 현황·문제 / 제안(동작·상태·모션·마이크로카피) / 근거(디자인 원칙명)
- **b사**: `[PRD 현황]` 3층 모델 위치 + 2~3줄 진단. `[제안] B1..` 각 항목에 `· 진짜 문제(What/Why)` + `· 한 층 위로(선제화)`.
- **t사**: `T1.. 제약 → 통념 → t사식 돌파 → 실행`. "전환을 없앤다 / 자동 추론한다 / 부분성공 허용" 처럼 **문제 재정의** 우선.
- **t사 디자이너**: `[n] 제목 · 임팩트 n/5` → 현황·문제 / 제안 / 근거 **임팩트 내림차순**, a사 리뷰와 비겹침.

### STAGE 5: 최종 출력 + 완료

1. 네 에이전트가 반환한 제안을 모아 **로컬 콘솔에 출력**(디자인제안(a) · 디자인 제안(t) · 관점2(b) · 관점(t) 순) + `dev/active/prd-qa-{대상}/report.md` 에 합쳐
   저장(참고용).
2. STAGE 1에서 **에스컬레이션한 코드-불가 항목**이 있으면 헤더 `검증` 줄 아래에 `⚠️ 에스컬레이션: {항목} ({사유})` 한 줄을 덧붙인다.
3. 완료 알림:

```
📝 Notion: 4개 섹션(디자인제안(a)·디자인 제안(t)·관점2(b)·관점(t)) 갱신 · {page url}
prd-qa 작업완료!
```

| 대상 화면 URL 불명확                  | AskUserQuestion으로 확인                     |
| 로컬 서버 미기동                      | 기동 안내 후 중단(QA 진행 불가)                     |
| 코드로 해결 불가한 기능 항목               | 사용자에게 에스컬레이션. 그 외 기능 위반은 잔존 0까지 수정       |
| `--title` 매칭 2건+ (스크립트 exit 2) | 후보 제시 → 사용자 선택 → `--page` 로 재실행          |
| 에이전트 1개가 Notion 갱신 실패          | 해당 섹션만 그 에이전트(또는 메인)가 `--section` 으로 재실행 |

---

## 스크립트 참고 (`scripts/notion-prd-qa-report.mjs`)

```bash
# 섹션 모드 — 전용 에이전트가 자기 블록만 in-place 갱신 (핵심)
node scripts/notion-prd-qa-report.mjs <file> --page <id> --section "디자인제안(a)"

# 페이지 확보/생성 (--title 0건이면 상태=QA 새 페이지)
node scripts/notion-prd-qa-report.mjs <file> --title "{기능명}"

# 한 블록 전체 교체(레거시 단일 리포트용)
node scripts/notion-prd-qa-report.mjs <file> --page <id> --replace
```
