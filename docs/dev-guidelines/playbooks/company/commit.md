# 커밋 생성 (/commit)

**현재 세션에서 변경한 파일만** 분석하고 커밋을 생성합니다.

## 커밋 대상

- **현재 세션에서 수정/생성한 파일만 커밋**
- 이전 세션이나 다른 작업에서 변경된 파일은 제외
- 대화 컨텍스트를 분석하여 이번 세션에서 작업한 파일 식별

## 커밋 규칙

### 1. Co-Authored-By 미포함
- 커밋 메시지에 AI 작성 표시를 **절대 포함하지 않음**
- `Co-Authored-By: Claude` 등의 라인 제외

### 2. 커밋 본문 작성 원칙 (문제-원인-해결)
- **[문제]**: 무엇이 문제였는가
- **[원인]**: 왜 이 문제가 발생했는가
- **[해결]**: 왜 이 방식으로 해결했는가 ← **핵심**
  - 고려한 대안들 나열
  - 이 방식을 선택한 이유 (트레이드오프 분석)
  - 스태프 엔지니어 수준의 의사결정 근거

### 3. 커밋 분리
- 성격이 다른 변경사항은 별도 커밋으로 분리
- 타입: `fix`, `feat`, `test`, `refactor`, `chore`, `docs` 등

## 커밋 메시지 형식

```
<type>: <제목 (한글, 50자 이내)>

[문제] 현상, 버그, 요구사항

[원인] 근본 원인 분석

[해결] 선택한 해결 방식 + 왜 이 방식인가
대안으로 (1) ... (2) ... 를 고려했으나,
... 이유로 이 방식이 최선이라고 판단했다.
```

## 예시

```
fix: 매입정산 차주 정보 변경이 저장되지 않는 문제 수정

[문제] 차주 정보를 변경 후 저장해도 DB에 반영되지 않았다.

[원인] updateFields에 driverSnapshot이 누락되어 API로 전달되지 않았다.
기존 코드는 driverId, driverName만 전달하고 스냅샷은 빠져 있었다.

[해결] 프론트에서 driverSnapshot을 구성해서 API에 전달하도록 수정했다.
대안으로 (1) API에서 driverId로 조회해 스냅샷 생성 (2) 별도 동기화 API 호출을
고려했으나, 기존 아키텍처가 "프론트에서 스냅샷 구성 → API 전달"이고
companySnapshot 등 다른 스냅샷도 같은 방식이므로 일관성을 유지했다.
API 변경 없이 프론트만 수정하면 되어 영향 범위도 최소화된다.
```

## 수행 단계

### 0. 전처리: 빌드 & 테스트 검증 (필수)

커밋 전에 반드시 빌드와 테스트를 통과해야 한다.

```bash
# 1. 빌드 검증
pnpm build

# 2. 변경 파일 관련 유닛 테스트만 실행 (전체 테스트는 배포 시 실행)
pnpm test:changed
```

- **빌드 실패 시**: 에러를 수정하고 다시 시도
- **테스트 실패 시**: 실패한 테스트를 수정하고 다시 시도
- **변경 파일 없을 시**: 테스트 0개 실행되지만 정상 (전체 테스트는 /pre 배포 시 실행)
- **모두 통과해야만** 커밋 단계로 진행

### 1~4. 커밋 분류

1. **대화 컨텍스트 분석**: 이번 세션에서 수정/생성한 파일 목록 파악
2. `git status`로 해당 파일들의 상태 확인
3. `git diff`로 변경 내용 확인
4. 변경사항을 논리적 단위로 분류

### 5. Simple Design 스태프 엔지니어 리뷰 (필수)

각 논리적 commit 단위에 대해 `../../guidelines/company/simple-design-review.md` 의 프로토콜로 staged diff 를 리뷰한다.

- 페르소나, 8 카테고리 체크리스트, Severity 분류, 결과 처리 loop, 보고 형식, "안 짚은 것 + 이유" 단락 의무 — 모두 위 skill 파일이 단일 source.
- `simple-design-precommit.js` (정규식 hook) 가 통과해도 이 단계는 **반드시 수행**. hook 은 ~30% 만 커버, 본 프로토콜이 의미 분석 ~70-90%.

#### commit 통합 한정 정보 (이 파일에만)

- 🔴 / 🟠 발견 → spot fix 금지, **layer 차원 리팩토링** → re-stage → 재리뷰 loop (max 3회).
- 🟡 만 발견 → 진행하되 commit 메시지 [해결] 끝에 [한계] 단락 명시:
  ```
  [한계 / 후속]
  - <medium smell>: <왜 이 commit 에선 안 고치는지 — 별도 PR 등>
  ```
- 발견 0 → "[Simple Design 리뷰] 통과 — 8 카테고리 검토 완료" 한 줄 보고 후 다음 단계.

### 6. 커밋 생성

리뷰 통과 후 (또는 🟡 만 남은 후), 위 [문제]-[원인]-[해결] 형식에 맞게 commit 메시지 작성하고 `git commit` 실행.

### 7. Push (요청 시)

사용자가 명시 요청한 경우만 push. 별도 확인 없이 push 안 함.
