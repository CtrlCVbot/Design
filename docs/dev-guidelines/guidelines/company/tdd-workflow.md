# TDD Workflow Skill

## 적용 조건

- 모든 코드 수정/추가 작업 시 (필수)

---

## 핵심 원칙

```
프로덕션 코드보다 테스트를 먼저작성한다.
```

---

## 절차

### 1. 유즈케이스 정의

구현할 동작을 한 문장으로 정의한다.

```
예시:
- "만료된 JWT로 API 요청 시 401을 반환한다"
- "정산 금액이 0원이면 '협의중'으로 표시한다"
- "역발행 요청 시 공급자와 공급받는자가 올바르게 설정된다"
```

### 2. 실패하는 테스트 작성 (Red)

- 유즈케이스 하나당 테스트 하나
- 테스트를 실행하여 **실패를 확인**한 뒤 다음 단계로 넘어간다

### 3. 테스트를 통과시키는 최소한의 코드 작성 (Green)

- 테스트를 통과하는 것만 목표
- 과도한 설계, 미래 대비 코드 금지

### 4. 리팩토링 (Refactor)

- 테스트가 통과하는 상태를 유지하면서 코드 개선
- 중복 제거, 네이밍 개선, 구조 정리

---

## 테스트 작성 규칙

### 하나의 테스트는 하나만 검증한다

```typescript
// ✅ 좋은 예: 하나의 동작만 검증
it('만료된 JWT로 요청하면 401을 반환한다', () => { ...
});
it('만료된 JWT로 요청하면 에러 메시지에 correlation ID가 포함된다', () => { ...
});

// ❌ 나쁜 예: 여러 동작을 한 테스트에서 검증
it('만료된 JWT 처리', () => {
    // 401 반환 확인
    // 에러 메시지 확인
    // 로그 기록 확인
    // 리다이렉트 확인
});
```

### 테스트는 최대한 단순해야 한다

- Arrange-Act-Assert 각 단계가 명확해야 한다
- setup이 복잡하면 테스트 대상의 설계를 의심한다
- helper/factory로 setup을 숨기지 말고, 테스트 대상을 단순하게 만든다

### 테스트가 복잡하면 프로덕션 코드의 리팩토링 신호다

```
테스트 하나에 assert가 여러 개 필요하다
  → 프로덕션 코드가 너무 많은 일을 한다
  → 책임을 분리하라

테스트 setup이 길다
  → 의존성이 너무 많다
  → 인터페이스를 단순화하라

테스트를 작성하기 어렵다
  → 프로덕션 코드의 결합도가 높다
  → 구조를 개선하라
```

**테스트가 단순하게 작성되지 않는다면, 그것은 테스트의 문제가 아니라 프로덕션 코드에서 나는 리팩토링 냄새다.**

- 설정 파일 변경 (tsconfig, package.json 등)
- DB 마이그레이션 스크립트
- 타입 정의 파일 (`.d.ts`)
- 단순 오타 수정

## 체크리스트

- [ ] 유즈케이스를 먼저 정의했는가?
- [ ] 테스트를 먼저 작성하고 실패를 확인했는가?
- [ ] 각 테스트가 하나의 동작만 검증하는가?
- [ ] 테스트가 단순한가? 복잡하다면 프로덕션 코드 구조를 의심했는가?
- [ ] 최소한의 코드로 테스트를 통과시켰는가?
