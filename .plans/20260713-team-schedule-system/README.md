# 팀 일정관리 시스템 기획안

- 작성일: 2026-07-13
- 목표: 개발자 1명과 기획자 1명이 여러 프로젝트를 병렬로 진행할 때, **내부 실행 일정**과 **외부 보고 일정**을 분리하면서 기획부터 QA·배포 준비까지의 상태를 한 곳에서 관리한다.
- 적용 도구: Notion(일정·결정·보고의 기준), Git(구현 결과의 증거), HTML prototype(기획 결과의 빠른 확인), AI harness(개발 보조)
- 범위: 내부 팀 일정 운영체계와 Notion 템플릿 설계. 실제 Notion 생성·자동화 연결은 포함하지 않는다.

## 1. 운영 원칙과 결정

### 1.1 한 줄 원칙

**내부는 작업을 끝내기 위한 상세 운영판, 외부는 약속과 확인만 보여 주는 보고판으로 분리한다.**

상세 내부 논의, 예상과 실제 차이, 막힌 이유를 외부 보고에 그대로 노출하면 보고가 길어지고 변경에도 취약해진다. 반대로 외부 마일스톤만 내부에서 관리하면 오늘 무엇을 끝내야 하는지 보이지 않는다. 두 표는 같은 프로젝트와 작업을 참조하지만, 보는 정보와 갱신 주기가 다르다.

| 구분 | 내부 실행 일정 | 외부 보고 일정 |
| --- | --- | --- |
| 목적 | 오늘의 실행·병목 해소·완료 증거 관리 | 범위·일정·결정 요청의 명확한 공유 |
| 공개 범위 | 개발자·기획자 | 이해관계자/고객 등 외부 공유 대상 |
| 단위 | 작업(Task), 검토 게이트, 커밋, QA 케이스 | 프로젝트, 마일스톤, 공개 가능한 산출물 |
| 상태 | 세부 상태 11개 | 요약 상태 5개 |
| 갱신 시점 | 08:30 계획, 작업 중, 16:00 증거 등록 | 확정·변경·완료 시 또는 주간 보고 |
| 금지 정보 | 없음(단, 비밀값 제외) | 내부 추정, 개인별 작업량, 미확정 논쟁, 기술 부채의 상세 |

### 1.2 병렬 처리 한도(WIP limit)

동시에 시작할 수 있는 프로젝트 수와 동시에 **활성 실행 상태**인 프로젝트 수를 분리한다.

| 구분 | 허용 | 시작 조건 | 활성 작업 한도 |
| --- | --- | --- | --- |
| Main 1개 | 큰 기획, 신규 핵심 로직, 다수 화면/도메인 변경 | `PRD 초안 검토 대기` 이상 | 1개 |
| Sub 1~3개 | 작은 기능, 개선, 조사, 독립 기획 | Main 일정·의존성에 영향 없음 | 합계 2개 권장, 최대 3개 |

`활성`은 `내부 상세 기획`, `HTML prototype 제작`, `개발 진행`, `QA 진행` 상태를 뜻한다. Sub 3개를 모두 시작하더라도, 개발자 1명이 동시에 구현하는 일은 1개만 둔다. 나머지는 기획자 조사·피드백 대기 또는 다음 순서 대기로 둔다. 이것이 일정이 "진행 중"으로 과도하게 쌓이는 것을 막는 기준이다.

### 1.3 일정 산정의 기본값

- Main: **발견(문제·대안 확인) → 합의 → prototype → 구현 → QA**의 다섯 묶음으로 나누어 산정한다.
- Sub: 같은 흐름을 따르되, `PRD 초안`과 `prototype`을 짧은 단일 산출물로 축소할 수 있다.
- 하루 작업은 원칙적으로 "오늘 끝나면 확인할 수 있는 산출물 1개"를 기준으로 쪼갠다. 예: `회원가입 화면 구현`보다 `가입 실패 상태까지 포함한 HTML prototype 링크 공유`가 좋다.
- 일정은 `목표일`과 `확정 약속일`을 구분한다. 목표일은 내부 조정 가능, 확정 약속일은 외부 변경 안내가 필요하다.

## 2. 프로젝트와 작업의 공통 흐름

### 2.1 전체 상태 흐름

```text
아이디어/요청
  → PRD 초안 작성
  → 1차 회의 및 확인
  → 내부 상세 기획
  → HTML prototype 확인
  → 구현 준비 자료 확정
  → AI harness 구현
  → 개발자 자체 검증
  → QA
  → 완료/외부 보고
```

기획자와 개발자가 확인해야 하는 공식 게이트는 아래 두 개만 둔다.

| 게이트 | 확인 자료 | 목적 | 통과 기준 | 통과 후 |
| --- | --- | --- | --- | --- |
| G1: 방향 확인 | `PRD 초안` | 문제와 범위, 우선순위가 맞는지 결정 | 문제·주요 기능·러프 User Flow·논의 항목이 있고, 보류/제외 범위가 합의됨 | 내부 상세 기획 시작 |
| G2: 화면/흐름 확인 | `HTML prototype` | 실제 사용 흐름과 화면 수준의 오해 제거 | 핵심 사용자 흐름과 주요 상태가 클릭/전환 가능한 형태로 보임 | 구현 자료 확정 및 개발 시작 |

`내부 상세 기획`(상세 PRD, wireframe, 디자인 스펙, 상세 User Flow)은 공유를 위한 문서가 아니라 구현 품질을 위한 작업 파일이다. 모든 문서를 회의 안건으로 만들지 않고, G1과 G2 산출물만 확인·피드백 대상으로 삼는다.

### 2.2 내부 실행 상태 11개

| 상태 | 의미 | 담당 기본값 | 다음 상태 |
| --- | --- | --- | --- |
| `Inbox` | 요청만 수집됨 | 기획자 | `Triage` |
| `Triage` | Main/Sub, 우선순위, 의존성 판단 중 | 기획자 | `PRD 초안 작성` / `보류` |
| `PRD 초안 작성` | G1 자료 작성 중 | 기획자 | `PRD 초안 검토 대기` |
| `PRD 초안 검토 대기` | 1차 회의 안건 | 기획자 | `내부 상세 기획` / `보류` |
| `내부 상세 기획` | 구현에 필요한 상세 자료 정리 중 | 기획자 | `HTML prototype 제작` |
| `HTML prototype 제작` | G2 확인용 결과물 제작 중 | 기획자 중심, 개발자 검토 | `prototype 검토 대기` |
| `prototype 검토 대기` | G2 피드백·확인 | 기획자 | `개발 준비` / `내부 상세 기획` |
| `개발 준비` | 구현 작업 분해·harness 입력·수용 기준 확정 | 개발자 | `개발 진행` |
| `개발 진행` | 코드 구현·테스트 진행 | 개발자 | `QA 진행` |
| `QA 진행` | 수용 기준 및 회귀 확인 | 기획자 + 개발자 | `완료` / `개발 진행` |
| `보류` / `완료` | 우선순위 중단 또는 모든 종료 기준 충족 | 공동 | 필요 시 `Triage` |

차단은 별도 상태가 아니라 `Blocker` 관계형 항목으로 관리한다. 차단 원인을 상태에 섞으면 실제 진행률이 왜곡되기 때문이다.

## 3. Notion 구조: 데이터베이스 4개

초기에는 데이터베이스를 네 개로 제한한다. 화면을 많이 만들기보다, 관계(Relation)와 롤업(Rollup, 연결된 항목을 자동 집계하는 기능)으로 같은 사실을 반복 입력하지 않는 편이 운영에 유리하다.

```text
[프로젝트] 1 ── N [내부 작업]
     │                    │
     ├── N [외부 마일스톤] │
     └────────────────────┘
                    │
                    N
                 [결정/차단]
```

### DB-1. 프로젝트(Projects)

| 속성 | 유형 | 입력 규칙 |
| --- | --- | --- |
| 프로젝트명 | Title | `YYYYMM-프로젝트명` 또는 팀이 정한 고유 이름 |
| 구분 | Select | `Main`, `Sub` |
| 상태 | Status | `준비`, `진행`, `주의`, `완료`, `보류` |
| 오너 | Person | 최종 책임자 1명 |
| 기획 담당 / 개발 담당 | Person | 각 1명 |
| 목표일 | Date | 내부 목표일 |
| 확정 약속일 | Date | 외부에 공유된 경우에만 입력 |
| 진행률 | Rollup | 내부 작업의 완료 수 / 전체 수 |
| 현재 단계 | Rollup | 활성 내부 작업의 상태 |
| 차단 수 | Rollup | 열린 결정/차단 수 |
| G1 / G2 링크 | URL 또는 Files | PRD 초안, HTML prototype |
| Git 저장소 / 브랜치 | URL / Text | 코드가 있는 경우 |
| 외부 공개 가능 | Checkbox | 외부 보고 일정 노출 전 확인 |

**템플릿 본문**: 목표, 문제 인식, 범위/제외, 성공 기준, G1·G2 링크, 결정 로그, 외부 보고 요약 순서로 만든다.

### DB-2. 내부 작업(Internal Tasks) — 매일 보는 핵심 보드

| 속성 | 유형 | 입력 규칙 |
| --- | --- | --- |
| 작업명 | Title | 동사 + 확인 가능한 결과물. 예: `가입 실패 상태가 포함된 prototype 게시` |
| 프로젝트 | Relation | DB-1 프로젝트 필수 연결 |
| 상태 | Status | 2.2의 상태 사용 |
| 역할 | Select | `기획`, `개발`, `공동`, `QA` |
| 담당 | Person | 실행 담당 1명 |
| 우선순위 | Select | `P0`, `P1`, `P2` |
| 계획일 / 목표일 | Date | 오늘 계획과 완료 목표를 분리 |
| 실제 완료일 | Date | 완료 전환 시 기록 |
| 예상 시간 | Number | 0.5, 1, 2, 4시간 단위만 허용 |
| 완료 증거 | URL / Text | Notion 문서, prototype, Git commit SHA/URL, 테스트 결과 |
| 수용 기준 | Text | 완료 판정이 가능한 1~3개 문장 |
| Blocker | Relation | DB-4 연결. 없으면 비움 |
| 다음 액션 | Text | 막힌 경우에도 다음 행동 하나를 기록 |
| 외부 노출 | Select | `비공개`, `마일스톤 요약`, `외부 공유 가능` |

**작업 템플릿 본문**

```markdown
## 오늘의 완료 정의
- [ ] 확인할 수 있는 결과물:
- [ ] 수용 기준:

## 맥락과 입력
- 관련 PRD/prototype/Git 링크:
- 선행 작업:

## 16:00 증거
- 커밋 SHA 또는 문서/HTML 링크:
- 테스트·QA 결과:
- 계획 대비 차이와 이유:

## 다음 액션
- 다음 작업 1개:
```

### DB-3. 외부 마일스톤(External Milestones) — 보고용 타임라인

외부에는 Task를 직접 노출하지 않는다. `프로젝트별 3~6개 마일스톤`만 만든다.

| 속성 | 유형 | 입력 규칙 |
| --- | --- | --- |
| 마일스톤명 | Title | 예: `G1 PRD 초안 확인`, `G2 prototype 확인`, `MVP QA 완료` |
| 프로젝트 | Relation | DB-1 연결 |
| 상태 | Status | `예정`, `진행`, `검토 요청`, `완료`, `일정 조정 필요` |
| 예정일 | Date | 외부에 설명 가능한 날짜 |
| 확정일 | Date | 실제 완료 시 입력 |
| 공개 산출물 | URL / Files | PRD 초안, HTML prototype, 릴리스 노트 등 |
| 외부 요약 | Text | 2~3문장, 내부 사정 제외 |
| 결정 요청 | Relation | 외부 판단이 필요한 DB-4 항목만 연결 |
| 내부 준비도 | Rollup | 연결 프로젝트의 진행률·차단 수 참고용 |

### DB-4. 결정/차단(Decisions & Blockers)

| 속성 | 유형 | 입력 규칙 |
| --- | --- | --- |
| 제목 | Title | 질문형 또는 결정형. 예: `회원가입 약관 범위 확정 필요` |
| 종류 | Select | `결정 필요`, `차단`, `리스크`, `변경 요청` |
| 프로젝트 / 작업 | Relation | 최소 하나 필수 연결 |
| 영향도 | Select | `높음`, `중간`, `낮음` |
| 발생일 / 결정 기한 | Date | 두 날짜를 구분 |
| 제안안 | Text | 추천안 1개와 대안·trade-off |
| 결정자 | Person | 답을 줄 사람 |
| 상태 | Status | `열림`, `논의 중`, `결정됨`, `해결됨` |
| 결정 내용 | Text | 최종 선택과 근거 |
| 외부 공개 필요 | Checkbox | 외부 보고 일정에 올릴지 판단 |

## 4. Notion 뷰와 대시보드

### 매일 사용하는 내부 뷰

| 뷰 | 원본 DB | 필터/정렬 | 사용 시점 |
| --- | --- | --- | --- |
| 오늘 실행판 | 내부 작업 | 계획일=오늘, 상태≠완료·보류 / 우선순위 오름차순 | 08:30, 수시 |
| 역할별 칸반 | 내부 작업 | 담당/역할 기준, 상태별 Board | 실행 중 |
| Main 집중 보기 | 프로젝트 + 내부 작업 | 구분=Main, 상태≠완료 | 하루 2회 |
| 차단 해소함 | 결정/차단 | 상태=열림·논의 중, 영향도=높음 우선 | 08:30, 16:30 |
| 이번 주 완료 증거 | 내부 작업 | 실제 완료일=이번 주, 증거 비어있지 않음 | 주간 회고/보고 |

### 외부 공유 대시보드

외부 페이지는 `External Milestones`와 각 프로젝트의 외부 요약만 연결한다. 내부 작업 DB, 개인 담당, 예상 시간, 차단 사유는 연결하지 않는다.

- `이번 주 확인 요청`: 상태=`검토 요청`인 마일스톤
- `진행 중`: 상태=`진행`
- `완료`: 최근 2주 이내 완료
- `일정 조정 필요`: 변경 사유를 1문장으로만 노출하고, 내부 대책은 내부 DB에 둔다.

## 5. 하루 운영 프로세스

### 08:30 데일리 스크럼 — 5분, 실행 약속 만들기

각자 최대 60초, 총 5분을 넘기지 않는다.

1. 오늘 반드시 끝낼 작업 1개와 완료 증거를 말한다.
2. 어제 계획 대비 차이가 있으면 차단/결정 항목으로만 말한다.
3. 우선순위 충돌이 있으면 Main을 우선하고, Sub는 16:30에 재배치한다.
4. 합의가 끝나면 `내부 작업`의 계획일·담당·수용 기준을 갱신한다.

**발화 템플릿**: `오늘 [작업]을 [증거]까지 완료합니다. 완료 기준은 [기준]입니다. 막힌 점은 [없음/결정 항목]입니다.`

### 16:00 증거 등록 — 문서와 코드를 Git 기준으로 남기기

- 기획: PRD 초안·상세 기획·HTML prototype 중 변경한 결과를 링크하고 커밋한다.
- 개발: 구현 코드, 테스트 결과, commit SHA를 작업의 `완료 증거`에 붙인다.
- QA: 확인한 수용 기준, 재현 결과, 발견 이슈를 남긴다.
- 미완료: 상태를 억지로 완료로 바꾸지 않고 `다음 액션`과 계획 대비 차이를 기록한다.

### 16:30 퇴근 전 데일리 스크럼 — 10분, 약속과 증거 대조

기획자가 오전 `오늘 실행판`과 16:00의 완료 증거를 대조한다.

| 확인 항목 | 확인 질문 | 결과 처리 |
| --- | --- | --- |
| 계획 대비 | 오전에 약속한 증거가 등록되었는가? | 완료 / 이월 / 차단으로 정확히 표시 |
| 품질 | 수용 기준·테스트·검토 결과가 있는가? | 부족하면 `QA 진행` 또는 보완 작업 생성 |
| 일정 | 목표일 또는 확정 약속일이 흔들렸는가? | 내부 조정 또는 외부 마일스톤 `일정 조정 필요` |
| 의사결정 | 내일 시작을 막는 질문이 있는가? | DB-4에 담당·기한·추천안 등록 |

회의의 결과는 긴 회의록이 아니라 `완료 증거`, `다음 액션`, `결정/차단` 세 가지 필드에 남긴다.

## 6. AI와 harness의 사용 경계

AI는 일을 대신 완료했다는 표시가 아니라, 각 단계의 초안·구현·검증을 빠르게 만드는 보조 수단이다. 최종 완료 여부는 사람의 수용 기준과 Git/QA 증거로 판단한다.

| 단계 | AI 활용 | 사람이 확인할 것 | Notion에 남길 증거 |
| --- | --- | --- | --- |
| PRD 초안 | 문제 정리, User Flow 초안, 논의 질문 생성 | 범위·용어·우선순위 | PRD 초안 링크, G1 결과 |
| 내부 상세 기획 | 요구사항 누락 점검, 화면/상태 목록 정리 | 실제 업무 규칙·예외 | 내부 문서 링크 |
| HTML prototype | 화면 구조·상태 전환 초안 구현 | 핵심 흐름과 피드백 반영 | prototype URL, G2 결과 |
| harness 구현 | 구현 task 분해, 코드/테스트 초안, 반복 작업 | 요구사항 충족, 보안·품질, 테스트 | Git branch/commit, 테스트 결과 |
| QA | 수용 기준 기반 테스트 케이스/회귀 후보 생성 | 실제 동작·실패 케이스 | QA 결과, 이슈 링크 |

AI 입력에는 비밀값, 고객 개인정보, 운영 데이터 원문을 넣지 않는다. 작업 항목에는 `AI 사용: 초안/구현/검증/미사용` 정도만 표시하고, 생성 과정 전체를 일정 관리에 기록하지 않는다.

## 7. 외부 공개·연동 정책

### 7.1 공개 전 확인 규칙

외부 마일스톤을 `검토 요청` 또는 `완료`로 바꾸기 전에 다음을 확인한다.

- 프로젝트의 `외부 공개 가능`이 체크되었는가.
- G1은 PRD 초안, G2는 HTML prototype처럼 약속한 공개 산출물이 연결되었는가.
- 미확정 일정, 개인별 예상 시간, 내부 갈등/차단의 상세가 없는가.
- 결정 요청이 있다면 질문·기한·추천안이 한 문장으로 정리되었는가.

### 7.2 연동 우선순위

처음부터 복잡한 자동화는 만들지 않는다. 운영 안정성을 확인한 뒤 아래 순서로 확장한다.

1. **1단계: 수동 링크** — Notion 작업에 Git commit URL/SHA, prototype URL을 수동으로 기록한다.
2. **2단계: 반자동 집계** — Git PR/commit 알림이 Notion `완료 증거` 또는 팀 알림 채널에 쌓이도록 연결한다. 완료 상태 변경은 사람이 한다.
3. **3단계: 보고 자동 초안** — 주간 완료 작업과 외부 마일스톤을 AI가 요약하되, 공개 전에는 기획자가 확인한다.

자동화가 "커밋이 있으니 일정도 완료"로 상태를 바꾸면 안 된다. 커밋은 증거 중 하나일 뿐, 수용 기준과 QA가 남아야 완료다.

```text
- verification: 요구사항 추적, 운영 흐름, Notion 속성, 공개 경계, 도입 단계 self-review

```

## 12. Self-review

- 요구사항 반영: 내부/외부 일정 분리, 기획~QA 상태, 개발자 1명·기획자 1명, Main 1 + Sub 1~3, G1 PRD 초안, G2 HTML prototype, 08:30/16:00/16:30 리듬, Git 증거, AI harness, 외부 공유·연동을 모두 포함했다.
- 범위 확인: Notion 구조·템플릿·프로세스만 제안했으며 실제 외부 공개, 자동화, 코드 변경을 하지 않았다.
- 남은 리스크: 실제 프로젝트 크기와 외부 이해관계자 수가 확정되지 않아 Sub 3개 동시 활성은 아직 권장하지 않는다. 2주 관찰 후 조정한다.
