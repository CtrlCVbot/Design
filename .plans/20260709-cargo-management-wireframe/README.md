# 화물 관리 3열 워크스페이스 기획

- 작성일: 2026-07-09
- 범위: 오더 등록부터 정산까지 이어지는 관리자 TMS 중 1차 화면인 화물 관리 페이지
- 기준 자료: `.plans/20260708-nbbb1-dispatches-analysis`
- 산출물 성격: 실제 구현이 아닌 기획 검토용 HTML wireframe

## 목표

기존 `nbbb1.vercel.app/dispatches` 역설계에서 확인한 콕핏형 TMS 패턴을 새 화물 관리 업무에 맞게 재구성한다.

이번 기획은 다음 원칙을 우선한다.

1. 관리자 화면답게 정보 밀도를 높게 유지한다.
2. 목록, 상세, 등록, 조회, 선택, 배차를 페이지 이동 없이 한 화면에서 처리한다.
3. `drawer`와 `dialog` 대신 열(column) 확장으로 작업 맥락을 유지한다.
4. 필터는 새 컬럼을 추가하지 않고, 기존 전역 메뉴 사이드바가 필터 모드로 확장되는 구조로 처리한다.
5. 데스크톱은 최대 3열 워크스페이스, 모바일은 카드형/단계형 구조로 별도 설계한다.

## 핵심 설계 요약

### 데스크톱

```text
1열: 화물 목록
필터 모드: 전역 메뉴 사이드바(DB/화/차/정/톡)가 상세 필터 사이드바로 확장 + 화물 목록 유지
2열: 화물 목록 + 화물 상세/등록
3열: 화물 목록 + 화물 상세/등록 + 차주/화주 조회/등록/선택
```

### 모바일

```text
목록 카드 -> 상세/수정 -> 차주/화주 선택 -> 배차 정보 입력
```

모바일은 데스크톱 테이블 축소판이 아니라 별도 업무 구조로 본다. 목록 카드에는 상태, 화주, 노선, 상차일시, 운임/배차 상태만 우선 표시한다.

## 기존 분석에서 가져온 패턴

- 페이지 이동 없이 한 화면에서 업무를 이어가는 콕핏형 구조
- 상태/날짜/검색 필터가 서로 영향을 주는 실시간 카운트
- 거래처/구간/주소/운임으로 이어지는 입력 가속 체인
- 배차 난이도, 추천 사유, 차주 상태처럼 의사결정에 필요한 보조 정보
- 등록 또는 수정 후 목록 최상단 갱신

## 이번 기획에서 바꾼 점

- 기존 좌/우 패널 모드 전환을 1~3열 workspace로 재구성했다.
- 행 아코디언 상세 대신 독립된 2열 상세/등록 패널을 둔다.
- 차주/차량, 화주 조회/등록/선택을 모두 3열 패턴으로 통일했다.
- 배차 금액과 배차 콜센터 입력은 3열 하단의 선택 후 작업 영역으로 둔다.
- 모바일은 테이블이 아니라 필수 정보 카드와 단계형 상세 흐름으로 분리했다.

## 작업 경계

- 실제 서비스 코드, API, DB, 인증 로직은 포함하지 않는다.
- HTML은 기획 검토용 목업이며, 화면 상태 전환을 확인하기 위한 최소 JavaScript만 포함한다.
- 디자인은 기존 토스 계열 TMS 톤을 참고하되, 새 3열 업무 구조를 우선한다.
