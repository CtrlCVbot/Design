# 새소망 방문요양 웹사이트 구현 계획

> 상태: Next.js + Vercel Preview 구현 전 검토 대기
> 작성 기준: `핸드오프 문서.dc.html` v1.1, `디자인 시스템.dc.html` v1.0, `모션과 인터랙션.dc.html` v1.1 및 6개 화면 HTML
> 계획 범위: Next.js 구현 준비, Vercel Preview 검수 흐름 정의까지. 이 문서는 구현·배포 승인으로 간주하지 않는다.

## 1. 결론

새소망 방문요양 웹사이트는 새 독립 Next.js App Router 앱 `apps/alliance-homecare`로 구현한다. 6개 공개 페이지와 공용 내비게이션·푸터를 먼저 reference HTML과 동일하게 재현하고, 각 완료 슬라이스는 Vercel Preview 배포 URL에서 검수한다. 상담 신청 기능·반응형·SEO·접근성은 후속 구현 슬라이스로 붙인다.

현재 바로 구현을 시작하면 안 되는 항목은 다음 네 가지다.

1. 브랜드명, 전화번호, 후기, 통계, 연혁, 언론사명을 실데이터로 교체할지 결정되지 않았다.
2. 10개 이미지 슬롯을 대체할 실사진과 사용 권한이 없다.
3. 상담 신청을 어디에 저장하고 누구에게 어떤 방식으로 알릴지 정해지지 않았다.
4. reference는 데스크톱 1180px 이상만 정의하며 모바일·태블릿 디자인은 없다.

따라서 G0 결정 게이트를 통과한 뒤 TDD(Test-Driven Development, 테스트를 먼저 작성해 동작을 확인하는 방식)로 구현한다. 데스크톱은 exact fidelity 대상으로, 반응형은 별도 승인된 adaptation(적응 설계) 대상으로 구분한다.

## 2. 목표와 비목표

### 목표

- 6개 공개 페이지의 구조·카피·링크·시각 토큰을 reference와 동일하게 재현한다.
- 내비 축소, 카드 호버, 스크롤 리빌, 카운트업, FAQ, 상담 폼 상태를 구현한다.
- 상담 신청을 서버에서 검증하고 승인된 저장·알림 채널로 전달한다.
- 키보드 조작, 모션 축소 설정, 오류 안내를 포함한 접근성을 확보한다.
- 공개 사이트에 필요한 SEO 메타데이터, OG 이미지, sitemap, robots 정책을 적용한다.
- 데스크톱 fidelity 증거와 반응형·기능·접근성 검증 결과를 남긴다.

### 비목표

- 관리자 페이지, 회원가입·로그인, 결제, 예약 일정 관리
- CMS(Content Management System, 운영자가 콘텐츠를 관리하는 도구) 구축
- 채용 상세 페이지, 개인정보처리방침, 이용약관의 실제 법률 문안 작성
- 전화 트래킹 서비스의 임의 선정 또는 외부 계정 생성
- reference에 없는 새로운 브랜드 스타일이나 화면의 임의 추가
- production 배포

## 4. 권장 기술·구조

구현 기반은 Next.js App Router + TypeScript로 확정한다. 실제 패키지 버전은 앱 생성 시 공식 지원 범위를 확인해 lockfile에 고정한다. 현재 저장소에는 공용 `package.json`이나 Next.js 앱이 없으므로, 기존 `apps/cargo-order-admin-hifi` 정적 preview를 수정하지 않고 독립 npm 프로젝트로 만든다.

- Next.js App Router + TypeScript: 6개 정적 콘텐츠 페이지의 SEO와 상담 API를 한 앱에서 관리
- React Server Component 우선: 정적 섹션은 서버 렌더링하고 상호작용만 Client Component로 분리
- CSS Modules + `app/globals.css`: inline style을 그대로 복사하지 않고 디자인 토큰과 상태 클래스로 재구성
- Zod 또는 동등한 스키마 검증: 클라이언트 편의 검증과 서버 신뢰 경계를 분리
- Vitest + Testing Library: 순수 함수와 사용자 행동 중심 컴포넌트 테스트
- Playwright: 페이지 이동, 폼, 키보드, reduced motion, 반응형 E2E와 screenshot evidence
- Vercel: PR 또는 수동 Preview 배포 URL에서 디자인·기능 검수. Production은 별도 승인 후에만 배포

같은 패턴이 실제로 세 번 이상 반복되기 전에는 범용 section renderer나 거대한 JSON 기반 페이지 엔진을 만들지 않는다. 페이지 컴포넌트가 의도를 드러내도록 유지하고, 공용 내비·푸터·버튼·카드·모션 primitive만 재사용한다.

Next.js 경계는 다음처럼 유지한다.

- `page.tsx`, `layout.tsx`, 정적 콘텐츠 섹션은 Server Component를 기본으로 한다.
- 스크롤 내비, FAQ, 카운트업, reveal, 상담 폼처럼 브라우저 상태가 필요한 작은 leaf component에만 `'use client'`를 둔다.
- `next/link`로 페이지를 이동하고, 모든 공개 route에 `metadata`를 둔다.
- 히어로·서비스 사진은 `next/image`를 사용하며, LCP 이미지에는 `priority`를 적용한다.
- `app/api/consultations/route.ts`는 파싱·Zod 검증·서비스 호출만 담당한다. 저장·알림 정책은 route 내부에 섞지 않는다.
- 공개 가능한 값만 `NEXT_PUBLIC_*`로 둔다. 상담 수신 키·메일 키·CRM 토큰은 Vercel server environment variable로만 관리한다.

### Vercel Preview 검수 흐름

| 단계 | 실행 위치 | 결과 | Production 경계 |
| --- | --- | --- | --- |
| 로컬 검증 | `apps/alliance-homecare` | lint, typecheck, unit/integration, build, E2E 결과 | 통과 전 Preview 배포하지 않음 |
| Preview 배포 | Vercel의 `apps/alliance-homecare` Root Directory | 검수용 고유 URL | 자동/수동 배포 모두 `noindex` |
| Preview 검수 | 1440px reference 비교와 4개 viewport QA | screenshot evidence, gap board 갱신 | critical/high gap이 있으면 재배포 |
| Production 승인 | 사용자·콘텐츠·법률·운영 담당자 | 승인된 Preview build 식별 | 명시 승인 전 `--prod` 사용 금지 |
| Production 배포 | Vercel Production | 실제 도메인 URL | robots·분석·실제 상담 수신은 이 시점에 재검토 |

Vercel 프로젝트 설정 초안:

- Framework Preset: `Next.js`
- Root Directory: `apps/alliance-homecare`
- Install Command / Build Command: Vercel 기본값 사용 (`npm ci`, `next build` 계열). 정적 앱용 `outputDirectory: "."` 설정을 재사용하지 않는다.
- Node.js 버전: `package.json`의 `engines.node`와 Vercel 설정을 같은 지원 버전으로 고정한다.
- Preview: `X-Robots-Tag: noindex` 또는 동등한 `robots` 정책 적용. Preview는 검색 노출 금지.
- Environment Variables: Preview와 Production을 분리하고, 실상담 수신 시크릿은 Preview에 주입하지 않는다.

`vercel.json`은 Preview noindex header처럼 기본 설정만으로 표현할 수 없는 요구가 확인될 때만 추가한다. framework, install command, output directory를 고정해 Vercel의 Next.js 감지를 막지 않는다.

Preview의 상담 폼은 실제 개인정보를 수집하지 않는다. G5 이전에는 제출을 허용하지 않거나, 명확히 표시한 demo mode에서 어떤 데이터도 저장·전송하지 않는다. 실제 상담 접수는 법률·수신 채널·비밀값이 승인된 Production 환경에서만 활성화한다.

| `VF-SEO` | 페이지별 metadata·OG·sitemap 미정 | medium | G5에서 실제 도메인·공개 정책 확정 후 적용 |

## 7. 상담 신청 계약 초안

### 요청 필드

| 필드 | 필수 | 서버 검증 |
| --- | --- | --- |
| `name` | 예 | 공백 제거, 허용 길이 제한 |
| `phone` | 예 | 정규화 후 국내 전화번호 정책 검증 |
| `region` | 예 | 공백 제거, 허용 길이 제한 |
| `service` | 예 | `nursing/aide/chronic/membership/senior/pediatric/etc` allowlist |
| `message` | 아니요 | 허용 길이 제한, 출력 시 escape |
| `consent` | 예 | 반드시 `true`, 정책 버전과 동의 시각 기록 여부 결정 |

### 서버 동작

1. 클라이언트 검증은 빠른 피드백용이며 서버에서 동일한 신뢰 경계를 다시 검증한다.
2. 성공 응답 전 저장·알림 결과를 확인하고 중복 제출 방지 정책을 적용한다.
3. 개인정보를 애플리케이션 로그, 분석 도구, 오류 추적 도구에 원문으로 남기지 않는다.
4. 알림 공급자 장애는 사용자에게 재시도 가능한 오류로 표현하고 운영자가 확인할 수 있는 비식별 correlation ID만 남긴다.
5. rate limit, spam 방지, 보유 기간, 삭제 절차는 공개 전 필수 결정한다.

DB, 이메일, Slack, CRM 중 어떤 채널을 사용할지는 계획에서 확정하지 않는다. 운영자가 실제로 처리할 수 있는 단일 수신 채널을 G0에서 선택한 후 최소 구현한다.
