## 0. 한 문장 대조

두 시스템은 **같은 문제("AI에게 규칙과 능력을 어떻게 공급하나")를 정반대 방향에서 푼다**:

- **하네스(logishm)**: 프로젝트 안에 모든 것을 복사해 넣는다 → "이 레포를 여는 누구든 같은 환경" (거버넌스 우선)
- **Codex 전역 환경**: 사용자 홈(`~/.codex`, `~/.agents`)에 계층·플러그인으로 쌓는다 → "이 사람이 여는 어떤 프로젝트든 같은 능력" (개인 생산성 우선)

둘은 경쟁 관계가 아니라 **보완 관계**다. 아래에서 무엇을 서로 배울 수 있는지 짚는다.

## 1. Codex 전역 환경의 구조 분석 (문서에서 읽어낸 것)

### 1-1. 스코프가 3+1 계층이다

| 계층 | 위치 | 내용물 |
|---|---|---|
| **전역 지침** | `~/.codex/AGENTS.md` (92줄, "core-sized") | 모든 프로젝트에 적용되는 개인 지침 |
| **전역 스킬 (사용자 제작)** | `~/.agents/skills/` 24개 | 도구 공용 계층 — health-check, verification-engine, skill-factory 등 |
| **전역 스킬 (개인)** | `~/.codex/skills/` 10개 | playwright, figma, vercel-deploy 등 개인 유틸 |
| **플러그인 (버전 관리됨)** | `~/.codex/local-plugins/` + `plugins/cache/<마켓>/<이름>/<버전>/` | claude-kit 2.3.0 (스킬 32·커맨드 38·훅 14·에이전트 21), OpenAI 공식/큐레이티드 플러그인들 |
| (참고) 프로젝트 계층 | 각 레포의 `.codex/`, `AGENTS.md` | ← 여기가 하네스가 채우는 자리 |

핵심 관찰: **콘텐츠가 "설치 위치"가 아니라 "적용 범위"에 따라 배치**된다. 회사 공통급 능력(TDD 가드, 검증 엔진)은 전역에 한 번만 있고, 프로젝트에는 프로젝트 것만 남는다.

### 1-2. 플러그인 = 버전 있는 배포 단위

- claude-kit이 `2.3.0+codex.20260624065732` 같은 **버전 문자열**을 갖고, 마켓플레이스(`~/.agents/plugins/marketplace.json`)에 등록되며, **버전별 캐시 디렉터리**로 풀린다.
- 원본(`local-plugins/claude-kit`)과 런타임 캐시(`plugins/cache/user-local/...`)가 분리 → 원본 수정 후 재빌드하면 캐시가 갱신되는 **업데이트 전파 경로**가 존재한다.
- config.toml이 "10 plugins, hook trust 7/7 current"를 추적 — **훅 신뢰(trust)가 상태로 관리**되고 최신 여부까지 점검된다.

### 1-3. 하네스에 없는 구성 요소 4종

| 구성 요소 | Codex 전역 환경에서의 모습 |
|---|---|
| **서브에이전트 정의** | claude-kit `agents/` 21개 — dev-architect, dev-code-reviewer, dev-security-reviewer, plan-prd-writer 등 역할별 에이전트 md + **결정적 게이트 JS**(plan-dev-gate.js: "기획 단계를 통과해야 개발 단계로") |
| **메모리 시스템** | `~/.codex/memories/MEMORY.md`(레지스트리) + rollout_summaries + 세션 인덱스/아카이브 — 경험이 파일로 누적·요약되는 구조 |
| **헬스체크** | `agentic-health-check` 스킬 — 전역 환경 전체(AGENTS.md 크기, 플러그인, 스킬 중복, 훅 trust)를 pass/fail 표로 자가 진단. 이 인벤토리 문서 자체가 그 실행 결과물 |
| **크로스 AI 리뷰** | `claude-reviewer` — Claude를 MCP 서버로 감싸 **Codex 안에서 Claude에게 리뷰를 시키는** 래퍼. "작성자와 다른 모델이 검토"하는 교차 검증 |

### 1-4. 훅의 관심사가 더 넓다

하네스 훅은 "코드 품질·구조"(TDD, 레이어, simple design)에 집중한다. claude-kit 훅 14개는 범위가 다르다:

- `output-secret-filter` — **출력 시점** 비밀값 필터 (하네스는 커밋 시점만)
- `dev-db-guard` — DB 파괴 작업 가드
- `dev-feature-scope-guard` / `plan-doc-guard` / `plan-idea-move-guard` — **작업 범위·문서 규율** 가드 (스코프 크리프 방지)
- `feedback-collector` / `feedback-subagent-collector` — **피드백 수집** (하네스의 probe는 "발사됐나"를 재고, 이쪽은 "잘했나"의 원료를 모음)
- `security-auto-trigger` — 보안 관련 변경 감지 시 보안 파이프라인 자동 발동
- `continuous-learning` — 세션에서 배운 것을 자산화하는 루프

### 1-5. 기획(plan)·카피(copy) 워크플로우가 1급 시민이다

claude-kit의 스킬·커맨드·에이전트는 `plan-*`(아이디어→스크리닝→PRD→와이어프레임→리뷰→아카이브), `copy-*`(카피 QA·정합성 검증), `dev-*` 3계열로 나뉜다. **비개발 직군(기획자)의 작업 파이프라인 자체가 스킬·게이트로 정의**돼 있다. 하네스는 "비개발자가 안전하게 *개발*하게" 하는 데 집중하고, 기획 업무 자체의 하네스는 없다.

## 2. 하네스가 우위인 것 (Codex 전역 환경에 없는 것)

공정한 비교를 위해 반대 방향도 명확히:

| 하네스의 강점 | 왜 Codex 전역 환경엔 없나 |
|---|---|
| **도구 무관 강제 3겹** (git 게이트 → CI → 브랜치 보호) | 전역 환경은 "내 도구 안"의 강제뿐 — 사람이 손으로 커밋하거나 다른 도구를 쓰면 안 막힘. 하네스의 git/CI 게이트는 **누가 뭘로 작업하든** 막는다 |
| **단일 규칙 소스** (rules.json을 검사기 3개가 공유) | claude-kit 훅들은 각자 JS에 규칙 내장(하네스 v0.3 이전 방식 — rules.json 선언화 이전) — 규칙 드리프트 위험 존재 |
| **팀 배포·재현성** | 전역 환경은 **한 사람의 PC** 설정 — 팀원에게 그대로 복제하는 공식 경로가 이 문서엔 없다(문서 자체가 "백업하려면 목록을 떠라"고 권할 정도). 하네스는 clone+한 줄이 곧 배포 |
| **비개발자용 실패 메시지** | 전역 환경의 가드는 파워유저 전제 |
| **파리티 실측 검증 문서화** (PARITY.md) | 전역 환경은 검증 기록이 아니라 인벤토리만 |
| **AI 설치 프로토콜** (AGENTS.md가 곧 설치 절차) | 전역 환경 구축은 수작업 누적의 결과 |

정리하면: **하네스는 "팀·거버넌스" 축이 강하고, Codex 전역 환경은 "개인 능력·수명주기 관리" 축이 강하다.**

## 3. 하네스의 부족한 점과 추가 반영 후보

위 대조에서 도출되는 하네스 개선 후보를 우선순위로:

### ① 업데이트 전파 문제 — add-only의 숨은 비용 ★ 가장 시급

하네스의 add-only 복사는 충돌 안전성을 주지만, 치명적 이면이 있다: **한 번 설치된 `.claude/` 파일은 라이브러리가 업데이트돼도 영원히 갱신되지 않는다** (이미 있는 파일은 건너뛰므로). `harness update`는 라이브러리 클론만 갱신할 뿐이다. Codex의 버전 있는 플러그인 캐시는 정확히 이 문제의 해법이다.

**반영 제안**: (a) 번들 파일에 버전 스탬프/매니페스트를 두고 `harness <대상> --sync-harness` 같은 "변경 없는 파일만 갱신, 사용자가 수정한 파일은 보존+보고" 모드 추가, 또는 (b) 이미 준비된 Claude 플러그인 마켓플레이스([06 §1-5](./06-generalization.md))로 배포 채널 자체를 전환 — 플러그인은 버전 업데이트가 도구 차원에서 해결된다.

### ② 전역(사용자) 계층의 부재

하네스는 회사 공통 지침조차 **프로젝트마다 복사**한다. 프로젝트 10개면 tdd-workflow.md 사본 10개 — 드리프트와 갱신 누락의 온상이다. Codex 3계층처럼:

- **회사 공통 + 스택** → 전역/플러그인 계층 (PC당 1회, 버전 관리)
- **프로젝트 고유 규칙 + rules.json** → 프로젝트 계층 (지금처럼 레포에)

로 나누면 "규칙은 코드와 함께"라는 하네스 철학은 지키면서(프로젝트 규칙은 그대로 레포에) 공통분은 중복이 사라진다. 단, 트레이드오프 주의: 전역 계층은 "레포만 clone하면 끝"이라는 재현성을 약화시키므로, **git 게이트·rules.json 같은 강제 장치는 반드시 프로젝트 계층에 남겨야** 한다 (전역은 "능력", 프로젝트는 "강제").

### ③ 헬스체크 부재

설치 시점 검증(AGENTS.md 프로토콜 4단계)은 있지만 **운영 중 진단**이 없다. 훅 배선이 깨졌는지, rules.json이 파싱되는지, 게이트가 여전히 hooksPath에 걸려 있는지(`git config --unset` 우회 감지!), 버전이 뒤처졌는지를 `harness doctor <대상>` 한 명령으로 pass/fail 표를 내는 기능은 Codex의 agentic-health-check에서 직수입할 가치가 있다. 특히 **"게이트 우회 감지"는 비개발자 거버넌스 도구로서 하네스에 더 절실**하다.

### ④ 서브에이전트·역할 정의 계층

하네스의 절차서(playbooks)는 "병렬 서브에이전트 실행을 언급하되 없으면 순차로"라고만 한다. claude-kit처럼 리뷰어·검증자 역할을 **에이전트 정의 파일**로 표준화하면 (dev-code-reviewer, dev-security-reviewer 등) solid-review·verify 계열 절차서의 품질과 재현성이 올라간다. 결정적 게이트 JS(plan-dev-gate)와 절차서를 결합하는 패턴도 하네스의 파이프라인 커맨드(fix-pipeline, refactor-pipeline)에 이식 가능.

### ⑤ 피드백·학습 루프

harness-probe는 "훅이 발사됐나"(활성화)를 잰다. claude-kit의 feedback-collector/continuous-learning은 "결과가 좋았나"의 원료를 모아 지침 개선으로 되먹인다. 하네스에도 "차단당한 뒤 실제로 고쳐졌나", "어떤 규칙이 오탐이 많나"를 기록하면 rules.json을 데이터 기반으로 다듬을 수 있다 — 지표(§[04](./04-enforcement.md) §5)의 자연스러운 다음 단계.

### ⑥ 출력 시점 비밀값 필터

하네스의 비밀값 검사는 커밋 시점이다. output-secret-filter처럼 **AI 응답 출력 시점** 필터를 추가하면, 커밋 전에 이미 채팅 로그·문서로 새는 경로를 막는다. 하네스 훅 체계(PreToolUse/PostToolUse)에 한 파일 추가로 가능.

### ⑦ 기획 직군용 파이프라인 (선택)

하네스의 대상이 "비개발자(기획자·팀장)"인데 정작 내용물은 전부 개발 규율이다. claude-kit의 plan-* 계열(아이디어→PRD→와이어프레임→리뷰)은 **기획 산출물에도 하네스를 채울 수 있음**을 보여준다 — 이 Design 레포(기획/디자인 전용)야말로 그런 하네스의 수요처다. 예: `.plans/` 구조 규칙, 와이어프레임 체크리스트 게이트, PRD 템플릿 강제.

## 4. 기존 범용화 분석(06)과의 대조 — 무엇이 달라지나

[06-generalization.md](./06-generalization.md)는 **"배포(distribution) 축"**으로 범용화를 봤다: 하드코딩 제거(수평), 조직 표준화(수직), 언어 확장(심도). Codex 전역 환경 문서는 거기에 빠져 있던 **두 축을 추가**한다:

| 축 | 06의 관점 | Codex 문서가 드러낸 것 | 결론 |
|---|---|---|---|
| **배포 단위** | "파일 복사를 파라미터화하자" | 복사가 아니라 **버전 있는 플러그인 구독**이 상위 호환 | 06 로드맵 3순위(마켓플레이스 전환)를 **1순위로 승격**해야 함 — add-only 업데이트 문제(§3-①)까지 함께 풀리므로 |
| **스코프 계층** | "엔진은 전역화, 규칙(데이터)은 프로젝트에" — 2분법 | 실제로는 **전역/플러그인/프로젝트 3계층**이 유효 — "회사 공통 능력"은 전역이 맞고, "프로젝트 강제"만 레포에 | 06의 2분법을 3계층으로 정교화. 단 강제 장치의 프로젝트 잔류 원칙은 유지 |
| **수명주기** | 설치·재설치까지만 다룸 | 설치 후의 **진단(health check)·신뢰(trust)·학습(feedback)** 이 운영의 절반 | 로드맵에 `harness doctor`(§3-③)와 피드백 루프(§3-⑤) 신설 |
| **콘텐츠 범위** | 개발 지침·강제에 한정 | 서브에이전트·메모리·기획 파이프라인도 배포 대상 자산 | "하네스 = 훅+스킬+게이트"라는 정의를 "에이전트·메모리 규약 포함"으로 확장 검토 |

06에서 유지되는 것 (Codex 문서가 바꾸지 못한 결론):

- **"강제는 도구 무관 지점(git/CI)에"** — Codex 전역 환경이 오히려 이 층이 없어서, 하네스의 이 설계가 옳았음을 반증으로 확인
- **"프로젝트 규칙은 코드와 함께"** — 전역 계층에 올릴 것은 회사 공통 *능력*이지 프로젝트 *규칙*이 아님
- **"knowledge·시크릿·개인 메모리는 공유 금지"** — Codex의 memories도 개인 계층에 있지 공유 대상이 아님 (같은 결론)

### 갱신된 로드맵 (06 §4의 개정판)

| 순위 | 작업 | 출처 |
|---|---|---|

## 5. 결론

- Codex 전역 환경은 하네스가 못 보던 **"수명주기(버전·진단·학습)와 스코프 계층"**을 보여주는 거울이다. 특히 **add-only 배포의 업데이트 전파 문제**는 이 비교로 드러난 하네스의 가장 실질적인 약점이다.
- 반대로 하네스의 **도구 무관 강제(git/CI)와 단일 규칙 소스, 팀 재현성**은 Codex 전역 환경에 없는 자산이며, 3계층으로 재편하더라도 이 부분만은 프로젝트 계층에 남겨야 한다.
- 이상적 종착점: **"능력은 전역 플러그인으로 구독하고(버전·업데이트·진단 자동), 강제는 프로젝트에 박아 넣는(레포만 clone하면 게이트 동작)"** 하이브리드 — 두 시스템의 강점 합집합이다.
