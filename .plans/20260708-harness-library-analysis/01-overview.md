# 01. 개요 — 무엇이고, 왜 이렇게 만들어졌나

## 1. 풀려는 문제

mm-broker 프로젝트에는 오랜 기간 다듬은 AI 개발 환경이 있다 — 스킬 32종, 슬래시 명령어 27개, 자동 검사 훅 16개(+공유 헬퍼 1), QA 지식 자료. 문제는 이것이 **한 사람의 한 프로젝트에 갇혀 있었다**는 것:

1. **다른 프로젝트**에서 새로 시작하면 아무것도 없다.
2. **다른 도구**(Codex 등)를 쓰면 `.claude/` 설정이 무의미하다.
3. **비개발자(기획자·팀장)**가 AI로 개발하면, "문서로 안내"만으로는 실수를 못 막는다.

이 라이브러리는 세 문제를 한 번에 푼다: **"clone 한 번 + 명령 한 줄이면, 누구든·어떤 도구든·어떤 프로젝트든 같은 환경과 같은 강제를 받는다."**

## 2. 핵심 설계 원칙 5가지

### ① 원본은 도구 중립적 마크다운 (canonical)

지침의 진짜 원본은 `guidelines/`(규칙)와 `playbooks/`(절차서)의 **평범한 .md 파일**이다. Claude 전용 포맷(SKILL.md, frontmatter)이나 Codex 전용 포맷은 전부 **파생물(어댑터)**로 취급하고 스크립트가 자동 생성한다. 덕분에:

- 사람도 같은 글을 읽는다 (도구별 사본 관리 불필요)
- 새 도구가 나와도 어댑터 스크립트 하나만 추가하면 된다

### ② 입구는 AGENTS.md 하나

여러 AI 도구(Codex, Cursor, Copilot)가 공통 인식하는 표준 파일 `AGENTS.md`를 유일한 입구로 삼는다. Claude Code는 `AGENTS.md`를 직접 읽지 않으므로 `CLAUDE.md`에 `@AGENTS.md` 한 줄(셔틀)을 둔다. **모든 도구와 사람이 같은 문서 하나를 읽게 된다.**

### ③ 규칙(데이터)과 검사기(엔진)의 분리 — 단일 규칙 소스

"Service가 drizzle을 직접 import하면 안 된다" 같은 구조 규칙은 원래 Claude 훅 JS 안에 하드코딩돼 있었다. 이를 **선언적 설정 `rules.json` 한 파일**로 빼내고, 검사기 3개(Claude 가드, Codex 가드, git 커밋 게이트)가 **모두 같은 파일을 읽게** 했다. 규칙을 한 번 고치면 세 군데 다 반영되고, 도구 간 규칙 드리프트가 사라진다.

### ④ add-only — 사용자 소유 파일은 덮지 않는다

충돌 정책이 대상별로 다르다. **add-only(있으면 건너뜀)가 적용되는 것**: `.claude/`·`.codex/` 번들, `.mcp.json`, `AGENTS.md`(있으면 `.logishm-new`로 별도 생성), `CLAUDE.md`(있으면 안내만), 활성 `rules.json`(없을 때만 기본값 생성). 반면 **라이브러리가 소유한 사본은 재실행 시 최신본으로 덮어써 갱신된다**: `docs/dev-guidelines/**` 지침 사본(cpSync), 게이트 스크립트 — 이게 업데이트 전파 경로다. 또 `--project`는 번들 규칙을 `enforce/rules.json`에 덮어쓴다. 원칙은 **"사용자 파일은 멈추고 묻고, 라이브러리 파일은 갱신한다"**이며, 여러 번 실행해도 결과가 같다(멱등).

### ⑤ 안내가 아니라 강제 — 3겹 방어

비개발자는 코드가 맞는지 스스로 판단하기 어렵다. 그래서 "지침 문서"(1겹) 위에 "로컬 커밋 게이트"(2겹), "CI + 브랜치 보호"(3겹)를 쌓는다. 1·2겹은 우회 가능하지만 3겹(서버)은 물리적으로 우회 불가 — 이게 진짜 방어선이다. 자세히: [04-enforcement.md](./04-enforcement.md)

## 3. 4단계 분류 체계 (콘텐츠를 어디에 두나)

모든 지침을 **"이 문서를 이해하려면 무엇을 알아야 하나"** 기준으로 나눈다 (지금 누가 쓰는지가 아니라 내용 기준):

| 분류 | 판단 질문 | 두는 곳 |
|---|---|---|
| 공유 불가 | 비밀값·개인 취향·내 PC 경로인가? | `.env`/개인 설정 (어디에도 안 넣음) |
| 프로젝트 고유 | 그 프로젝트의 코드 구조·도메인을 알아야 이해되나? | 각 프로젝트에 남김 (번들로 스냅샷만) |
| 기술스택 | 특정 기술(React, Drizzle…)을 알아야 이해되나? | `guidelines/stack/<스택>/` |
| 회사 공통 | 셋 다 아니면 | `guidelines/company/` |

부수 원칙: **원칙(글)과 그 원칙의 검사 코드는 따로 분류한다** (예: TDD 원칙 글=회사 공통, tdd-guard 훅=TS 스택 — 검사 코드가 .ts 파일 대상이므로). 파일이 아니라 **내용 단위**로 쪼갠다.

## 4. 역사 (버전 전환으로 보는 설계 진화)

AUDIT.md에 기록된 전환점 3개가 이 시스템의 사고 과정을 보여준다:

- **v0.1** — mm-broker `.claude/`를 감사(audit)해 82개 항목 전부를 4단계로 분류. 애매한 것 5건은 "보류 안건"으로 팀장 판단에 넘김.
- **v0.2 도구 불문 canonical 전환** — Claude 전용 구조를 버리고 "원본=순수 md, 도구 포맷=파생물" 구조로 전환. migrate.mjs가 이 변환을 담당.
- **v0.3 강제-우선 전환** — 비개발자 대상임이 명확해지면서 "안내"에서 "게이트"로 무게중심 이동. precommit-gates.mjs, rules.json, ci-gates.yml이 이때 등장.

이후 **Codex 파리티 작업**(2026-06-22 실측 검증)으로 `.codex/` 어댑터가 완성되어, "같은 위반을 Claude와 Codex 양쪽이 편집 시점에 차단"함을 실측으로 확인했다 (PARITY.md).

## 5. 무엇이 특징적인가 (다른 접근과 비교)

| 흔한 접근 | 이 라이브러리의 접근 |
|---|---|
| 도구별로 설정 따로 관리 (CLAUDE.md, .cursorrules, …) | canonical md 하나 + 어댑터 자동 생성 |
| 규칙을 훅 코드에 하드코딩 | rules.json 선언 + 엔진 3개가 공유 |
| "AI에게 하지 말라고 지시" | 지시(1겹) + 로컬 게이트(2겹) + 서버 차단(3겹) |
| 설치 문서를 사람이 따라 함 | AGENTS.md에 **AI용 설치 프로토콜** 내장 — AI가 직접 설치하고 검증까지 |
| 효과를 감으로 판단 | harness-probe가 훅 발사·차단·스킬 참조를 `.harness-metrics.jsonl`로 계측 |

특히 **"AI용 설치 프로토콜"**(레포 루트 AGENTS.md)은 눈여겨볼 패턴이다: 클론된 레포를 AI로 열고 "설치해줘"라고만 하면, AI가 ① 대상 경로 질문(유일한 필수 질문) ② package.json 기반 스택 자동 추론 ③ 설치 실행 ④ 4단계 검증 ⑤ 쉬운 한국어 보고까지 스스로 수행하도록 문서가 설계돼 있다. 문서가 곧 실행 가능한 온보딩 자동화다.
