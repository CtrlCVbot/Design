# 차주용 앱 기획을 위한 관리자 final-handoff 분석

## 1. 목적

이 문서는 `화물관리 > 오더 접수/수정` 최종 인계 패키지를 차주용 모바일 앱 관점에서 재해석한 분석 문서입니다.

차주용 앱은 관리자/주선사용 시스템에서 등록되고 배차된 화물을 모바일로 확인하고, 차주가 `상차완료`, `하차완료` 진행 상태를 직접 업데이트하며, 주선사가 운행완료와 정산 가능 판단을 빠르게 할 수 있도록 돕는 역할을 맡습니다. 하차 담당자 확인 링크와 서명 페이지는 MVP 이후 단계로 분리합니다.

## 4. 차주 앱에 연결해야 할 데이터 모델

| 모델 | 관리자 기준 역할 | 차주 앱 사용처 | 차주 앱 노출 정책 |
| --- | --- | --- | --- |
| `CargoOrder` | 화물ID, 상태, 처리시간, 플래그 | 배차 목록, 배차 상세, 운행 상태 | 필수 노출. 내부 운영용 상태와 차주 표시 상태는 분리 |
| `Location` | 상차/하차 주소, 상세주소, 담당자, 연락처 | 상차지/하차지 상세, 담당자 연락 | 주소/담당자/연락처는 개인정보 마스킹 정책 필요 |
| `VehicleRequirement` | 톤수, 차종, 대수, 독차/혼적 | 배차 상세 화물 조건 | 차주 운행 판단에 필요한 수준으로 노출 |
| `CargoDetail` | 품목, 실중량, 특이사항 | 배차 상세, 운행 전 확인 | 특이사항은 상하차 작업에 직접 중요 |
| `Pricing` | 청구금, 배차금, 수수료, 합계 | 운임 표시, 정산/송금 확인 | 차주에게는 배차금/송금 대상 금액 중심 노출 |
| `PricingAdjustment` | 조정금, 사유, 담당자, 시각 | 정산 상세, 운임 변동 이력 | 차주에게 공개 가능한 조정 사유만 노출 |
| `DispatchManager` | 배차 담당자명, 팀, 변경 권한, 이력 | 상세 문의, 긴급 연락 | 차주가 연락 가능한 대표 정보 필요 |
| `DriverAssignment` | 차주명, 전화, 차량번호, 차종, 톤수 | 로그인 차주 매칭, 배차 대상 판정 | 본인 차량/프로필 정보와 매칭 필요 |
| `SettlementDocument` | 인수증, 계산서, 위탁증 상태 | 운행완료, 정산/송금 조회 | 하차 담당자 확인 결과 연결은 Phase 2에서 다룸 |

## 5. 관리자 상태와 차주 앱 상태 매핑 초안

관리자 최종 인계 문서의 `new-*`, `submit-*` 상태는 등록 화면 내부 상태입니다. 차주 앱에는 실제 운송 업무 상태를 별도로 정의해야 합니다.

| 차주 앱 상태 | 의미 | 관리자/주선사용 시스템 연결 | 차주 앱 액션 |
| --- | --- | --- | --- |
| `assigned` | 차주에게 배차 확정됨 | `CargoOrder` + `DriverAssignment` 확정 | 상세 확인, 배차 수락/문의 |
| `accepted` | 차주가 배차를 확인 또는 수락함 | 상태 이력에 차주 확인 시각 저장 | 상차지 이동 준비 |
| `pickup_done` | 상차완료 | 등록 화물 상태에 상차완료 공유 | `상차완료` 버튼 |
| `dropoff_done` | 차주가 하차완료를 직접 등록함 | 등록 화물 상태에 하차완료 공유 | `하차완료` 버튼 |
| `operation_completed` | 운행완료 | 주선사가 특이사항 없음을 확인하면 매입정산 가능 후보 | 정산 대기 확인 |
| `settlement_pending` | 주선사 정산 처리 전 | 주선사 정산 검토 상태 | 내역 조회 |
| `paid` | 송금 완료 | 송금일/금액 확정 | 송금 내역 확인 |
| `issue_hold` | 사고, 파손, 금액 이견, 하차 이슈 등 보류 | 운행완료/정산 가능 상태 차단 | 특이사항 제출/문의 |

## 6. 차주 앱 디자인 초안 분석

| 현재 초안 화면 | 확인 내용 | 기획 반영 방향 |
| --- | --- | --- |
| Home | 검색형 구조와 하단 탭 중심. 일부 여행 앱 템플릿 문구가 남아 있음 | 차주 앱 MVP에서는 홈보다 `내 배차` 중심 첫 화면 권장 |
| Results | `서울 -> 부산`, 운임순/거리순/상차순/평점순 필터, 배차 카드 구조 | 실제 배차 받은 목록과 신규 오더 탐색을 분리해야 함 |
| Ticket/Detail | `배차 상세`, 배차번호, 화주, 상하차지, 차량번호, 운임, 바코드형 영역 | 배차 상세 기반으로 상차완료/하차완료 버튼과 상태 타임라인을 추가 |

현재 초안은 시각적으로는 모바일 카드 구조가 잡혀 있지만, 업무 관점에서는 `배차 검색/받기`와 `배차받은 내역 관리`가 섞여 있습니다. 이번 기획의 1차 목표는 차주가 이미 배차된 화물 내역을 확인하고 운송 상태를 처리하는 앱이므로, MVP 첫 화면은 `내 배차` 또는 `오늘 운행`이 더 적합합니다.

## 7. 차주 앱에서 반드시 이어야 할 연결 지점

| 연결 지점 | 이유 | 필요한 데이터 |
| --- | --- | --- |
| 배차 확정 이벤트 | 차주 앱에 배차 목록이 생기는 시작점 | `CargoOrder.id`, `DriverAssignment`, 배차 담당자 |
| 운송 상태 업데이트 | 차주가 상차/하차 진행 상태를 공유해야 함 | 상태값, 차주ID, 위치/시각, 사진 여부 |
| 하차완료 등록 | 차주가 하차 도착과 작업 완료를 주선사에게 빠르게 공유 | 상태값, 차주ID, 하차완료 시각, 위치/사진/메모 optional |
| 하차 담당자 확인 | Phase 2 인수증 대체 또는 보조 증빙 역할 | 하차 담당자명, 전화, 서명, 확인 시각, IP/device optional |
| 운행완료 전환 | 매입정산 가능 여부 판단의 기준점 | 하차완료 등록, 특이사항 없음, 보류 사유 없음 |
| 정산/송금 조회 | 차주 신뢰와 문의 감소 | 운임, 조정금, 송금 상태, 송금일 |
| 특이사항/이슈 보류 | 무조건 운행완료 처리되는 위험 방지 | 사고/파손/미수령/금액 이견/사진/메모 |

## 8. MVP 관점의 핵심 결론

| 결론 | 설명 |
| --- | --- |
| 첫 화면은 `내 배차` 중심이어야 함 | 차주에게 가장 중요한 것은 검색보다 오늘/이번 주 배차 수행입니다. |
| `배차 상세`는 운행 작업 화면이어야 함 | 단순 티켓 조회가 아니라 상차완료, 하차완료, 특이사항 보고가 들어가야 합니다. |
| MVP 상태 변경은 직접 처리합니다 | 차주는 하차 담당자 링크 없이 `상차완료`, `하차완료`를 직접 등록하고, 주선사가 이후 운행완료/정산 가능 여부를 판단합니다. |
| 정산은 조회 중심이어야 함 | 정산 확정은 주선사용 시스템 권한이고, 차주 앱은 송금/정산 상태를 확인하는 역할이 적합합니다. |
| 하차 담당자 링크 보안은 Phase 2 정책입니다 | 하차 담당자 링크가 인수증 역할을 하려면 만료, 1회성, 확인 기록 보관 정책이 필요하지만 MVP 범위에서는 제외합니다. |
