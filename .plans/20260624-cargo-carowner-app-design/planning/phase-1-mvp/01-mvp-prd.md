# Phase 1 MVP PRD

## 1. Overview

Phase 1 MVP는 차주가 모바일에서 본인에게 배차된 운행을 확인하고, 현장에서 `상차완료`와 `하차완료`를 직접 등록하며, 운행 내역과 정산/송금 상태를 조회하는 차주용 업무 앱 범위다.

이번 범위는 기능, UI/UX, 유저플로우, 와이어프레임 중심이다. API, DB, 백엔드 상세 계약은 다루지 않는다.

## 2. Problem Statement

차주는 배차 후 상차지, 하차지, 시간, 담당자, 운임, 특이사항을 현장에서 빠르게 확인해야 하지만, 전화나 문자 확인에 의존하는 비중이 높다. 운행 상태가 늦게 공유되면 주선사는 운행완료 판단과 정산/송금 준비를 수동으로 확인해야 한다.

특히 `하차완료`와 `운행완료`가 같은 의미로 오해되면 권한과 책임이 섞인다. MVP에서는 차주가 현장 작업 완료 사실을 `하차완료`로 등록하고, 주선사가 특이사항과 내부 기준을 확인해 `운행완료` 및 정산 가능 여부를 판단하는 구조를 명확히 해야 한다.

## 3. Goals & Non-Goals

### Goals

| --- | --- | --- |
| 배차 확인 단순화 | 차주가 앱 첫 화면에서 오늘/예정/진행 배차를 확인한다. | Phase 1A |
| 현장 상태 공유 | 차주가 `상차완료`, `하차완료`를 직접 등록한다. | Phase 1A |
| 상태 판단 분리 | 차주 등록 상태와 주선사의 `운행완료` 판단 권한을 분리한다. | Phase 1A |
| 정산 문의 감소 | 차주가 정산 홈에서 월별 송금완료/미정산/보류 금액을 확인하고, 운행 내역에서 건별 정산 상태를 조회한다. | Phase 1B |

| 제외 항목 | 후속 위치 | 이유 |

| 하차 담당자 확인 링크 | Phase 2 | MVP는 차주의 직접 상태 등록과 조회에 집중한다. |

## 6. UX Requirements

| 항목 | 요구사항 |
| --- | --- |
| 모바일 우선 | 360px~430px 폭의 모바일 화면을 기준으로 단일 열 구조를 사용한다. |
| 첫 화면 | `내 배차 목록`을 핵심 홈으로 두고, 검색형 신규 오더 화면은 노출하지 않는다. |
| CTA 정책 | 현재 상태에서 가능한 1차 액션만 강하게 표시한다. 예: 운행 준비 상태는 `상차완료`, 상차완료 상태는 `하차완료`. |
| 확인 modal | 상태 변경 버튼은 즉시 완료하지 않고, 장소/시간/상태 의미를 다시 보여주는 확인 modal을 거친다. |
| 상태 badge | `배차됨`, `운행 준비`, `상차완료`, `하차완료`, `운행완료`, `정산대기`, `송금완료`, `보류`를 명확한 문구와 색상으로 구분한다. |
| 권한 안내 | `하차완료` 이후에는 "주선사 확인 후 운행완료/정산대기로 전환됩니다."처럼 판단 주체를 알려준다. |
| 정산 표시 | 금액 정보는 차주에게 필요한 배차금, 조정 요약, 송금금액, 송금일, 송금상태만 노출한다. |
| 정산 탭 | 하단 `정산` 탭은 특정 운행의 `정산/송금 상세`를 직접 열지 않는다. `정산 홈`에서 월별/기간별 집계와 다음 송금 예정, 미정산/보류 요약을 먼저 보여준다. |
| 개인정보 최소 노출 | 담당자 연락처는 업무 연락 목적 문구와 함께 노출하고, 계좌 정보는 Phase 1에서 표시하지 않는다. 주소는 운행 수행에 필요한 수준으로만 표시한다. |
| 접근성 | CTA와 badge는 색상만으로 의미를 전달하지 않고 텍스트를 함께 표시한다. 버튼 터치 영역은 모바일에서 44px 이상을 권장한다. |

## 7. Technical Considerations

| 항목 | 고려사항 |
| --- | --- |
| 범위 제한 | API, DB, 백엔드 상세 계약은 이번 문서 범위에서 제외한다. |
| 상태 모델 | 기획상 표시 상태는 `assigned`, `driver_confirmed`, `pickup_done`, `dropoff_done`, `operation_completed`, `settlement_pending`, `paid`, `issue_hold`를 기준으로 한다. |
| 권한 경계 | 차주는 `상차완료`, `하차완료`를 등록한다. 주선사는 `운행완료`, 정산 가능 여부, 송금 상태를 확정한다. |
| 이벤트 이력 | 상태 변경과 특이사항 보고는 추후 구현에서 시각, 주체, 메모, 사진 여부를 이력으로 남길 수 있어야 한다. |
| 개인정보 | 담당자 연락처, 주소, 송금 정보는 최소 노출과 마스킹 정책이 필요하다. 상세 정책은 후속 구현 단계에서 확정한다. |
| 정산 노출 | 내부 청구금, 수수료, 내부 메모, 송금 계좌 정보는 차주 앱 노출 대상이 아니다. |

## 8. Milestones

| 단계 | 범위 | 완료 기준 |
| --- | --- | --- |
| Phase 1A 운행 수행 | 내 배차 목록, 배차 상세, `상차완료`, `하차완료`, 상태 타임라인 | 차주가 목록에서 상세로 들어가 상차/하차 상태를 등록하는 flow가 정의된다. |
| Phase 1B 운행/정산 조회 | 운행 내역, 기간/월별 조회, 정산 홈, 정산/송금 상태, 송금일, 송금금액 | 차주 노출 가능한 정산 항목, 월별 집계, 건별 상세 이동이 정의된다. |
| Phase 1C 예외 처리 | 특이사항 보고, 사진/메모 optional, 보류 상태 표시, 담당자 문의 | 예외 제출 후 보류 표시와 문의 동선이 정의된다. |

## 9. Risks & Mitigations

| 리스크 | 영향 | 대응 |
| --- | --- | --- |
| `하차완료`와 `운행완료` 오해 | 차주가 정산 확정까지 완료된 것으로 이해할 수 있다. | 화면 문구와 badge에서 차주 등록 상태와 주선사 판단 상태를 분리한다. |
| 정산 정보 과다 노출 | 내부 청구금, 수수료, 내부 메모 노출로 운영 리스크가 생길 수 있다. | 차주 노출 항목을 배차금, 조정 요약, 송금금액, 송금일, 송금상태로 제한한다. |
| 정산 탭 의미 부족 | 하단 `정산` 탭이 특정 건 상세로 바로 열리면 `운행 내역`과 기능이 중복된다. | `정산` 탭은 월별/기간별 정산 홈으로 재정의하고, 건별 상세는 하위 화면으로 둔다. |
| 특이사항 미보고 | 파손/수량 차이 건이 바로 운행완료로 오해될 수 있다. | 배차 상세에 `특이사항 보고` CTA를 상시 제공하고, 제출 시 보류 후보 표시를 명확히 한다. |
| 현장 입력 부담 | 사진 필수화가 상태 등록 속도를 늦출 수 있다. | Phase 1C에서 사진은 optional로 두고 메모 중심 보고를 허용한다. |
| 디자인 초안의 검색형 구조 혼재 | MVP 목적이 신규 오더 탐색으로 흐릴 수 있다. | 첫 화면을 `내 배차 목록`으로 고정하고 배차 검색/수락을 제외한다. |

## 10. Success Metrics

| 지표 | 목표 | 측정 방법 |
| --- | --- | --- |
| 상태 등록 완료율 | Phase 1 출시 후 배차 상세 진입 건의 `상차완료`/`하차완료` 등록 완료율을 추적하고 기준값을 수립한다. | 상태 이벤트 전환율 |
| 상태 확인 문의 감소 | 출시 후 4주 내 상차/하차 진행 확인 전화 또는 메시지 감소율을 측정한다. | 운영 문의 유형 집계 |
| 정산 조회 이용률 | 송금 관련 문의 전 앱 정산/송금 상세 조회 비율을 측정한다. | 정산 상세 조회 로그 |
| 정산 홈 이용률 | 송금 관련 문의 전 정산 홈 방문과 상태별 정산 묶음 조회 비율을 측정한다. | 정산 탭 진입 로그, 정산 묶음 클릭 로그 |
| 보류 식별 정확도 | 특이사항 제출 건이 주선사 검토 대상으로 식별되는 비율을 추적한다. | 특이사항 제출 후 상태 전환 확인 |
| 핵심 flow 완주 시간 | 목록 진입부터 상태 등록까지 걸리는 시간을 측정해 UX 개선 기준값을 만든다. | 화면 체류 및 완료 이벤트 기준 |
