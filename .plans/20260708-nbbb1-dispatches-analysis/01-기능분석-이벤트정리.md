# nbbb1.vercel.app/dispatches — 기능 분석 및 이벤트 정리

- 분석일: 2026-07-08
- 대상: https://nbbb1.vercel.app/dispatches (T-Logis PARTNERS TMS · 배차 관리)
- 분석 방법: Playwright(Edge headless)로 전 인터랙션 직접 조작 + 렌더 DOM/스타일 추출
- 기술 특성: Vite 빌드 SPA(단일 JS 번들, API 호출 없음 — **전 데이터·로직이 클라이언트 목업**). 외부 연동은 카카오(다음) 우편번호 검색 1종.

## 1. 화면 구조 개요

```
┌──────┬───────────────────────────┬─────────────────────────────────────┐
│ 사이드바 │  좌측 패널 (3개 모드 전환)      │  우측 패널 (2개 모드 전환)                │
│ 70px  │  ① 운행 등록 (기본)          │  ① 운행 내역 리스트 (기본)               │
│ 아이콘   │  ② 차량/차주 배정            │  ② 주요 상하차지 추천 목록                │
│ 내비    │  ③ 대화방                  │  * 접기 시 우측 가장자리 세로 탭으로 축소       │
└──────┴───────────────────────────┴─────────────────────────────────────┘
```

핵심 설계 패턴: **페이지 이동 없이 좌/우 패널의 모드 전환으로 모든 업무(등록→배차→소통→정산정보 관리)를 한 화면에서 처리**하는 콕핏(cockpit)형 TMS.

### 2-C. 행 확장(아코디언) 상세 — 인라인 워크스페이스

확장 영역은 3개 카드 + 조건부 AI 배너로 구성:

**(c-0) AI 실시간 배차 분석 어드바이저** — `배차중` 상태에서만 표시. 3단계:
| 상태 | 신호 | 내용 구성 |
|---|---|---|
| 🟢 배차 원활 | 초록 | "가용 차주 충분… 운임 유지 권장" |
| 🟡 배차 지연 주의 | 주황 | "매칭 지연 주의… 운임 1~2만원 보강 또는 대기시간 연장 제안" |
| 🔴 배차 지연 우려 | 빨강 | "경쟁 화물 집중 심각… 운임 긴급 보강 권고" |

공통 지표 3종: `대기 차주(인근 N명)` / `경쟁 화물(N건)` / `구간 평균 운임(N원, 적정/유사/평균 이하 판정)`

**(c-1) 상세 정보 카드**: 거래처명, 상차지·하차지 주소+일시, 정산 수단/수수료, 정산 예정일, 화물품목, 기사 전달사항 및 메모. 헤더에 `+ 경유지 추가`, `대화방` 버튼.

**(c-2) 운임 및 수수료 정보 수정 카드**:
| # | 요소 | 이벤트 | 동작 |
|---|---|---|---|
| C1 | `운`/`수` 토글 | click | 스테퍼 적용 대상(운임/수수료) 선택 |
| C2 | 스테퍼 `+1만/+5천/+1천` / `-1만/-5천/-1천` | click | 선택 대상 금액 증감 (300,000 → +1만 → 310,000) |
| C3 | 운임/수수료 input | input | 직접 수정 (콤마 포맷) |
| C4 | `수정` | click | 적용 + 우상단 토스트 **"운임 및 수수료 정보가 수정 완료되었습니다!"** + 리스트 행 운임 즉시 갱신 |

**(c-3) 차주 배정 및 상태 제어 카드**:
| # | 요소 | 이벤트 | 동작 |
|---|---|---|---|
| C5 | 차량번호/차주명/연락처 input | input | 수동 배정 정보 입력 (배정된 행은 프리필 + 연락처 옆 `대화방` 버튼) |
| C6 | `차량배정` | click | **좌측 패널이 "차량/차주 배정" 모드로 전환** (→ 2-D) |
| C7 | 상태 버튼 6개 (배차완료/상차완료/하차완료/운행완료/배차대기/배차취소) | click | 즉시 상태 변경. **현재 상태 버튼이 상태 색으로 하이라이트**. 변경 시 상태 필터·날짜 필터의 건수가 전부 실시간 재계산되고, 현재 필터에 안 맞으면 행이 리스트에서 이탈 |

### 2-D. 차량/차주 배정 패널 (좌측 패널 모드 2)

| 구성 | 내용 |
|---|---|
| 헤더 | "차량/차주 배정" + `닫기`(운행등록 복귀) |
| 선택된 배차 정보 | 화주, 운임, 노선, 차량 스펙 요약 카드 |
| ⭐ AI 최적 추천 차주 (최대 3명) | 추천 사유 태그 포함 카드 (아래) |
| 전체 배정 가능 차주 목록 | 동일 카드 형식 ~20명 |

**차주 카드 구성**: 차주명 + 소속 구분 뱃지(`소속`/`외부`/`지입`) + (`⭐ AI 추천`) + `선택 배정` 버튼 / 차량번호 / **현위치 거리 (운행중 차주는 "연계지 거리" — 하차 예정지 기준 거리로 대체되는 로직)** / 스펙 / 월 총운임 / 추천 사유 태그(동종화물 유경험·상차지 인접·현재 대기중) / 현재 상태 및 주소(대기중(배정 가능) 또는 운행중(구간)+하차지) / 계좌 정보(은행+계좌번호) / 최근 배차 이력(구간+화주+`완료/취소/진행중` 뱃지, 없으면 "배차 이력 없음").

| # | 이벤트 | 동작 (실측) |
|---|---|---|
| D1 | `선택 배정` click | 해당 운행에 차주 배정 → **상태 자동 전환: 배차중 → 배차완료** (배차중 10→9, 배차완료 10→11 실측) |
| D2 | `닫기` click | 운행 등록 모드 복귀 |

### 2-E. 주요 상하차지 추천 목록 패널 (우측 패널 모드 2)

거래처 `검색` 또는 자주 쓰는 구간 `...`으로 진입. 헤더: "주요 상하차지 추천 목록" + `닫기`. 안내문: "[거래처명] 거래처의 주요 상하차지 추천 주소 목록입니다."

| 섹션 | 내용 | 행 이벤트 |
|---|---|---|
| ⚡ 운행데이터기반 주요구간 (상하차 동시선택) | 순위 1~5 구간 | `선택` → 상차지+하차지 동시 입력 |
| 📍 거래처 저장기반 주요상차지 | 순위 1~5 주소 | `선택` → 상차지 입력 |
| ※ 거래처 저장기반 주요하차지 | 순위 1~5 주소 | `선택` → 하차지 입력 |

### 2-F. 대화방 패널 (좌측 패널 모드 3)

행 상세의 `대화방` 버튼으로 진입.

| # | 요소 | 이벤트 | 동작 |
|---|---|---|---|
| F1 | 헤더 `전체 화면` | click | **/chats 페이지로 이동** |
| F2 | 헤더 `운행등록으로 복귀` | click | 좌측 패널 운행 등록 모드 복귀 |
| F3 | 대화 입력 + `전송` | click | 내 메시지 버블(파랑, 우측 정렬) 즉시 추가 + 타임스탬프 |
| F4 | 하단 아이콘 2개 (문서/즐겨찾기) | click | 첨부·템플릿성 액션 (목업 수준) |

상대 정보 바: `{거래처명} | 거래처 | {전화번호}`. 시스템 인사 메시지 자동 표시.

## 3. 상태 모델 (배차 라이프사이클)

```
배차대기(배차중) ──차주 배정──▶ 배차완료 ──▶ 상차완료 ──▶ 하차완료 ──▶ 운행완료
      │
      └────────────────▶ 배차취소 (어느 단계에서든 상태 버튼으로 직접 전환 가능)
```

- 상태 버튼 6개로 **비순차 전환도 허용**(목업 특성) — 실제 구현 시 전이 규칙 정의 필요.
- `배차중` 상태에서만: 신호등 난이도 점 + AI 어드바이저 배너 활성.
- 상태별 뱃지 색: 배차중=주황, 배차완료=파랑, 상차완료·하차완료·운행완료=청록(성공), 배차취소=빨강.

## 4. 데이터 연쇄(리액티비티) 특징

1. 거래처 선택 → 자주 쓰는 구간 칩 갱신 → 구간 선택 → 주소·운임 추천(최근/최빈) 노출 — **입력이 진행될수록 다음 입력을 좁혀주는 추천 체인**.
2. 상태 변경·등록·운임 수정 → 리스트 행, 상태 필터 카운트, 날짜 필터 카운트가 전부 즉시 재계산.
3. 날짜 필터 카운트는 상태 필터 적용 결과 기준으로 교차 계산(반대도 동일).

## 5. 반응형

- **~390px(모바일)**: 단일 컬럼. 운행 내역이 상단, 테이블은 상태/화주/노선 3컬럼으로 축소, 상태 필터 칩 숨김(검색만 노출).
- 데스크톱 1440~1600px: 좌 ~600px 고정 + 우 유동 2패널.

## 6. 관찰된 한계(목업 특성 — 벤치마킹 시 참고)

- 새로고침 시 모든 변경 소실 (클라이언트 상태만 존재, 영속화 없음)
