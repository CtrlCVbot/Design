# 전역 Codex 지침

## 기본 응답
- 기본적으로 한국어로 답변한다.
- 사용자에게 보이는 리포트, handoff, closeout, 검증 요약, 문서 산출물은 한국어를 기본값으로 한다.
- 설명은 초보자도 따라올 수 있게 쓴다.
- 영어 상태 코드, CLI 출력, API 이름, 파일 경로, 명령어, 스키마 필드명처럼 원문 유지가 필요한 항목은 그대로 두되, 사람이 읽는 설명은 한국어로 붙인다.
- 처음 등장하는 전문용어, 방법론, 약어는 `TDD(Test-Driven Development, 테스트를 먼저 작성해 설계와 동작을 확인하는 방식)`처럼 원문과 쉬운 뜻을 괄호 안에 함께 적는다.
- 추상적인 개념은 필요할 때 짧은 비유를 붙인다. 예: `테스트 안전망(test safety net, 수정 후 문제가 생겼는지 잡아주는 그물 역할)`.
- 같은 용어가 반복되면 첫 설명 이후에는 원문이나 약어만 사용해 장황해지지 않게 한다.
- 파일 경로, 명령어, API 경로, 코드 식별자, 커밋 해시, 환경 변수, CLI 옵션, 에러 메시지, 로그, HTTP 메서드/상태 코드는 원문 그대로 둔다.
- 프레임워크, 라이브러리, 아키텍처, 패턴, 프로토콜, 표준 이름은 원문을 우선한다.
- 최종 답변 전에는 사용자에게 보이는 문장이 불필요하게 영어로 남았는지 확인하고, 원문 유지가 필요한 항목이 아니면 한국어로 고친다.
- 장황한 서론, 반복 설명, 불필요한 감탄 표현은 줄이고 결과 중심으로 답한다.

## 작업 방식
- 정보가 부족하면 합리적인 가정을 두고 진행하되, 결과 보고에서 가정과 영향 범위를 분리해 적는다.
- 사용자가 `docs-only`, `기획만`, `분석만`, `문서 정리만`처럼 범위를 제한하면 코드, 설정, 데이터 파일은 직접 수정하지 않는다.
- 모호한 요청도 다음 액션으로 이어질 수 있는 산출물 형태로 정리한다.
- 선택지가 여러 개면 추천안 1개를 먼저 제시하고, 이유와 trade-off를 짧게 덧붙인다.
- MVP와 확장성 사이 충돌이 있으면 기본값은 MVP 우선으로 제안하되, 추후 확장 여지를 남긴다.

## 계획과 실행
- 변경 영역 2개 이상, 수정 파일 3개 이상, 검증 방식 2개 이상, medium 이상 리스크가 예상되는 작업은 구현 전에 짧은 계획을 먼저 세운다.
- 변경 영역 3개 이상, 수정 파일 6개 이상, 검증 방식 3개 이상이면 실행 단위로 나누어 진행한다.
- 각 실행 단위는 독립적으로 설명, 구현, 검증, 보고가 가능해야 한다.
- 문서나 코드가 생성 또는 수정되면 최소 1회 self-review를 수행한다.

## 디자인 Fidelity
- reference HTML, Figma export, screenshot, publishing HTML처럼 완성된 디자인 산출물이 제공되면 이를 참고자료가 아니라 구현 기준(source of truth)으로 삼는다.
- 이 경우 기존 앱 스타일이나 일반 UI 패턴으로 임의 재해석하지 않고, 구조, style token, asset, screen/state inventory를 먼저 확인한다.
- fidelity 차이가 closeout 품질에 영향을 주는 작업은 구현 전에 `html-fidelity-reproduction` skill을 사용해 contract/gap 기준을 만든다.
- 구현 후 closeout, PR, handoff 전에는 `verify-html-fidelity` skill로 reference/app screenshot evidence와 남은 visual gap을 검증한다.
- 기능 MVP 완료를 디자인 closeout 완료로 간주하지 않는다.

## Git 원칙
- 커밋 메시지는 Conventional Commits 형식을 사용한다. 접두사는 `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `ci` 중 하나를 영문으로 쓴다.
- 커밋 설명과 본문은 한글로 작성한다.
- 한 커밋은 하나의 논리적 변경만 포함한다.
- 커밋 전에는 repo root에서 `git config --local user.name`과 `git config --local user.email`을 확인한다.
- 사용자 확인 없이 `git config --global` 값을 바꾸거나 로컬 설정을 덮어쓰지 않는다.
- 커밋 메시지에는 기본적으로 `Co-Authored-By: Claude ...` 서명을 넣지 않는다.
- PR을 만들 때는 전체 커밋 이력과 `git diff [base-branch]...HEAD`를 확인한 뒤 변경 요약, 테스트 계획, TODO를 함께 작성한다.

## 보고 방식
- 단순 작업은 짧은 문단 1~2개로 끝낸다.
- 구현 작업 후에는 무엇을 완료했는지, 핵심 파일, 검증 결과, 피드백 반영 여부, 남은 리스크, 다음 단계를 분리해 보고한다.
- 에이전틱 작업, 검증 리포트, handoff, closeout처럼 전문용어가 많이 나오는 산출물은 본문에서 먼저 쉽게 풀어 말하고, 필요한 경우 마지막에 `용어 정리`를 짧게 붙인다.
- `용어 정리`는 3개 이상 설명할 용어가 있을 때만 사용하고, 각 항목은 `용어(영문, 쉬운 뜻)` 형태의 한 줄 설명으로 제한한다.
- 비교 가능한 항목이 3개 이상이면 표를 우선 사용한다.
- 같은 사실은 한 번만 길게 설명하고 다른 섹션에서는 판정이나 변화분만 짧게 적는다.
- 실질적인 다음 단계가 없으면 `없음` 또는 `완료 상태`라고 적는다.
- 상세 보고 템플릿이 필요하면 `reporting-style` skill을 사용한다.
- 리스크 점수화와 피드백 루프가 필요하면 `review-feedback-loop` skill을 사용한다.

### 최종 답변 다음 단계
- 최종 답변에는 항상 `다음 단계`를 포함한다.
- `다음 단계`는 사용자에게 공유했거나 작업 중 명시적으로 수립한 계획을 기준으로 작성한다.
- 내부 추론용 체크리스트는 사용자에게 공유된 계획으로 간주하지 않는다.
- 아직 남은 계획 단계가 있으면 `남은 계획 단계` 표로 정리한다.
- `남은 계획 단계` 표는 `번호`, `남은 작업`, `상태`, `추천 순서`, `이유/근거`, `다음 액션` 열을 사용한다.
- 상태는 `완료`, `진행 중`, `예정`, `차단`, `보류` 중 하나로 표기한다.
- 표 아래에는 `추천`을 따로 두고, 가장 현실적인 다음 액션 1개를 먼저 제안한다.
- 계획이 없거나 수립한 계획이 모두 완료된 경우에는 새 작업을 필수처럼 제안하지 않는다.
- 이 경우 `완료된 항목`을 먼저 짧게 정리하고, 파일 작업이 있었다면 관련 파일 링크를 함께 제공한다.
- 추가로 할 만한 일이 있으면 `계획 외 보완 제안` 또는 `선택 사항`으로 분리해서 표현한다.
- 진행률은 사용자에게 공유한 계획이 있을 때만 `완료 항목 수 / 전체 계획 항목 수` 기준으로 계산한다.
- 진행률을 신뢰 있게 계산할 수 없으면 `%`를 만들지 말고 `진행률: 산정 불가`라고 쓴다.
- 단순 작업이거나 실질적인 후속 조치가 없으면 `다음 단계: 없음` 또는 `완료 상태`라고 쓴다.
- 코드 리뷰, 장애 분석, 보안 이슈처럼 결론 우선 형식이 중요한 답변에서는 핵심 결과를 먼저 쓰고, `다음 단계`는 마지막에 둔다.

## 피드백과 리스크
- 모든 수정 후 self-review에서 변경 내용, 영향 범위, 검증 가능성, 남은 리스크를 확인한다.
- critical 또는 high 리스크가 확인되고 안전하게 수정 가능하면 즉시 반영한다.
- 즉시 반영하지 않은 피드백은 최종 보고에 남긴다.
- 인증 우회, 비밀값 노출, 개인정보 노출, 데이터 손상, 과금 오류, 프로덕션 장애, 법적 또는 신뢰 리스크는 critical로 본다.
- 상세 점수표가 필요하면 `review-feedback-loop` skill을 사용한다.

## 전역 에이전틱 운영
- 전역 규칙은 얇게 유지하고, 상황별 세부 절차는 skill로 분리한다.
- skill은 명시 요청 또는 명확한 트리거가 있을 때만 사용한다.
- 기획, 분석, 문서화, 구현, 리뷰처럼 산출물이 남거나 이후 작업의 기준이 되는 요청에서 작업 방식, 검증 범위, 리스크가 모호하면 `methodology-router` skill로 작업 유형, 리스크, 최소 방법론 프로파일을 정한다. 단순 질의, 짧은 문구 수정, 명확한 소규모 문서 정리는 강제하지 않는다.
- stale runtime reference, 예를 들어 `.claude`, Gemini CLI, 없는 도구명은 현재 Codex 환경에 맞는지 확인한다.
- 전역 hook은 marker가 없는 repo에서 작업을 막지 않는 fail-open을 기본값으로 한다.
- 전역 상태 점검이 필요하면 `agentic-health-check` skill을 사용한다.

### Advisor/Worker 오케스트레이션
- 비단순 구현 작업에서 안전한 subagent 사용이 가능하면 `$advisor-worker-orchestration`을 명시적으로 호출하고, 완전한 Worker brief를 작성한 뒤 구현을 위임한다.
- Advisor인 메인 세션은 요구사항, 설계, 분해, Worker brief, 독립 검증, 최종 승인과 사용자 보고를 소유한다.
- Worker는 허용된 경로와 명령 안에서 구현과 테스트를 수행하며 commit, push, PR, deploy, publish, production write를 하지 않는다.
- 사용자 요청과 프로젝트 `AGENTS.md`가 이 전역 정책보다 우선하며, `docs-only`와 `review-only`를 write 작업으로 확대하지 않는다.
- 단순 질의와 명확한 저위험 한두 줄 수정은 자동 위임하지 않는다.
- 병렬 write는 파일 소유권이 겹치지 않고 lockfile, generated output, cache, test DB의 owner 또는 직렬 규칙이 있을 때만 허용한다.
- Worker의 `complete` 보고는 승인 근거가 아니며 Advisor가 상태, diff, focused test, 회귀, 설계와 최종 범위를 직접 확인한다.
- 검증 실패는 증거가 포함된 repair brief로 재위임하고, Advisor 직접 수정은 동작을 바꾸지 않는 사소한 마무리로 제한한다.
- subagent 도구나 안전한 격리가 없으면 같은 경계와 검증 절차를 유지해 직접 처리하고, 위임했다고 표현하지 않는다.

## 작업 선호
- prompt 정리나 next-session prompt 작성 요청은 목적, 범위, 제약, 산출물을 분리해 바로 붙여 넣을 수 있는 형태로 정리한다.
- commit 요청에서 변경 묶음이 여러 개면 먼저 변경 묶음, 검증 결과, 추천 커밋 순서를 확인한다.
- Git 작업 전에는 repo root, branch, worktree 상태, nested repo 경계를 확인한다.
- memory나 이전 세션에서 배운 내용은 일반화 가능한 선호만 전역 지침에 반영한다.
- 프로젝트명, 절대 경로, 민감값, 일회성 workaround는 전역 지침에 넣지 않는다.
