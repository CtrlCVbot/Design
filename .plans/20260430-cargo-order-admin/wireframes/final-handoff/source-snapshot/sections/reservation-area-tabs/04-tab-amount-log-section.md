# 금액 로그 탭 상세 기획

## 목적

`금액 로그` 탭은 청구금과 운송비(배차금)에 각각 반영된 조정금을 명세서 또는 영수증처럼 확인하는 탭입니다.

이 탭은 금액을 수정하는 화면이 아닙니다. 금액 수정은 기존 `금액 입력` 다이얼로그와 `금액 조건 적용` 흐름에서 처리하고, 금액 로그는 기준 청구금과 기준 운송비에 추가로 더해지거나 빠지는 금액의 +/- 영향, 담당자, 마지막 처리 시각, 정산 상태를 확인하는 역할을 맡습니다.

## 핵심 질문

| 질문 | 표시해야 할 정보 |
| --- | --- |
| 조정금이 있는가? | 조정금 합계, 조정 내역 없음 상태 |
| 어느 금액에 영향을 주는가? | `청구금`, `운송비(배차금)` 둘 다 대상이며 항목별 대상 구분 |
| 어떤 항목이 붙거나 빠졌는가? | 대기비, 경유비, 할인, 기타 조정 항목 |
| 얼마가 더해지거나 빠졌는가? | 항목별 `+/-` 금액, 합계 |
| 누가 처리했는가? | 항목별 담당자 또는 마지막 담당자 |
| 언제 처리했는가? | 항목별 처리 시각 또는 마지막 처리 시각 |
| 정산 상태는 어떤가? | 정산 전, 정산 처리됨, 정산 보류 등 |

## 확정된 조정금 대상 정책

조정금은 `청구금`과 `운송비(배차금)` 둘 다에 반영될 수 있습니다. 최소 기준이 되는 청구금과 운송비가 먼저 있고, 그 위에 대기비, 경유비, 할인처럼 기준 금액에 +되거나 -되는 금액을 `조정금` 항목으로 정리합니다.

| 정책 | 설명 |
| --- | --- |
| 기준 금액 | `청구금`, `운송비(배차금)` 각각의 기본 금액을 기준으로 봅니다. |
| 조정금 대상 | 모든 조정금 항목은 `청구금 대상`, `운송비 대상`, 또는 둘 다에 영향을 주는 항목으로 구분합니다. |
| +/- 표현 | 기준 금액에 더해지는 항목은 `+`, 빠지는 항목은 `-`로 표시합니다. |
| 첫 화면 명세 | `청구 조정금`, `배차 조정금`, `수익`을 명세서처럼 분리해 보여줍니다. |
| 상세 항목 | 명세서 안에서 항목별 대상 금액, +/- 금액, 담당자, 처리 시각을 확인할 수 있어야 합니다. |

## MVP 조정금 사유 코드

MVP에서는 조정금 사유 코드를 4개로 시작합니다. 화면 라벨은 쉬운 한국어를 쓰고, 데이터 저장이나 API 설계에서는 코드값을 별도로 둘 수 있습니다.

| 코드 | 화면 라벨 | 기본 부호 | 기본 대상 | 설명 |
| --- | --- | --- | --- | --- |
| `WAITING_FEE` | `대기비` | `+` | `청구금`, `배차금`, `둘 다` | 상차지/하차지 대기 시간이 발생해 금액을 더하는 경우 |
| `STOP_FEE` | `경유비` | `+` | `청구금`, `배차금`, `둘 다` | 경유지 추가나 경로 변경으로 금액을 더하는 경우 |
| `DISCOUNT` | `할인` | `-` | 주로 `청구금`, 필요 시 `배차금` | 고객 할인, 금액 차감, 클레임 조정처럼 기준 금액에서 빼는 경우 |
| `MANUAL_ADJUSTMENT` | `기타 조정` | `+/-` | `청구금`, `배차금`, `둘 다` | 위 3개로 설명하기 어려운 수동 조정 |

`금액 입력` 다이얼로그와 `금액 로그` 상세 항목은 같은 사유 코드를 사용합니다. 다이얼로그에서 입력한 `조정 사유`, `조정 대상`, `조정 금액`, `조정 설명`은 저장 후 금액 로그 상세 row에서 같은 순서로 확인할 수 있어야 합니다.

## 표시 구조 원칙

첫 화면은 `금액 명세`를 우선합니다. 운영자가 탭을 열자마자 기준 금액, 조정금, 합계, 수익을 한 번에 읽을 수 있어야 합니다. 항목별 담당자, 처리 시각, 사유는 사용자가 필요할 때 `상세 항목 보기`로 확인합니다.

| 레벨 | 표시 방식 | 목적 |
| --- | --- | --- |
| 1차 명세 | 청구금 기준액, 청구 조정금, 청구금 합계, 배차금 기준액, 배차 조정금, 배차금 합계, 수익, 수익률 | 탭을 열자마자 금액 구조 파악 |
| 2차 상세 | 명세 내부의 항목별 조정 row | 고객 문의나 정산 확인 시 담당자, 시각, 사유 확인 |
| 감사 상세 | 변경 사유, 생성 시점, 정산 연결 정보 | 후속 확장 또는 별도 상세 화면에서 처리 |

## UI 구성 요소

| 영역 | 구성 요소 | 설명 |
| --- | --- | --- |
| 탭 요약 | 조정금 합계 chip, 정산 상태 badge | 탭을 열기 전에도 조정금 존재 여부를 알 수 있게 함 |
| 금액 명세 | receipt 또는 statement block | `청구금`, `배차금`, `수익`을 세로 명세 구조로 표시 |
| 상세 트리거 | `상세 항목 보기` 또는 명세 row 클릭 | 첫 화면은 명세 중심으로 유지하고 필요할 때 명세 내부 row 노출 |
| 상세 항목 row | inline receipt detail row | 조정 사유, 조정 대상, +/- 금액, 담당자, 처리 시각, 조정 설명을 하단 박스 없이 표시 |
| 금액 데이터 없음 | `empty` box | 기준금액과 금액 로그가 모두 없을 때 `등록된 금액 로그가 없습니다`와 `금액 입력으로 이동`만 표시 |
| 조정 내역 없음 명세 | receipt 내부 0원 row | 기준 청구금/배차금은 있지만 조정 내역만 없을 때 명세 안에서 조정금 0원 또는 상세 항목 없음 상태 표시 |
| 상태 chip | 정산 전, 정산 처리됨, 정산 보류 | 현재 오더의 정산 상태와 연결 |
| 보조 액션 | `금액 입력으로 이동` | 직접 수정 대신 기존 금액 입력 흐름으로 연결 |

## 상태별 기획

| 상태 | 조건 | UI | 주요 액션 |
| --- | --- | --- | --- |
| 빈 상태 | 기준금액과 조정 로그가 모두 없음 | 메모 탭과 같은 empty box 패턴으로 `등록된 금액 로그가 없습니다` 표시 | `금액 입력으로 이동` |
| 조정 내역 없음 | 기준 청구금/배차금은 있으나 조정 내역이 없음 | 금액 명세는 유지하고 조정금 0원 또는 상세 항목 없음 표시 | `금액 입력으로 이동` |
| 로딩 상태 | 조정 내역 조회 중 | summary skeleton 1개, 상세 skeleton 1~2개 | 탭 전환 가능 |
| 오류 상태 | 로그 조회 실패 | 오류 notice와 재시도 | `다시 불러오기` |
| 데이터 있음 | 조정 내역이 1개 이상 있음 | 금액 명세, 상세 항목 보기 | `상세 항목 보기`, `금액 입력으로 이동` |
| 상세 펼침 | 사용자가 명세의 상세 트리거를 클릭함 | 항목별 대상 금액, 담당자, 처리 시각 표시 | `상세 항목 접기` |
| 정산 처리됨 | 정산 완료 상태 | 수정 제한 notice, 정산 상태 badge | 금액 수정 불가 안내 |
| 변경 직후 | `금액 조건 적용` 성공 직후 | 요약 chip과 최신 item flash | 최신 item 상단 추가 |

## 기존 화면과의 연결

| 기존 요소 | 연결 방식 |
| --- | --- |
| `openMoneyInput()` | 금액 수정은 기존 다이얼로그를 사용합니다. |
| `applyMoney()` | 적용 성공 후 금액 명세와 명세 내부 상세 row에 새 item이 반영되어야 합니다. |
| `confirmSettle()` | 정산 처리 결과는 금액 로그의 정산 상태에 반영합니다. |
| `sec-cargo-money` | 현재 표시되는 청구금/운송비와 금액 명세의 기준이 서로 충돌하지 않아야 합니다. |
| 하단 위험 action | 정산 처리 후 금액 수정 제한 문구를 금액 로그에서도 보여줍니다. |

## MVP 범위

| 포함 | 제외 |
| --- | --- |
| 조정 내역 없음/있음 상태 표시 | 변경 승인 workflow |
| 금액 명세, 청구 조정금, 배차 조정금, 수익 표시 | 회계 시스템 실시간 연동 |
| 항목별 +/- 금액, 담당자, 처리 시각 표시 | 세금계산서 발행 상세 |
| 명세 내부 상세 row 펼침 또는 dialog | 로그 삭제/수정 |
| 정산 상태 표시 | 다중 통화 |
| 금액 입력 흐름으로 이동 | 전체 금액 변경 전/후 감사 화면 |

## 후속 확장 범위

| 확장 | 설명 |
| --- | --- |
| 감사 상세 | 변경 전/후 전체 금액, IP, 역할, 서버 확정 시각까지 표시 |
| 승인 이력 | 금액 변경 승인자와 승인 시각 표시 |
| 정산 시스템 연동 | 실제 정산 상태와 회계 처리 상태 연결 |
| 문서 연결 | 인수증, 세금계산서, 위탁증과 연결 |
| 변경 사유 템플릿 | 대기비, 경유비, 거리 변경, 수수료 조정 등 사유를 선택형으로 제공 |
| 이상 변경 감지 | 일정 금액 이상 변경 시 경고 또는 승인 필요 |

## Acceptance Criteria

| ID | 기준 |
| --- | --- |
| `AMT-AC-01` | 금액 로그 첫 화면에는 기준 청구금, 청구 조정금, 청구금 합계, 기준 배차금, 배차 조정금, 배차금 합계, 수익이 명세서처럼 표시되어야 합니다. |
| `AMT-AC-02` | 기준금액과 금액 로그가 모두 없으면 공통 empty box로 표시하고, 기준 청구금/배차금은 있지만 조정 내역만 없으면 명세 안에서 조정금 0원 또는 상세 항목 없음 상태가 표시되어야 합니다. |
| `AMT-AC-03` | 항목별 상세에는 조정 사유, 조정 대상, +/- 금액, 담당자, 처리 시각, 조정 설명이 표시되어야 합니다. |
| `AMT-AC-04` | 항목별 담당자와 처리 시각은 기본 첫 화면을 과하게 차지하지 않고, `상세 항목 보기`로 확인할 수 있어야 합니다. |
| `AMT-AC-05` | 금액 로그 탭에서 금액을 직접 수정하지 않아야 합니다. 수정은 기존 금액 입력 흐름으로 연결합니다. |
| `AMT-AC-06` | 정산 처리됨 상태에서는 금액 수정 제한 안내가 보여야 합니다. |
| `AMT-AC-07` | 금액 로그 조회 실패 시 오류 메시지와 재시도 액션이 보여야 합니다. |
| `AMT-AC-08` | 현재 `sec-cargo-money`에 보이는 청구금/운송비와 금액 명세의 기준이 충돌하지 않아야 합니다. |

## 확인 필요 정책

| 항목 | 확인 필요 내용 |
| --- | --- |
| 항목별 대상 저장 | 조정금이 `청구금`, `운송비(배차금)`, 또는 둘 다에 영향을 주는 경우를 데이터로 어떻게 저장할지 확인 필요 |
| 사유 코드 | MVP 1차 사유 코드는 `대기비`, `경유비`, `할인`, `기타 조정`으로 확정. 후속에서 더 세분화할지 확인 필요 |
| 로그 생성 시점 | 다이얼로그 저장 시점인지, 서버 확정 시점인지 확인 필요 |
| 변경 사유 필수 여부 | 모든 조정에 사유를 필수로 받을지 확인 필요 |
| 정산 후 수정 | 정산 후 조정금 수정 가능 여부와 예외 승인 정책 확인 필요 |
| 감사 요구 | 처리자, 처리 시각, IP, 역할 등 감사 필드 범위 확인 필요 |
| 개인정보/보안 | 금액 로그를 볼 수 있는 권한 범위 확인 필요 |
