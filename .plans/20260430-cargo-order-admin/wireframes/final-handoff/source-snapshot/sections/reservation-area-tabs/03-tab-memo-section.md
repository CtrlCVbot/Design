# 메모 탭 상세 기획

## 목적

`메모` 탭은 화물 운송건에 대해 운영자가 알아야 할 사람의 기록을 모아 보여주는 탭입니다.

여기서 메모는 시스템 로그와 다릅니다. 시스템이 자동으로 남기는 상태 변경 기록이 아니라, 운영자나 고객 응대 과정에서 남기는 설명, 요청, 주의사항을 다룹니다.

## 메모 유형

| 유형 | 의미 | 예시 |
| --- | --- | --- |
| 운영자 메모 | 내부 운영자가 남긴 일반 기록 | `화주가 오후 3시 이후 통화 가능` |
| 고객 요청 | 화주 또는 고객이 요청한 내용 | `하차 전 30분 전에 연락 요청` |
| 배차 특이사항 | 차주/차량/상하차 조건과 관련된 주의사항 | `현장 진입로 좁음, 5톤 이상 진입 확인 필요` |
| 정산 참고 | 금액이나 정산 처리 참고 사항 | `수수료 조정 사유는 금액 로그 확인` |

MVP에서는 `운영자 메모`, `고객 요청`, `배차 특이사항` 3개 유형을 우선 권장합니다. `정산 참고`는 금액 로그와 역할이 겹칠 수 있으므로 후속 확장 또는 정책 확인 후 도입합니다.

## UI 구성 요소

| 영역 | 구성 요소 | 설명 |
| --- | --- | --- |
| 탭 요약 | 메모 수 badge | 전체 메모 수 또는 미확인 중요 메모 수 표시 |
| 상단 액션 | `추가` 버튼 | 기본 화면은 리스트만 보여주고, 작성은 dialog에서 처리 |
| 추가 dialog | 유형 선택, 메모 textarea, 저장/취소 버튼 | 한 줄 입력보다 2~3줄 textarea가 운영 메모에 적합 |
| 필터 | 전체, 운영자, 고객 요청, 배차 특이사항 | MVP에서는 필터를 생략하고 유형 chip만 표시해도 가능 |
| 리스트 | 메모 item | 유형, 본문, 작성자, 작성 시각, 수정 여부 |
| 중요 표시 | pin 또는 warning chip | 후속 확장. MVP에서는 `중요` checkbox를 보류 가능 |
| 보조 액션 | 수정, 삭제 | MVP에서는 정책 확인 전 보류. 추가만 우선 |

## 상태별 기획

| 상태 | 조건 | UI | 주요 액션 |
| --- | --- | --- | --- |
| 빈 상태 | 저장된 메모가 없음 | `아직 메모가 없습니다.`와 `메모 추가` 버튼 | `메모 추가` |
| 로딩 상태 | 메모 목록 조회 중 | 리스트 skeleton 2~3줄 | 탭 전환 가능 |
| 오류 상태 | 조회 또는 저장 실패 | 오류 notice와 재시도 | `다시 불러오기`, `다시 저장` |
| 데이터 있음 | 메모가 1개 이상 있음 | 최신순 리스트, 유형 chip, 작성자/시각 | `메모 추가` |
| dialog 입력 중 | 추가 dialog의 textarea에 값이 있음 | 저장 버튼 활성화, dialog 닫기 전 이탈 확인 검토 | `저장`, `취소` |
| 저장 중 | 추가 요청 전송 중 | dialog 저장 버튼 loading, 중복 클릭 방지 | 완료 후 dialog 닫고 리스트 상단 추가 |

## 메모 추가 규칙

| 규칙 | 설명 |
| --- | --- |
| 작성 위치 | 메모 작성은 탭 본문이 아니라 `메모 추가` dialog에서 처리합니다. |
| 필수값 | 메모 본문은 필수입니다. 유형은 기본 `운영자 메모`로 둡니다. |
| 길이 | MVP 기준 500자 이내를 권장합니다. |
| 저장 위치 | 현재 오더 ID 기준으로 저장합니다. |
| 정렬 | 최신 메모가 위에 오도록 정렬합니다. |
| feedback | 저장 성공 시 새 메모 item만 field flash합니다. 전체 row 흔들림은 사용하지 않습니다. |
| 실패 | dialog를 닫지 않고 입력값을 유지한 채 오류 메시지를 보여줍니다. |

## 기존 화면과의 연결

| 기존 요소 | 연결 방식 |
| --- | --- |
| 운송 구간 | 상하차 특이사항은 `배차 특이사항` 유형으로 연결 가능 |
| 차주 정보 | 차주 적용 또는 화물맨 배차 결과에 대한 운영자 판단을 메모로 남길 수 있음 |
| 하단 actionbar | `화물 취소`, `배차 취소` 같은 위험 액션 전후 메모 필요 여부는 정책 확인 필요 |
| 금액 로그 | 금액 변경의 공식 사유는 `금액 로그`에 남기고, 사람의 추가 설명만 메모에 남김 |

## MVP 범위

| 포함 | 제외 |
| --- | --- |
| 메모 리스트 표시 | 메모 수정/삭제 |
| 메모 추가 dialog | 첨부파일 |
| 유형 chip 표시 | 멘션/알림 |
| 작성자와 작성 시각 표시 | 내부/외부 공개 범위 |
| 조회/저장 오류 상태 | 중요 메모 pin |

## 후속 확장 범위

| 확장 | 설명 |
| --- | --- |
| 수정/삭제 권한 | 작성자 본인, 관리자 등 권한별 액션 분리 |
| 중요 메모 고정 | 탭 상단에 중요 메모를 고정 표시 |
| 멘션/알림 | 특정 운영자에게 확인 요청 |
| 변경 이력 | 메모 수정 전/후 이력 표시 |
| 외부 공유 | 고객에게 보여줄 수 있는 메모와 내부 메모 분리 |

## Acceptance Criteria

| ID | 기준 |
| --- | --- |
| `MEMO-AC-01` | 메모가 없을 때 빈 상태와 `메모 추가` 버튼이 보여야 하며, 탭 본문에 상시 입력 영역을 노출하지 않아야 합니다. |
| `MEMO-AC-02` | 메모 item에는 유형, 본문, 작성자, 작성 시각이 표시되어야 합니다. |
| `MEMO-AC-03` | `메모 추가` 버튼을 누르면 유형과 본문을 입력할 수 있는 dialog가 열려야 하며, 본문이 비어 있으면 저장할 수 없어야 합니다. |
| `MEMO-AC-04` | 저장 성공 후 새 메모가 리스트 상단에 추가되어야 합니다. |
| `MEMO-AC-05` | 저장 실패 시 dialog 입력값을 잃지 않고 오류 상태를 보여야 합니다. |
| `MEMO-AC-06` | 금액 변경의 공식 사유는 `금액 로그`가 담당하고, 메모 탭은 보조 설명을 담당한다는 역할 분리가 유지되어야 합니다. |

## 확인 필요 정책

| 항목 | 확인 필요 내용 |
| --- | --- |
| 작성 권한 | 모든 운영자가 작성 가능한지, 담당자만 가능한지 확인 필요 |
| 수정/삭제 | MVP에서 제외할지, 작성자 본인만 허용할지 확인 필요 |
| 공개 범위 | 고객 요청 메모가 외부에 노출될 가능성이 있는지 확인 필요 |
| 보존 기간 | 메모를 오더와 함께 영구 보존할지 확인 필요 |
| 감사 로그 | 메모 수정/삭제가 필요해질 경우 별도 감사 로그가 필요한지 확인 필요 |
