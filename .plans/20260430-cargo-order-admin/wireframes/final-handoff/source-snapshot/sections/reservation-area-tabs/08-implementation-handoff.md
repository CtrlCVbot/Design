# 보조 정보 탭 구현 인계 체크리스트

## 목적

이 문서는 `보조 정보` 탭 기획을 구현 단계로 넘기기 위한 최종 인계 문서입니다.

상세 배경은 `01`~`07` 문서에 있고, 이 문서는 구현자가 먼저 확인해야 할 최종 결정, 구현 순서, 검증 기준만 모읍니다.

## 최종 결과물 기준

구현자는 화면 기준으로 `../baseline/html/cargo-order-admin-hifi-master.html`을 먼저 확인합니다.

| 상태 | 확인 방법 |
| --- | --- |
| 데이터 있음 | master 파일을 그대로 열어 확인 |
| 데이터 없음 | master 파일에서 `신규(F3)` 클릭 후 확인 |
| 메모만 있음 | `신규(F3)` 후 화주 선택 적용 |
| 지도 있음 | 상차지와 하차지를 모두 적용 |
| 금액 로그 있음 | 정산 정보 또는 금액 입력 적용 |

과거 후보 HTML은 비교와 rollback을 위한 참고 파일입니다. 구현 기준과 최종 안내는 master 파일을 우선합니다.

## 최종 결정 요약

| 항목 | 최종 결정 |
| --- | --- |
| 화면 노출명 | `보조 정보` |
| 내부 작업명 | `예약 영역`, `reservation-area-tabs` |
| 디자인 기준 | `results/html/Design System.html`의 shadcn 기준 |
| 기본 탭 | `메모` |
| 탭 순서 | `메모`, `금액 로그`, `운송 구간 지도` |
| 표시 원칙 | 오더가 선택된 접수/수정 화면에서는 기본 표시 |
| 데이터 없음 | 섹션 전체를 숨기지 않고 세 탭 모두 같은 empty box 패턴 사용 |
| 숨김 조건 | 오더 미선택, 목록 전용, 인쇄, 외부 공유 화면 |
| 접힘 조건 | 좁은 화면, 사용자 접기 선택, 권한 제한 등 |

## 탭별 구현 기준

| 탭 | MVP 구현 | 제외 또는 후속 |
| --- | --- | --- |
| `메모` | 리스트 중심, `메모 추가` 버튼, 작성 dialog, 유형 chip | 수정/삭제, 필터, 중요 표시 |
| `금액 로그` | 청구금/배차금/수익 명세, 조정금 상세 펼침, 정산 상태 | 전체 회계 감사 화면, 승인 workflow |
| `운송 구간 지도` | 지도/핀/상태 chip, 거리, 예상 시간, 경유 여부, 주소 미입력 empty | 실제 provider 최적화, 다중 경유 상세, 실시간 교통 |

## 금액 로그 구현 기준

금액 증감 용어는 `조정금`으로 통일합니다. 추가와 할인은 별도 용어를 나누지 않고 `+/-` 부호로 구분합니다.

| 사유 코드 | 화면 라벨 | 기본 부호 | 대상 |
| --- | --- | --- | --- |
| `WAITING_FEE` | `대기비` | `+` | `청구금`, `배차금`, `둘 다` |
| `STOP_FEE` | `경유비` | `+` | `청구금`, `배차금`, `둘 다` |
| `DISCOUNT` | `할인` | `-` | 주로 `청구금`, 필요 시 `배차금` |
| `MANUAL_ADJUSTMENT` | `기타 조정` | `+/-` | `청구금`, `배차금`, `둘 다` |

`금액 입력` 다이얼로그와 `금액 로그` 상세 row는 같은 기준을 사용합니다.

| 입력/표시 항목 | 설명 |
| --- | --- |
| `조정 사유` | 위 4개 사유 중 하나 |
| `조정 대상` | `청구금`, `배차금`, `둘 다` |
| `조정 금액` | `+30,000원`, `-10,000원`처럼 부호 포함 |
| `조정 설명` | 운영자가 고객 문의나 정산 확인에 쓸 수 있는 짧은 설명 |
| `담당자`, `처리 시각` | 저장 후 로그 상세 row에서 확인 |

## 구현 순서 제안

1. `보조 정보` 패널 shell, 헤더, 탭 구조, 표시/접힘/숨김 조건을 먼저 구현합니다.
2. `메모` 탭을 구현합니다. 기본 탭이므로 리스트, empty, dialog 저장 흐름을 먼저 안정화합니다.
3. `금액 로그` 탭을 구현합니다. 첫 화면은 compact 명세로 두고, 상세 항목은 펼침으로 분리합니다.
4. `운송 구간 지도` 탭을 구현합니다. 실제 지도 provider 전에는 placeholder 또는 mock route로 시작할 수 있습니다.
5. 접근성, keyboard, reduced motion, 오류/로딩/빈 상태를 공통으로 검증합니다.

## 데이터 필드 초안

구현 전 API와 DB 설계에서 아래 필드를 기준으로 논의합니다. 필드명은 예시이며 실제 코드 스타일에 맞게 바꿀 수 있습니다.

| 영역 | 필드 초안 |
| --- | --- |
| 메모 | `id`, `orderId`, `type`, `content`, `authorName`, `createdAt` |
| 금액 요약 | `baseChargeAmount`, `chargeAdjustmentTotal`, `chargeTotal`, `baseDispatchAmount`, `dispatchAdjustmentTotal`, `dispatchTotal`, `profitAmount`, `profitRate`, `settlementStatus` |
| 조정금 item | `id`, `reasonCode`, `targetType`, `amountSigned`, `note`, `actorName`, `changedAt` |
| 지도 | `loadAddressStatus`, `unloadAddressStatus`, `loadCoordinate`, `unloadCoordinate`, `distanceKm`, `estimatedMinutes`, `hasStopover`, `routeStatus` |

## 구현 전 확인 필요

| 항목 | 필요한 결정 |
| --- | --- |
| 지도 provider | 실제 지도 API, 비용, 쿼터, fallback 방식 |
| 예상 시간 | 실시간 교통 포함 여부, 상하차 대기 시간 포함 여부 |
| 조정금 저장 방식 | `둘 다` 대상 조정금을 하나의 item으로 저장할지, 청구/배차 item으로 분리 저장할지 |
| 변경 사유 필수 여부 | 모든 조정금에 `조정 설명`을 필수로 받을지 여부 |
| 정산 후 수정 | 정산 완료 후 조정금 수정 가능 여부와 승인 예외 |
| 메모 권한 | 작성, 수정, 삭제, 공개 범위, 보존 기간 |
| 금액 권한 | 금액 로그 조회 가능 역할과 마스킹 기준 |

## 검증 체크리스트

| 검증 | 기준 |
| --- | --- |
| 탭 순서 | `메모`, `금액 로그`, `운송 구간 지도` |
| 기본 탭 | 최초 진입 시 `메모` 활성 |
| 빈 상태 | 데이터가 없어도 패널을 숨기지 않고 각 탭 empty box 표시 |
| 금액 명세 | 기준 청구금, 청구 조정금, 청구금 합계, 기준 배차금, 배차 조정금, 배차금 합계, 수익 표시 |
| 금액 상세 | `상세 항목 보기`에서 조정 사유, 대상, 금액, 담당자, 시각, 설명 표시 |
| 지도 중복 방지 | 좌측 `2. 운송 구간`과 겹치는 상차/하차 전체 주소 summary row 반복 없음 |
| 메모 작성 | 리스트 화면에서 `메모 추가` 버튼으로 dialog 열기 |
| 기존 섹션 충돌 | 주소/금액 직접 수정은 기존 접수/수정 섹션으로 연결 |
| 접근성 | `tablist`, `tab`, `tabpanel`, `aria-selected`, keyboard 이동 확인 |
| motion | 기존 접수/수정 field flash, handoff dot/pulse와 충돌 없음 |

## 구현 착수 판정

기획 문서 기준으로는 구현 착수가 가능합니다.

단, 실제 데이터 연동 구현 전에 `구현 전 확인 필요` 항목 중 지도 provider, 금액 권한, 정산 후 수정 정책은 우선 확인해야 합니다. 이 항목들은 화면 구조를 막지는 않지만 API 필드, 권한 처리, 오류 문구에 영향을 줍니다. 기획 마무리 기준과 우선순위는 `09-planning-closure.md`를 따릅니다.
