# Data Contract: 3. 화물 운송정보 -> 4. 화물정보 요약

## 목적

이 문서는 B 통합본 기준 `3. 화물 운송정보`에서 입력된 값이 `4. 화물정보 요약` 문장으로 어떻게 전달되는지 정의합니다.

결론부터 말하면, **원본 데이터의 source of truth는 `3. 화물 운송정보`이고, `4. 화물정보 요약`은 파생 문장과 사용자 수정 override를 표시하는 영역**입니다.

## 연결 대상

| 구분 | 섹션 | 역할 |
| --- | --- | --- |
| 입력 원천 | `3. 화물 운송정보` | `톤수`, `차종`, `대수`, `실중량`, `품목`, 금액 조건 입력 |
| 보조 원천 | `2. 운송 구간` | 상차/하차 시간, 상하차 방법, 상하차 조건 요약 |
| 표시 대상 | `4. 화물정보 요약` | 차주/운영자가 빠르게 읽는 한 문장 요약 |

## Source 필드

| 우선순위 | 필드 | 출처 | 요약 반영 |
| --- | --- | --- | --- |
| 1 | `품목` | `3. 화물 운송정보`의 `운송+품목` 2번째 row | 문장의 핵심 물품명 |
| 2 | `실중량` | `3. 화물 운송정보`의 `실중량` | `5.5톤`처럼 중량 표현에 사용 |
| 3 | `톤수` | `3. 화물 운송정보`의 `톤수` | 차량 조건 표현 또는 실중량 누락 시 대체 |
| 4 | `차종` | `3. 화물 운송정보`의 `차종` | `축차`, `윙바디` 등 차량 조건 표현 |
| 5 | `대수` | `3. 화물 운송정보`의 `대수` | 2대 이상일 때만 문장에 포함 |
| 6 | 상차/하차 일시 | `2. 운송 구간` | `지금상`, `당착` 같은 간단 조건 |
| 7 | 상차/하차 방법 | `2. 운송 구간` | `상하차 지게차 조건` 같은 작업 조건 |

금액 조건(`결제방법`, `청구비용`, `운송비용`, `수수료`, `수익/차주운임`)은 요약 문장에 기본 포함하지 않습니다. 금액은 정산/목록/출력 흐름의 정보이고, 화물정보 요약은 차주가 화물 조건을 이해하는 데 필요한 문장으로 제한합니다.

## 자동 요약 생성 규칙

### 기본 템플릿

```text
{상차조건} {하차조건} {품목} {주소/거점요약}. {실중량 또는 톤수} {차종}, {상하차방법} 조건.
```

B 통합본 현재 예시는 아래 형태를 기준으로 합니다.

```text
지금상 당착 장비 운송 지/지-A. 판넬 12파렛트, 5.5톤 축차, 상하차 지게차 조건.
```

### 문장 조립 순서

| 순서 | 구성 요소 | 규칙 |
| --- | --- | --- |
| 1 | 일정 조건 | `지금상`, `당착`, `내일상`, `월착`처럼 짧은 토큰으로 표현 |
| 2 | 품목 | `품목` 값이 있으면 가장 앞쪽에 배치 |
| 3 | 거점/상세 설명 | 주소 전체가 아니라 운영자가 빠르게 읽을 수 있는 거점 요약만 사용 |
| 4 | 중량/차량 | `실중량`이 있으면 우선 사용, 없으면 `톤수` 사용 |
| 5 | 차종 | `축카고`는 요약 문장에서는 `축차`처럼 짧게 표현 가능 |
| 6 | 상하차 방법 | 상차/하차 방법이 같으면 `상하차 지게차 조건`, 다르면 `상차 지게차 / 하차 수작업` |

## 필드별 문장화 규칙

| 필드 | 값 예시 | 요약 표현 | 비고 |
| --- | --- | --- | --- |
| `품목` | `장비 운송` | `장비 운송` | 필수에 가까운 핵심값 |
| `실중량` | `5.50톤` | `5.5톤` | 문장에서는 불필요한 trailing zero 제거 |
| `톤수` | `5톤` | `5톤` | 실중량이 없을 때 대체 |
| `차종` | `축카고` | `축차` 또는 `축카고` | 운영 용어가 확정되면 하나로 통일 |
| `대수` | `1대` | 생략 | 1대는 생략, 2대 이상만 `2대` 포함 |
| 상차일시 | `지금` | `지금상` | 운송구간의 표시 규칙 사용 |
| 하차일시 | `당일` | `당착` | 운송구간의 표시 규칙 사용 |
| 상하차방법 | `지게차`/`지게차` | `상하차 지게차 조건` | 같으면 병합 |
| 상하차방법 | `지게차`/`수작업` | `상차 지게차 / 하차 수작업` | 다르면 분리 |

## Override 정책

요약 문장은 자동 생성값과 사용자 수정값을 분리합니다.

| 상태 | 의미 | 동작 |
| --- | --- | --- |
| `auto` | 원본 필드 기준으로 생성된 요약 | `3. 화물 운송정보` 또는 `2. 운송 구간` 변경 시 자동 갱신 |
| `overridden` | 사용자가 `4. 화물정보 요약`에서 직접 수정한 문장 | 원본 필드 변경 후에도 즉시 덮어쓰지 않음 |
| `stale` | 원본 필드가 바뀌어 기존 override와 불일치 가능 | `요약 재생성` 또는 `수정 유지` 선택 필요 |
| `empty` | 요약 생성에 필요한 핵심값 부족 | `화물 운송정보 입력 후 요약 생성` placeholder 표시 |

최종 UI에서는 `자동/수정됨` chip의 필요 여부가 아직 확정되지 않았습니다. 다만 데이터 모델에서는 `auto`, `overridden`, `stale` 상태를 구분할 수 있어야 합니다.

## 갱신 트리거

| 변경 이벤트 | 자동 요약 영향 | 권장 처리 |
| --- | --- | --- |
| `품목` 변경 | 높음 | auto 상태면 즉시 재생성 |
| `실중량` 변경 | 높음 | auto 상태면 즉시 재생성 |
| `톤수` 변경 | 중간 | 실중량도 바뀌므로 재생성 |
| `차종` 변경 | 중간 | 차량 조건 토큰 재생성 |
| `대수` 변경 | 낮음 | 2대 이상일 때만 문장 반영 |
| 상차/하차 일시 변경 | 중간 | 일정 토큰 재생성 |
| 상하차 방법 변경 | 중간 | 작업 조건 토큰 재생성 |
| 금액 조건 변경 | 없음 | 요약 문장에는 기본 반영하지 않음 |

## 값 없음 기준

아래 핵심값 중 `품목`과 `중량/차량 조건`이 모두 비어 있으면 요약은 생성하지 않습니다.

| 조건 | 표시 |
| --- | --- |
| `품목` 없음, `톤수/실중량` 없음 | `화물 운송정보 입력 후 요약 생성` |
| `품목` 있음, 중량 없음 | 품목 중심 요약 생성 가능. 단, 차량 조건 누락 상태는 별도 validation 후보 |
| `품목` 없음, 중량 있음 | 중량/차량 중심 요약 가능. 단, 품목 누락 상태는 별도 validation 후보 |

## B 통합본 반영 기준

| 항목 | 기준 |
| --- | --- |
| 표시 형태 | 기존 `요약` row 한 줄 구조 유지 |
| 원본 연결 | `3. 화물 운송정보`의 `품목`, `실중량`, `톤수`, `차종`, `대수` |
| 보조 연결 | `2. 운송 구간`의 상하차 일정/방법 |
| 수정 방식 | 같은 row에서 input 전환 |
| 저장 방식 | 자동 생성값과 사용자 override 값을 분리할 수 있게 설계 |
| 금액 반영 | 기본 미반영 |
| 샘플 구현 | B 통합본 HTML에 `auto`, `overridden`, `stale` 상태 기준의 요약 갱신 샘플 반영 |

## Acceptance Criteria

| ID | 기준 | 상태 |
| --- | --- | --- |
| `AC-CS-DATA-001` | `3. 화물 운송정보`가 요약 문장의 원본 데이터임이 명시되어 있다 | 충족 |
| `AC-CS-DATA-002` | `품목`, `실중량`, `톤수`, `차종`, `대수`의 문장화 규칙이 정의되어 있다 | 충족 |
| `AC-CS-DATA-003` | 금액 조건은 요약 문장에 기본 반영하지 않는다고 정의되어 있다 | 충족 |
| `AC-CS-DATA-004` | 자동 생성값과 사용자 수정 override가 분리되어 있다 | 충족 |
| `AC-CS-DATA-005` | 원본 변경 시 요약 갱신 또는 stale 처리 기준이 있다 | 충족 |

## 후속 작업

1. `자동/수정됨` chip을 실제 화면에 표시할지 결정합니다.
2. 차종 축약어(`축차`/`축카고`)와 일정 토큰(`지금상`, `당착`)의 운영 표기를 확정합니다.
3. 실제 개발 단계에서 자동 요약 생성 함수의 저장 위치와 API payload 구조를 확정합니다.
