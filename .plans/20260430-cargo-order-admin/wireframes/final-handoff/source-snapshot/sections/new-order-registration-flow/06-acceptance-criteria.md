# Acceptance Criteria

## A. 신규 접수 시작

| ID | 기준 | 확인 방법 |
| --- | --- | --- |
| AC-A1 | `신규 접수` 클릭 시 화주, 상차지, 하차지, 운송+품목, 금액, 요약, 차주 관련 입력 상태가 초기화된다. | 상태값 또는 화면 표시 확인 |
| AC-A2 | 초기화 완료 후 상태는 `new-reset`이다. | 상태 로그 또는 UI state 확인 |
| AC-A3 | `new-reset`에서 섹션 헤더가 표시된다. | 화면 확인 |
| AC-A4 | 초기화 후 `화주 정보` 섹션 또는 `화주 정보 입력` 버튼에 focus가 이동한다. | 키보드 focus 확인 |
| AC-A5 | focus 이동은 render 완료 후 1회만 수행되고, 사용자가 직접 조작한 focus를 덮어쓰지 않는다. | 키보드/마우스 조작 확인 |

## B. 섹션 헤더 표시

| ID | 기준 | 확인 방법 |
| --- | --- | --- |
| AC-B1 | `idle-edit`에서 섹션 헤더는 표시되지 않는다. | 기본 화면 확인 |
| AC-B2 | 기존 화물 수정 중 섹션 헤더는 표시되지 않는다. | 수정 버튼 클릭 후 확인 |
| AC-B3 | `new-reset`, `new-wizard-active`, `new-required-complete`, `new-driver-optional`에서는 섹션 헤더가 표시된다. | 상태별 화면 확인 |
| AC-B4 | `new-submitted` 이후 섹션 헤더는 다시 숨김 처리된다. | 메인 화면 적용 후 확인 |
| AC-B5 | 화물을 선택해 조회하거나 수정하는 화면은 번호형 헤더 없이 본문 라벨과 값 중심의 명세서형 보기로 보인다. | 기존 화물 선택/수정 화면 확인 |
| AC-B6 | 신규 접수 화면의 섹션 헤더는 입력 안내용 번호를 포함한다. | 신규 접수 상태별 헤더 확인 |
| AC-B7 | 같은 정보 라벨이 섹션 헤더와 본문에서 과도하게 이중 반복되어 보이지 않는다. | 시각 QA |
| AC-B8 | `오더 요약`은 입력 단계 번호를 사용하지 않고, 신규 접수 안내형 보기에서만 회색 `확인` 배지로 표시된다. | 신규 접수 화면의 요약 섹션 헤더 확인 |
| AC-B9 | `오더 요약`은 별도 분리형 커스텀 블록이 아니라 기존 섹션 디자인 흐름으로 보인다. | 시각 QA |

## C. wizard 다이얼로그

| ID | 기준 | 확인 방법 |
| --- | --- | --- |
| AC-C1 | 신규 접수 최초 입력은 wizard 다이얼로그로 열린다. | `화주 정보 입력` 클릭 |
| AC-C2 | wizard 단계 순서는 `화주 정보 -> 상차지 -> 하차지 -> 운송+품목 -> 금액 -> 차주 정보 선택`이다. | 프로세스 패널 확인 |
| AC-C3 | 신규 접수 wizard 중 왼쪽 프로세스 패널이 표시된다. | 다이얼로그 레이아웃 확인 |
| AC-C4 | 화물 수정 또는 신규 접수 완료 후 개별 수정에서는 왼쪽 프로세스 패널이 표시되지 않는다. | 독립 다이얼로그 확인 |
| AC-C5 | 필수 단계는 validation을 통과해야 다음 단계로 이동할 수 있다. | 필수값 누락 테스트 |

## D. 금액 후 분기

| ID | 기준 | 확인 방법 |
| --- | --- | --- |
| AC-D1 | 금액까지 필수 입력이 완료되면 상태는 `new-required-complete`이다. | 상태 확인 |
| AC-D2 | 금액 마지막 다이얼로그에는 `화물 등록 완료`와 `차주 정보로 이동` 선택지가 표시된다. | footer 또는 마지막 화면 확인 |
| AC-D3 | `화물 등록 완료` 선택 시 API 통신 없이 wizard 전체 데이터가 메인 화면에 적용되고 신규 접수 UI가 종료된다. | 화면 전환과 네트워크 호출 없음 확인 |
| AC-D4 | `화물 등록 완료` 후 상태는 `new-submitted`이며 섹션 헤더는 숨김 처리된다. | 상태와 화면 확인 |
| AC-D5 | `차주 정보로 이동` 선택 시 상태는 `new-driver-optional`이며 차주 정보 단계가 표시된다. | 다이얼로그 확인 |
| AC-D6 | `new-submitted` 진입 후 메인 화면에 실제 API 통신 전 최종 CTA인 `화물 등록` 버튼이 표시된다. | 메인 화면 확인 |

## E. 차주 정보 선택

| ID | 기준 | 확인 방법 |
| --- | --- | --- |
| AC-E1 | 차주 정보는 선택 사항이다. | 건너뛰기 가능 여부 확인 |
| AC-E2 | 차주를 선택하면 차주 정보까지 메인 화면에 적용되고 `new-submitted`로 전환할 수 있다. | 차주 선택 후 완료 |
| AC-E3 | 차주를 건너뛰어도 차주 미지정 상태로 메인 화면에 적용되고 `new-submitted`로 전환할 수 있다. | 건너뛰기 후 완료 |
| AC-E4 | 차주 정보 단계에서도 신규 접수 wizard의 왼쪽 프로세스 패널이 유지된다. | 화면 확인 |

## F. 신규 접수 후 개별 수정

| ID | 기준 | 확인 방법 |
| --- | --- | --- |
| AC-F1 | `new-submitted` 이후 화주, 주소, 운송+품목, 금액, 차주 수정은 독립 다이얼로그로 열린다. | 각 수정 버튼 확인 |
| AC-F2 | `new-submitted` 이후 wizard는 다시 표시되지 않는다. | 수정 플로우 확인 |
| AC-F3 | `new-submitted` 이후 왼쪽 프로세스 패널은 표시되지 않는다. | 다이얼로그 확인 |
| AC-F4 | `new-submitted` 이후 섹션 헤더는 표시되지 않는다. | 메인 화면 확인 |
| AC-F5 | `new-submitted` 이후 개별 수정한 최신 값은 메인 `화물 등록` 버튼의 실제 API 전송 대상 데이터에 반영된다. | 수정 후 데이터 확인 |

## G. attention motion

| ID | 기준 | 확인 방법 |
| --- | --- | --- |
| AC-G1 | `new-reset` 진입 직후 `화주 정보 입력` 버튼에 soft focus ring, glow pulse, highlight sweep 중 하나를 적용할 수 있다. | 화면 확인 |
| AC-G2 | `new-reset` 진입 직후의 시작 attention motion은 2회 이하로만 반복된다. | animation count 확인 |
| AC-G3 | shake motion은 사용하지 않는다. | CSS/interaction 확인 |
| AC-G4 | `prefers-reduced-motion: reduce`에서는 animation을 생략하거나 정적 focus 표시로 대체한다. | reduced motion 환경 확인 |
| AC-G5 | 사용자가 클릭, 입력, focus 이동을 하면 attention motion은 중단된다. | 상호작용 확인 |
| AC-G6 | 기본/명세서형 화면에서 빈 필수 입력 행은 좌측 2px 상태 바로 표시된다. | `idle-edit` 또는 `new-submitted` 빈 상태 확인 |
| AC-G7 | 기본/명세서형 화면에서 빈 필수 입력 버튼은 최초 표시 시 최대 2회 pulse 후 멈춘다. | animation count 확인 |
| AC-G8 | 빈 필수값이 계속 남아 있으면 10초 간격으로 1회 reminder pulse만 다시 실행된다. | timer 동작 확인 |
| AC-G9 | 빈 필수 입력 버튼 클릭, 값 입력 완료, 행 제거 시 reminder timer가 정리되고 motion이 중단된다. | 클릭/입력 후 DOM 및 timer 동작 확인 |
| AC-G10 | 신규 접수 안내형 상태에서는 기본 화면용 빈 필수 입력 reminder가 표시되지 않는다. | `new-reset`, `new-wizard-active` 화면 확인 |

## H. 충돌 방지

| ID | 기준 | 확인 방법 |
| --- | --- | --- |
| AC-H1 | `idle-edit`에서 기존 독립 다이얼로그는 기존처럼 동작한다. | 회귀 확인 |
| AC-H2 | 신규 접수 상태가 아닌 경우 wizard 전용 UI가 노출되지 않는다. | 기본/수정 상태 확인 |
| AC-H3 | 신규 접수 중 취소 시 상태와 입력값 처리 정책이 일관되게 적용된다. | 취소 플로우 확인 |
| AC-H4 | 메인 `화물 등록` 버튼 클릭 전에는 실제 API 저장이 발생하지 않는다. | 네트워크 호출 확인 |
| AC-H5 | 메인 `화물 등록` 버튼 중복 클릭으로 동일 오더가 중복 등록되지 않는다. | 중복 클릭 확인 |

## I. 메인 `화물 등록` API 통신

| ID | 기준 | 확인 방법 |
| --- | --- | --- |
| AC-I1 | 메인 `화물 등록` 버튼 클릭 시 상태는 `submit-validating`으로 전환된다. | 상태 확인 |
| AC-I2 | 최종 validation은 메인 화면에 표시된 최신 데이터 기준으로 실행된다. | 개별 수정 후 등록 확인 |
| AC-I3 | validation 실패 시 API 요청은 발생하지 않는다. | 네트워크 호출 확인 |
| AC-I4 | validation 실패 시 첫 오류 섹션으로 scroll/focus가 이동한다. | 화면 동작 확인 |
| AC-I5 | validation 통과 시 상태는 `submit-pending`으로 전환되고 API 요청이 시작된다. | 상태와 네트워크 호출 확인 |
| AC-I6 | `submit-pending` 동안 버튼은 disabled이며 문구는 `등록 중...`으로 표시된다. | 버튼 상태 확인 |
| AC-I7 | 동일 데이터 재시도는 같은 `idempotencyKey`를 사용한다. | 요청 header/body 확인 |
| AC-I8 | 실패 후 데이터를 수정하면 `payloadVersion`이 증가하고 새 `idempotencyKey`를 사용한다. | 수정 후 요청 확인 |
| AC-I9 | API 실패 시 상태는 `submit-failed`이고 메인 화면 데이터는 유지된다. | 실패 응답 테스트 |
| AC-I10 | API 성공 시 상태는 `submit-complete`이고 pre-API `화물 등록` CTA는 숨김 처리된다. | 성공 응답 테스트 |
