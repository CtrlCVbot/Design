# 섹션 헤더 표시 정책

## 원칙

섹션 헤더는 신규 접수 중 사용자가 입력 순서와 현재 진행 맥락을 이해하도록 돕는 장치다.

기본 조회, 기존 화물 수정, 신규 접수 완료 후 개별 수정에서는 섹션 헤더를 숨긴다. 이 상태에서는 이미 선택된 화물의 라벨과 값이 본문에 있으므로, 번호가 붙은 헤더까지 보이면 전체 화물 라벨이 이중으로 반복되어 보인다.

따라서 기본/수정/화물 단독 조회 화면은 입력 wizard가 아니라 명세서처럼 보여야 한다. 반대로 신규 접수는 사용자가 아직 입력 순서를 따라가야 하는 상황이므로, 번호가 붙은 섹션 헤더를 안내 장치로 표시한다.

## 보기 모드

| 보기 모드 | 적용 상태 | 헤더 정책 | 의도 |
| --- | --- | --- | --- |
| 명세서형 보기 | `idle-edit`, `new-submitted` 이후 개별 수정/조회 | 번호형 섹션 헤더 숨김 | 본문 라벨과 값만 읽히게 하여 중복 라벨을 줄임 |
| 신규 접수 안내형 보기 | `new-reset`, `new-wizard-active`, `new-required-complete`, `new-driver-optional` | 번호형 섹션 헤더 표시 | 입력 순서와 현재 진행 단계를 안내 |

## 표시/숨김 기준

| 상태 | 섹션 헤더 | 이유 |
| --- | --- | --- |
| `idle-edit` | 숨김 | 명세서형 보기. 본문 라벨과 섹션 헤더가 이중으로 보이지 않게 함 |
| `new-reset` | 표시 | 신규 접수 시작 직후 입력 순서 안내 필요 |
| `new-wizard-active` | 표시 | wizard 진행 중 현재 단계 맥락 유지 |
| `new-required-complete` | 표시 | 필수 입력 완료 후 등록 완료/차주 이동 선택 맥락 유지 |
| `new-driver-optional` | 표시 | 선택 단계가 신규 접수의 일부임을 표시 |
| `new-submitted` | 숨김 | 메인 화면 적용 후 명세서형 보기로 전환 |

## 현재 화면 기준 매핑

현재 기준 화면에는 `.ds-section` 내부에 `.sec-head` 구조가 존재한다. 신규 정책에서는 이 구조를 삭제하지 않아도 된다.

구현 시에는 상태에 따라 표시 여부만 제어한다.

| 섹션 | 현재 단서 | 신규 접수 단계 |
| --- | --- | --- |
| 화주 정보 | `sec-shipper` | 1. 화주 정보 |
| 운송 구간 | `sec-route` | 2-1. 상차지, 2-2. 하차지 |
| 화물 정보 | `sec-cargo-transport` | 3. 운송+품목 |
| 금액 | `sec-cargo-money` | 4. 금액 |
| 오더 요약 | `sec-summary` | 등록 전/후 입력값 확인. 입력 단계가 아닌 확인용 섹션 |
| 차주/화물맨 | `driver-db`, `hm-panel` | 5. 차주 정보 선택 |

## 신규 접수 안내형 헤더 구조

신규 접수 중 섹션 헤더는 입력 순서 안내를 위해 아래 정보를 포함한다.

| 요소 | 역할 |
| --- | --- |
| 단계 번호 | 신규 접수의 큰 입력 순서 표시. 예: `1`, `2-1`, `2-2` |
| 섹션 제목 | `화주 정보`, `운송 구간`, `화물 정보`, `정산 정보`, `차주 정보` |
| 보조 메모 | 필수/선택 또는 현재 완료 상태를 짧게 표시 |
| 상태 표시 | 완료, 진행 중, 미입력 정도의 간단한 상태 |

번호형 헤더는 신규 접수의 길 안내 역할만 한다. 화물을 선택해 조회하거나 수정하는 화면에서는 같은 번호형 헤더를 재사용하지 않는다.

## 오더 요약 헤더 예외

`오더 요약`은 입력 단계가 아니라 입력값을 확인하는 정리용 섹션이다.

따라서 `화주 정보`, `운송 구간`, `화물 정보`, `정산 정보`, `차주 정보`처럼 단계 번호를 부여하지 않는다. 신규 접수 안내형 보기에서는 기존 섹션 디자인을 유지하되, 번호 박스 위치에 회색 `확인` 배지를 사용해 “다음 입력 단계”가 아니라 “검토 영역”임을 구분한다.

| 항목 | 정책 |
| --- | --- |
| 기본/수정/등록 완료 화면 | 다른 섹션과 동일하게 섹션 헤더 숨김 |
| 신규 접수 안내형 보기 | 기존 섹션 헤더 구조 사용 |
| 번호 박스 | 파란 번호 대신 회색 `확인` 배지 |
| 제목 | `오더 요약` |
| 보조 메모 | `입력값 확인`처럼 짧은 확인 목적 문구 |
| 시각 톤 | 별도 커스텀 요약 블록처럼 분리하지 않고 기존 섹션 디자인에 붙임 |

## 명세서형 보기의 UX 기준

`idle-edit`과 `new-submitted`에서는 아래를 모두 만족해야 한다.

- `.sec-head`가 화면에서 보이지 않는다.
- 본문 안의 라벨과 값이 정보 구조의 중심이 된다.
- 섹션 제목과 본문 라벨이 시각적으로 이중 반복되지 않는다.
- 화면은 입력 단계 안내보다 화물 명세서 또는 상세 내역처럼 읽힌다.
- 섹션 본문과 개별 수정 버튼은 기존처럼 사용할 수 있다.
- 독립 다이얼로그 제목과 본문만으로 수정 대상이 이해되어야 한다.
- wizard 단계 번호나 프로세스 패널이 노출되지 않는다.

## 신규 접수 완료 후 정책

`화물 등록 완료`를 선택하거나 선택 단계인 차주 정보까지 마치면 상태는 `new-submitted`가 된다.

이 상태는 API 저장 완료가 아니라 wizard 데이터를 메인 화면에 적용한 상태다. 섹션 헤더는 숨기고, 메인 화면에는 실제 API 통신 전 최종 CTA인 `화물 등록` 버튼을 표시한다.

이후 사용자가 화주, 상차지, 하차지, 운송+품목, 금액, 차주 정보를 다시 수정하더라도 신규 접수 wizard는 재개하지 않는다. 모든 수정은 `idle-edit`과 동일하게 독립 다이얼로그로 처리한다.

## 확인 필요

| 항목 | 확인 이유 |
| --- | --- |
| 헤더 숨김 방식 | `display: none`, `hidden`, 상태 클래스 중 접근성에 맞는 방식 결정 필요 |
| `new-submitted` 유지 시간 | 메인 `화물 등록` 버튼 클릭 전까지 유지할지, 내부적으로 `idle-edit`에 흡수할지 정책 확인 필요 |
| 요약 섹션 헤더 | 정책 확정. 신규 접수 안내형 보기에서만 회색 `확인` 배지로 표시하고, 메인 적용 후에는 명세서형 보기로 숨김 |
