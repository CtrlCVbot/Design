# PRD: 운송구간 섹션

## 1. Overview

`운송구간` 섹션은 화물 오더의 출발지와 도착지를 빠르게 확인하고, 필요할 때 상세 정보를 펼쳐 수정으로 이어지게 하는 화면 단위입니다.

현재 전체 오더 화면은 입력 필드가 많아 상차지와 하차지의 핵심 정보가 흩어져 보입니다. 이번 기획은 D `compact-hybrid`를 1차 기준안으로 삼아, 상차지와 하차지를 세로 2행 컴팩트 명세 카드 형태로 묶고 사용자가 한눈에 운송 경로를 확인하도록 만드는 것을 목표로 합니다.

## 2. Problem Statement

| 문제 | 영향 |
| --- | --- |
| 주소, 상세주소, 일시, 방법, 담당자가 여러 섹션에 분산됨 | 운송 구간 확인 시간이 길어짐 |
| 데이터가 있는 상태와 없는 상태의 시각 차이가 약함 | 신규 입력인지 수정인지 판단이 느림 |
| 상세주소와 담당자 정보가 기본 화면을 차지함 | 핵심 경로 정보가 묻힘 |
| 상차/하차 조건이 별도 섹션으로 떨어져 있음 | 실제 운송 구간 관점의 요약이 어려움 |
| 배차 유형이 운송구간과 떨어져 있음 | 독차/혼적 대표 선택과 긴급/왕복/예약 옵션, 경유 같은 전체 운송 조건을 경로 확인 중 놓칠 수 있음 |

## 3. Goals & Non-Goals

### Goals

- 데이터가 있는 운송구간을 명세서형 요약으로 표시한다.
- 1차 기준안은 D `compact-hybrid`로 둔다.
- 상차지와 하차지를 좌우 2열이 아닌 세로 2행으로 배치한다.
- 상세주소, 담당자, 연락처, 특이사항은 기본 화면에서 숨기고 펼침으로 제공한다.
- 주소가 없는 상태는 빈 상태로 명확히 구분한다.
- 기존 손그림 와이어프레임 톤을 유지한다.
- 거리와 기준 금액은 운송구간 단독 확정값이 아니라, 주소 적용 후 화물 운송정보의 `톤수`와 `차종`이 선택되어야 정확도가 올라가는 보조 정보로 다룬다.
- `배차 유형`은 상차나 하차 한쪽이 아니라 상하차지 전체에 걸리는 운송 조건으로 보고, 운송구간 요약의 보조 정보에 포함한다. 대표 유형은 `독차`와 `혼적` 중 하나이며 기본값은 `독차`로 둔다.

### Non-Goals

- 실제 주소 검색 API 연동은 다루지 않는다.
- 거리/운송료 계산 로직은 구현하지 않는다.
- 화물 운송정보, 화주 정보, 차주 정보 섹션은 수정하지 않는다.
- 저장, 수정, 배차취소 같은 전체 오더 액션은 구현하지 않는다.

## 4. User Stories

| ID | User Story |
| --- | --- |
| US-TR-001 | 운영자로서 데이터가 있는 오더를 열었을 때 상차지와 하차지를 한눈에 보고 싶다. 그래야 경로 확인 시간을 줄일 수 있다. |
| US-TR-002 | 운영자로서 상세주소와 담당자는 필요할 때만 펼쳐 보고 싶다. 그래야 기본 화면이 복잡해지지 않는다. |
| US-TR-003 | 운영자로서 주소가 없는 신규 오더에서 무엇을 먼저 입력해야 하는지 알고 싶다. 그래야 입력을 빠르게 시작할 수 있다. |
| US-TR-004 | 운영자로서 상차/하차 특이사항이 있는지 숫자로 먼저 보고 싶다. 그래야 위험하거나 확인이 필요한 오더를 놓치지 않는다. |

## 5. Functional Requirements

| REQ-ID | 우선순위 | 요구사항 | 수용 기준 |
| --- | --- | --- | --- |
| REQ-transport-route-001 | Must | 데이터 있음 상태에서 상차지와 하차지를 세로 2행으로 표시한다. | 상차지 행이 먼저, 하차지 행이 다음에 표시된다. |
| REQ-transport-route-002 | Must | 각 행은 구간명, 지명, 주소, 일시, 방법, 특이사항 배지를 표시한다. | D안 기준 접힘 상태에서는 라벨 과다 없이 한 행에 핵심 정보가 보인다. |
| REQ-transport-route-003 | Must | 상세주소는 기본 요약에서 숨긴다. | 상세주소 텍스트는 펼침 영역에서만 보인다. |
| REQ-transport-route-004 | Must | 펼침 상태에서 상세주소, 담당자명, 연락처, 특이사항 요약을 표시한다. | 상차/하차 각각의 상세 정보가 구분되어 보인다. |
| REQ-transport-route-005 | Must | 상차지명 또는 하차지명이 없으면 해당 줄을 렌더링하지 않는다. | 빈 값에 `-` 또는 빈 줄이 보이지 않는다. |
| REQ-transport-route-006 | Should | 주소가 없는 상태에서 입력 전 빈 상태를 표시한다. | 주소 검색 전임을 알리고 다음 행동을 안내한다. |
| REQ-transport-route-007 | Should | 특이사항은 긴 텍스트 대신 배지로 요약한다. | `특이사항 2`, `주의 1` 같은 짧은 표기가 보인다. |
| REQ-transport-route-008 | Could | HTML에서 데이터 있음, 빈 상태, 펼침 상태를 한 화면에서 비교한다. | 독립 HTML에서 세 상태를 모두 확인할 수 있다. |
| REQ-transport-route-009 | Should | 데이터 있음 상태의 표시 방식을 A/B/C/D 4가지로 비교한다. | `sheet-row`, `sheet-card`, `stacked-card`, `compact-hybrid`가 한 HTML에서 구분되어 보인다. |
| REQ-transport-route-012 | Should | D안의 닫힘, 열림, 입력 흐름을 별도 HTML에서 확인한다. | `transport-route-d-workflow.html`에서 주소 미적용, 상차 입력, 닫힘 요약, 상세 열림, 계산 완료 상태가 보인다. |
| REQ-transport-route-010 | Should | 거리와 기준 금액 보조 메타는 `주소 적용 여부`와 화물 운송정보의 `톤수/차종 선택 여부`를 함께 반영한다. | 톤수 또는 차종이 없으면 `차량 조건 선택 후 계산` 같은 대기 상태로 보인다. |
| REQ-transport-route-011 | Should | 배차 유형 정보를 운송구간 전체 보조 정보로 표시한다. | 대표 유형은 `독차` 기본값 또는 `혼적` 중 하나만 보이고, `긴급`, `왕복`, `예약`은 선택된 항목만 멀티옵션 칩으로 보인다. 세 옵션이 모두 미선택일 수 있다. |
| REQ-transport-route-013 | Must | B 통합본에서 운송구간은 좌우 2컬럼 전체 폭을 사용한다. | `Cargo Order Wireframe B Original Tone.html`의 1번 운송구간이 `workspace` 전체 폭으로 표시된다. |
| REQ-transport-route-014 | Must | 기존 5번 `배차 유형`, 6번 `상차 조건`, 7번 `하차 조건`은 독립 섹션이 아니라 운송구간 내부 정보로 병합한다. | 배차 대표값/멀티옵션, 상차방법/상차일시, 하차방법/하차일시가 운송구간 요약, 펼침 상세, 보조 메타 중 하나에서 확인된다. |

## 6. UX Requirements

| 항목 | 요구사항 |
| --- | --- |
| 정보 밀도 | 기본 요약은 상차지/하차지 각 3줄 이내를 목표로 한다. |
| 레이아웃 | 운송구간 내부는 한 컬럼 안의 세로 2행 구조를 유지한다. |
| 라벨 | 반복 라벨은 줄이고 `[상차지]`, `[하차지] 수준의 구분만 유지한다. |
| 빈 상태 | 빈칸, 점선, 안내문으로 미입력 상태를 명확히 보여준다. |
| 접근성 | 특이사항 배지는 숫자만 단독으로 쓰지 않고 의미 텍스트를 함께 둔다. |
| 모바일 | 상차지/하차지 행 구조는 그대로 유지하고, 내부 텍스트만 줄바꿈한다. |
| 표시 방식 비교 | D안을 1차 기준안으로 두고, A/B/C는 비교 근거로 유지한다. |
| 계산 의존성 | 거리와 기준 금액은 주소만으로 확정하지 않고, `톤수/차종 선택 필요` 상태를 명확히 보여준다. |
| 배차 유형 | 상차/하차 개별 행 안에 반복하지 않고, 운송구간 전체에 적용되는 칩 묶음으로 표시한다. 대표 선택은 `독차` 기본, `혼적` 전환 가능으로 보고, `긴급/왕복/예약`은 독립 멀티선택으로 다룬다. |

## 7. Technical Considerations

- 단독 `transport-route-section.html`은 제거하고, 비교 기준은 `transport-route-section-variants.html`로 통합한다.
- 기존 전체 기준 HTML 파일은 수정하지 않는다.
- D안 워크플로우 검토는 `transport-route-d-workflow.html`을 기준으로 한다.
- 실제 데이터 바인딩은 포함하지 않고 정적 상태 샘플로 표현한다.
- 향후 구현 시 데이터 모델은 `pickup`과 `dropoff` 객체로 분리한다.
- 기본 요약에 표시할 주소는 `시/도 + 시/군/구 + 읍/면/동 + 법정리` 수준으로 제한한다.
- 거리/기준 금액/운송료 보조 메타는 운송구간의 주소 데이터와 화물 운송정보의 `톤수`, `차종` 값을 함께 참조하는 cross-section dependency로 본다.
- 배차 유형은 기존 B안의 5번 섹션 값과 `Cargo Order Wireframe.html`의 배차 유형 칩을 참조해 운송구간 전체 summary metadata로 반영한다. 대표 선택은 `독차` 기본값과 `혼적` 전환값으로 제한하고, `긴급/왕복/예약`은 선택된 값만 표시한다. B 통합본에서는 별도 5번 섹션을 제거하고 운송구간 내부로 병합했다.

## 9. Risks & Mitigations

| 리스크 | 수준 | 대응 |
| --- | --- | --- |
| 요약이 너무 짧아 실제 운영 정보가 부족할 수 있음 | medium | 펼침 영역에 담당자/상세주소/특이사항을 보강 |
| 특이사항 배지가 실제 위험도를 충분히 표현하지 못할 수 있음 | medium | 후속 단계에서 중요도 배지 또는 색상 단계 검토 |
| 기존 입력 폼 사용자에게 수정 위치가 낯설 수 있음 | medium | 요약 행 클릭 또는 펼침 후 `수정` 진입을 명확히 표시 |
| 주소/일시 포맷이 데이터마다 다를 수 있음 | low | 표시 포맷 규칙을 필드 매핑에 명시 |

## 10. Success Metrics

| 지표 | 목표 |
| --- | --- |
| 기본 경로 확인 시간 | 기존 입력 필드형 대비 30% 이상 단축 |
| 기본 화면 정보량 | 상차/하차 각각 3줄 이내 유지 |
| 빈 상태 이해도 | 주소 입력 전 상태와 다음 행동이 1문장으로 이해됨 |
| QA 기준 | 상세주소가 기본 요약에 노출되지 않음 |
| 매핑 완전성 | PRD 요구사항 100%가 와이어프레임과 HTML에 매핑됨 |
