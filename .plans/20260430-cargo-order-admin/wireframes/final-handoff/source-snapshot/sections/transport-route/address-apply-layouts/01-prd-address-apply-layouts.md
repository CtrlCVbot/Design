# PRD: 운송구간 주소 검색/적용 레이아웃

## 목적

`운송구간 주소 검색/적용`은 상차지와 하차지의 장소 정보, 주소, 상세주소, 현장 담당자 정보를 한 줄에서 확인하고, 변경이 필요할 때 주소 검색/적용 다이얼로그로 진입하는 하위 흐름입니다.

이번 기준에서는 2열 비교안, 오른쪽 메타 컬럼, 펼침 구조를 제거합니다. 상차와 하차는 각각 한 행으로 표시하고, 각 행 오른쪽 끝에 변경 버튼을 둡니다.

## 사용자 문제

| 문제 | 영향 |
| --- | --- |
| 펼침 구조가 있으면 기본 행에서 상세주소와 담당자 정보를 바로 확인하기 어려움 | 운영자가 상세 확인을 위해 한 번 더 열어야 함 |
| 주소와 상세주소가 같은 수정 방식으로 보이면 좌표 기준이 흔들릴 수 있음 | 거리/운임 판단 오류 가능 |
| 오른쪽 메타가 주소 적용 화면에 함께 있으면 주소 정리 목적이 흐려짐 | 사용자가 변경해야 할 대상이 분산됨 |
| 상차/하차 변경 버튼이 명확하지 않으면 주소 검색 진입점이 약해짐 | 수정 흐름이 불명확해짐 |

## 사용자 목표

| 사용자 | 목표 |
| --- | --- |
| 오더 접수자 | 상차지와 하차지의 핵심 정보를 한 줄에서 확인한다 |
| 배차 운영자 | 상차지/하차지 변경 버튼으로 바로 주소 검색에 진입한다 |
| 정산/운임 담당자 | 주소 정보와 운송 조건 메타가 섞이지 않은 상태로 주소만 확인한다 |

# 4차 추가 결정: 주소검색 다이얼로그 레이아웃과 미리보기 모드

이번 보강의 목표는 주소검색 다이얼로그를 화주 정보 섹션의 `화주/담당자 검색` 다이얼로그와 같은 정보 구조로 맞추는 것이다. 검색 영역, 출처 필터, 조회 결과 리스트, 선택 미리보기, 하단 적용 버튼의 흐름을 동일하게 두어 사용자가 섹션마다 다른 조작 방식을 다시 학습하지 않도록 한다.

## 추가 요구사항

| ID | 요구사항 | 상태 |

| `REQ-TR-ADDR-019` | `화주 등록` 주소를 선택하면 선택 미리보기는 기본적으로 라벨 모드로 보여준다 | 4차 반영 |
| `REQ-TR-ADDR-020` | `화주 등록` 주소의 상세주소, 지명, 담당자, 연락처는 `수정` 버튼을 눌렀을 때만 입력 컴포넌트로 전환한다 | 4차 반영 |
| `REQ-TR-ADDR-021` | `Kakao` 주소를 선택하면 상세주소, 지명, 담당자, 연락처를 바로 보강해야 하므로 선택 미리보기를 입력폼 모드로 보여준다 | 4차 반영 |
| `REQ-TR-ADDR-022` | 주소 원문은 좌표/정산 기준값 성격이 있으므로 `화주 등록`, `Kakao` 모두 미리보기에서 직접 수정하지 않는다 | 4차 유지 |

## UX 판단

`화주 등록` 주소는 이미 선택된 화주의 주소록 데이터이므로, 기본 상태를 라벨로 두는 편이 안전하다. 운영자는 등록된 주소를 확인하고 바로 적용할 수 있고, 실제 변경 의도가 있을 때만 `수정`으로 입력폼에 진입한다.

`Kakao` 주소는 외부 검색 결과라 상세주소, 지명, 담당자, 연락처가 부족할 가능성이 높다. 따라서 선택 즉시 입력폼을 보여주는 것이 더 자연스럽다. 특히 Kakao 주소를 주소록에 저장하려면 지명 또는 장소명이 필요하므로, 저장 조건을 사용자가 바로 이해할 수 있다.
