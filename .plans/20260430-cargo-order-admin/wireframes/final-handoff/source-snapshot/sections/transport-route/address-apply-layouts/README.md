# 운송구간 주소 검색/적용 레이아웃 패키지

## 목적

이 패키지는 `transport-route-address-apply-layouts.html`을 기준으로, 운송구간의 주소 검색/적용 흐름과 1열 단일 행 레이아웃을 독립적으로 정리한 1차 산출물입니다.

기존 `transport-route-d-workflow.html`은 그대로 유지합니다. 이 패키지는 D workflow를 대체하지 않고, 주소 검색/적용 화면만 별도로 정리합니다.

## 패키지 상태

| 항목 | 상태 | 메모 |
| --- | --- | --- |
| 1차 정리 | 완료 | 1열 단일 행, 상차/하차 변경 버튼, 주소 검색/적용 다이얼로그 정리 |
| HTML 구현 | 완료 | `../transport-route-address-apply-layouts.html` 생성 |
| B 통합본 반영 | 보류 | HTML 확인 후 확정안만 반영 |
| 기존 D workflow | 유지 | `../transport-route-d-workflow.html` 미수정 |

## 기준 항목

| 항목 | 역할 | 1차 처리 |
| --- | --- | --- |
| 상차지명 | 상차 장소 식별 | 기본 행에 표시 |
| 상차지주소 | 상차 행정주소 | 기본 행에 표시, 주소 검색/적용으로만 변경 |
| 상차지상세주소 | 현장 진입 보조 정보 | 기본 행에 표시 |
| 상차 담당자 | 현장 연락 대상 | 기본 행에 표시 |
| 상차 연락처 | 현장 연락 번호 | 기본 행에 표시 |
| 상차일시 | 상차 시간 조건 | 기본 행에 `지금` 표시 |
| 상차방법 | 상차 작업 방식 | 기본 행에 `지게차` 표시 |
| 하차지명/주소/상세/담당자/연락처/일시/방법 | 하차 정보 | 상차와 같은 구조로 기본 행에 표시 |

## 1차 결정

| 결정 | 내용 |
| --- | --- |
| HTML 위치 | 현재 사용 중인 URL을 유지하기 위해 HTML은 `transport-route` 루트에 둔다 |
| 문서 위치 | 이 하위 패키지에서 HTML 목적, 요구사항, 와이어프레임, closeout을 관리한다 |
| 표시 방식 | 1열 단일 행. 상차와 하차를 각각 한 줄 명세표로 표시한다 |
| 제거 범위 | 이 하위 패키지에서는 2열 비교안, 오른쪽 메타 컬럼, 펼침 구조를 제거한다 |
| 변경 버튼 | 상차 행 끝에는 `상차지 변경`, 하차 행 끝에는 `하차지 변경`을 둔다 |
| 주소 변경 | 지명과 행정주소는 주소 검색/적용 다이얼로그에서만 변경한다 |
| 상세 정보 | 상세주소, 담당자, 연락처는 기본 행에 표시한다 |
| 통합 검색 | 화주 등록 주소록과 Kakao API 결과를 같은 리스트에서 출처 배지로 구분한다 |
| Kakao 결과 저장 | 지명 또는 장소명을 입력해야 화주 주소록 저장이 가능하다 |
| 화주 등록 수정 | 상세주소, 지명, 담당자, 연락처 변경이 있을 때만 주소록 수정 저장이 가능하다 |
| 주소 수정 | 주소는 읽기 전용으로 표시하고 직접 수정하지 않는다 |
| B 통합본 반영 | HTML 확인 후 확정된 1열 단일 행 구조만 반영한다 |

## 사용 방법

1. `../transport-route-address-apply-layouts.html`을 브라우저에서 열어 `1열 정리안`, `주소 검색/적용`, `상세 수정` 탭을 확인합니다.
2. `상차지 변경` 또는 `하차지 변경` 버튼을 눌러 주소 검색/적용 다이얼로그가 열리는지 확인합니다.
3. `02-wireframe-address-apply-layouts.md`에서 최종 행 구조와 user flow를 확인합니다.
4. `03-closeout-address-apply-layouts.md`에서 1차 기준과 B 통합본 인계 기준을 확인합니다.
5. `04-kakao-shipper-address-search-plan.md`에서 화주 주소록 + Kakao 통합 검색 추천안을 확인합니다.

## 다음 단계

1. HTML에서 1열 단일 행의 가독성을 확인합니다.
2. 행이 너무 길 경우 컬럼 폭과 내부 스크롤 정책을 조정합니다.
3. Kakao 결과를 주소록에 저장할 때 저장 API 호출 시점을 결정합니다.
4. 확정된 구조만 `Cargo Order Wireframe B Original Tone.html`에 반영합니다.
# 4차 추가 결정: 주소검색 다이얼로그 UX

| 결정 | 내용 |
| --- | --- |
| 다이얼로그 구조 | 화주 정보 섹션의 `화주/담당자 검색` 다이얼로그와 같은 검색/필터/조회 결과/선택 미리보기/하단 적용 구조를 사용 |
| 화주 등록 미리보기 | 선택 직후 라벨 모드로 보여주고, `수정` 버튼을 눌렀을 때만 입력폼으로 전환 |
| Kakao 미리보기 | 선택 직후 입력폼 모드로 보여주고, 상세주소/지명/담당자/연락처를 바로 보강 |
| 주소 필드 | 주소 원문은 수정하지 않고 읽기 전용으로 유지 |
| 주소록 저장 | 체크박스 위치는 항상 유지하되, 조건 충족 전에는 비활성화 |

## 추천 의견

현재 방향을 유지하는 것을 추천한다. `화주 등록` 주소는 이미 관리 중인 주소록 데이터이므로 라벨 우선이 안전하고, `Kakao` 주소는 새 주소 후보라 입력폼 우선이 빠르다. 두 출처의 신뢰도와 입력 필요성이 다르기 때문에, 미리보기 모드를 다르게 두는 것이 UX적으로 더 명확하다.
