# Wireframe: 화물 운송정보 섹션

## 설계 목표

현재 B 통합본의 색감과 밀도는 유지하되, 화물 운송정보 섹션의 기준 필드를 더 작게 정리합니다.

운송 조건은 `톤수`, `차종`, `대수`, `실중량`만 남깁니다. `실중량`은 톤수 110% 자동 입력 후 수정 가능한 입력창으로 두고, `품목`은 같은 박스의 2번째 row로 표시합니다. `광폭`은 이번 섹션 기준 필드에서 제외하고, `도착`은 운송구간과 중복되므로 제외합니다.

금액 조건에는 `결제방법` selector를 포함합니다. 금액 입력 필드는 `청구비용`을 맨 앞에 둡니다. 결제방법이 `인수증`이든 `선착불`이든 컴포넌트 위치는 동일하게 유지하고, 상태와 표시만 바뀌게 설계합니다.

## 화면 식별자

| Screen ID | 이름 | 목적 |
| --- | --- | --- |
| `SCR-CT-001` | 기본 정리형 | 운송 조건 4개와 품목을 `운송+품목` 2행으로 묶고, 결제방법과 금액 입력 4개는 별도 금액 row로 배치 |
| `SCR-CT-002` | 인수증 상태 | 수수료를 숨김/비활성으로 처리하고 결과 라벨을 `수익`으로 표시 |
| `SCR-CT-003` | 선착불 상태 | 수수료를 표시하고 차주에게 청구되는 `차주운임`을 표시 |


## B 통합본 최종 채택안

B 통합본에서는 F안의 정보 구조를 유지하되, 실제 오더 접수 화면의 밀도에 맞춰 입력 전과 적용 후 상태를 아래 기준으로 고정합니다.

| 상태 | 최종 기준 |
| --- | --- |
| 입력 전 `운송+품목` | `[운송+품목] placeholder [운송+품목 입력] [톤수/차종/품목]` 한 줄, row 높이 29px 기준 |
| 입력 전 `금액` | `[금액] placeholder [금액 조건 선택] [인수증 기본]` 한 줄, row 높이 29px 기준 |
| 입력 전 보조 chip | dashed muted 스타일로 낮춰 보이고, CTA 버튼이 가장 먼저 보이게 함 |
| 적용 후 `운송` row | `[운송] [톤수] [차종] [대수] [실중량 + 110% 이하]` 한 줄, row 높이 28px 기준 |
| 적용 후 `품목` row | `[품목] [장비 운송]` 한 줄, row 높이 29px 기준. 내부 중복 `품목` 라벨은 숨김 |
| 적용 후 `금액` row | `[금액] [결제방법] [청구비용] [운송비용] [수수료] [수익/차주운임] [계산식]` 한 줄, row 높이 28px 기준 |
| 결과 강조 | `수익/차주운임` 셀만 결과값으로 강조하고, `청구-운송` 공식은 보조 정보로 표시 |
| 수정 배지 | 고정 노출하지 않고 각 셀 hover/focus에서만 `수정` 배지를 노출 |

최종 채택안의 목적은 “입력 전에는 버튼을 찾기 쉽고, 적용 후에는 데이터만 빠르게 스캔되는” 상태를 만드는 것입니다. 따라서 구조 설명성 chip은 약하게 두고, 실제 오더 데이터와 결과 금액을 우선합니다.

## 제안 A: 기본 정리형 (`SCR-CT-001`)

F안은 `운송+품목`과 `금액` 2개 그룹으로 나누되, B 통합본의 compact row 감각을 유지하는 안입니다.

```text
[3 화물 운송정보]                                   [기준 금액: 톤수·차종 선택 후 확정]

운송        톤수        차종        대수        실중량
            5톤         축카고      1           5.50
품목        장비 운송

금액        결제방법      청구비용      운송비용      수수료        수익/차주운임
            인수증        140,000       100,000       -             40,000
```

장점:
- 기존 화면의 위치 감각을 크게 해치지 않습니다.
- `광폭`, `도착`이 운송 조건 row에서 빠져 row가 더 compact해집니다.
- 청구/운송/수수료/수익·차주운임의 정산 의미가 한 줄에서 보입니다.
- `품목`은 `운송+품목` 박스의 2행으로 묶어 운송 조건과 함께 읽을 수 있습니다.

주의:
- 결과 금액 라벨이 인수증에서는 `수익`, 선착불에서는 `차주운임`으로 바뀌므로 보조 상태 문구가 필요합니다.

## 제안 B: 인수증 상태 (`SCR-CT-002`)

인수증 조건에서는 `수수료`를 숨김 또는 비활성 상태로 둡니다. 위치는 유지합니다.

```text
[3 화물 운송정보]                                   [결제방법: 인수증]

운송 조건
톤수        차종        대수        실중량
5톤         축카고      1           5.50

금액 조건
결제방법      인수증
청구비용      운송비용      수수료        수익
140,000       100,000       -             40,000
                           숨김/비활성     수익

품목        장비 운송
```

계산:

```text
청구비용 - 운송비용 = 수익
```

## 제안 C: 선착불 상태 (`SCR-CT-003`)

수수료를 입력하면 자동으로 선착불 상태가 됩니다. `청구비용`은 `0원`으로 고정하고, 포커스는 `운송비용`으로 이동합니다.

```text
[3 화물 운송정보]                                   [결제방법: 선착불]

운송 조건
톤수        차종        대수        실중량
5톤         축카고      1           5.50

금액 조건
결제방법      선착불
청구비용      운송비용      수수료        차주운임
0             100,000       20,000        120,000
고정          포커스        입력값        차주 청구 운임

품목        장비 운송
```

계산:

```text
운송비용 + 수수료 = 차주운임
```

## 입력 동작

| 사용자 행동 | 시스템 반응 |
| --- | --- |
| `결제방법` 선택 | `인수증`, `선불`, `착불`, `선착불`에 따라 수수료 표시와 결과 라벨/계산식 변경 |
| `톤수` 선택 | `실중량`에 톤수의 110%를 `0.00` 단위로 자동 입력 |
| `실중량` 수정 | 자동 입력값은 유지하지 않고 사용자가 수정한 값을 사용 |
| `청구비용` 입력 | `운송비용`에 같은 금액 자동 입력 |
| `운송비용` 수정 | `청구비용`은 유지하고 `운송비용`만 변경 |
| `수수료` 입력 | 결제방법을 `선착불`로 자동 변경 |
| `수수료` 입력 후 | `청구비용=0원` 고정, 포커스는 `운송비용`으로 이동 |
| 결제방법 변경 | 컴포넌트 위치는 그대로 두고 표시/숨김/비활성 상태만 변경 |

## 상태별 wireframe

### 값 없음

```text

[운송+품목]  -------------------------------  [운송+품목 입력]  [톤수/차종/품목]

[운송]  [톤수: 톤수 선택]  [차종: 차종 선택]  [대수: 1대]  [실중량: 0.00톤]

금액
```

값 없음 상태의 CTA 전환은 `08-empty-state-activation-transition.md`를 기준으로 합니다. `운송+품목 입력`, `금액 조건 선택` CTA가 각 라벨 방향으로 이동해 흡수된 뒤 다이얼로그를 열고, 적용 후 항상 노출 row로 전환됩니다.

## 다음 HTML 작업 기준

HTML variants 또는 B 통합본을 수정할 때는 다음을 반드시 확인합니다.

1. `광폭`, `도착`이 기준 운송 조건에서 빠져 있는가
2. `결제방법`이 금액 조건에 포함되어 있는가
3. 금액 입력 필드 중 `청구비용`이 첫 번째 위치에 있는가
4. `품목`이 `운송+품목` 박스의 2번째 row로 배치되어 있는가
5. `실중량`이 `대수` 오른쪽에 있고 톤수 110% 자동 입력/수정 가능 상태로 보이는가
6. 인수증과 선착불 상태에서 필드 위치가 동일한가
7. 인수증은 `수익`, 선착불은 `차주운임`으로 표기되는가
8. `수수료`는 조건부 표시로 처리되는가
9. 값 없음 상태에서 `운송+품목 입력`, `금액 조건 선택` CTA 흡수 전환이 확인 가능한가
10. 입력 전 row와 적용 후 row의 높이/폭 리듬이 B 통합본 최종 채택안과 맞는가
11. 보조 chip과 계산식이 실제 데이터보다 강하게 보이지 않는가
12. 적용 후 다이얼로그가 닫히고 CTA fallback이 중복 실행되지 않는가
