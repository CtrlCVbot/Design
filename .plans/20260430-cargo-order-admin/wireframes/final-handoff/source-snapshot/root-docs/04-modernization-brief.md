# 04. UX/UI 현대화 준비 메모

## 핵심 관찰

현재 화면은 숙련 운영자에게 빠른 입력을 제공하는 대신, 초보자에게는 어떤 정보가 필수이고 어떤 버튼을 눌러야 하는지 파악하기 어렵습니다. 개선의 목표는 기능을 줄이는 것이 아니라, 업무 순서와 위험도를 더 분명하게 만드는 것입니다.

## 보존해야 할 업무 가치

| 가치 | 현재 화면에서의 근거 | 새 UI 설계 시 보존 방식 |
| --- | --- | --- |
| 빠른 등록 | `신규(F3)`, 등록/수정 버튼이 항상 보임 | 고정 액션 바와 단축키 유지 |
| 한 화면 확인 | 입력 폼과 목록이 동시에 보임 | 데스크톱에서는 편집 패널과 목록 동시 노출 |
| 반복 입력 절감 | 같은 주소와 같은 운송+품목 조합을 반복 입력하는 업무가 많음 | 다이얼로그 안에 최근 사용 후보를 보조 상태로 제공 |
| 운영 책임 가시성 | 접수/수정 중 누가 배차를 담당하는지 빠르게 확인해야 함 | header 상태 chip 옆에 배차 담당자 chip을 상시 노출 |
| 배차 조건 상세성 | 독차/혼적, 상하차 방법, 시간, 차종, 톤수 제공 | 조건 그룹을 유지하고 선택 방식을 정리 |
| 정산/증빙 연결 | 운송료, 수수료, 계산서, 인수증, 사업자번호가 같은 흐름에 있음 | 운임/정산 섹션을 명확히 분리 |
| 운영 상태 파악 | 접수/완료/취소 색상, 오더 현황, 미확인 쪽지 | 상태 배지, 카운터, 필터로 재구성 |

## 현재 UX 문제

| 문제 | 영향 | 개선 방향 |
| --- | --- | --- |
| 정보 밀도가 매우 높음 | 초보자가 입력 순서를 놓치기 쉬움 | 업무 순서대로 섹션화 |
| 색상 의미가 분산됨 | 빨강/파랑/노랑이 상태, 경고, 강조에 섞임 | 상태 색과 경고 색 규칙 분리 |
| 버튼이 많고 우선순위가 약함 | 등록, 수정, 취소 같은 핵심 액션이 보조 액션과 섞임 | 주요/보조/위험 액션 구분 |
| 체크박스가 과다함 | 독립 옵션과 상호 배타 옵션 구분이 어려움 | 라디오, 세그먼트, 체크박스 역할 재정리 |
| 시간 입력 규칙이 텍스트 경고에 의존 | 잘못 입력해도 사전 예방이 약함 | 조건부 입력 제한과 즉시 검증 |
| 목록 컬럼이 너무 넓고 많음 | 오른쪽 컬럼이 잘리고 핵심 정보 스캔이 어려움 | 고정 컬럼, 컬럼 그룹, 사용자별 컬럼 저장 |
| 민원/광고/업무 정보가 섞임 | 업무 집중도가 낮아짐 | 전역 정보와 광고성 정보 분리 |
| 비활성 버튼 사유가 보이지 않음 | 왜 클릭할 수 없는지 알기 어려움 | disabled tooltip 또는 조건 메시지 제공 |
| 반복 입력을 매번 검색/입력함 | 같은 장소와 같은 운송+품목 조합을 다시 입력해야 함 | 최근 사용 후보를 보여주되 기존 적용 버튼으로만 확정 |

## 필수 입력 후보

실제 필수 여부는 기존 프로그램 규칙 확인이 필요하지만, 화면과 업무 맥락상 아래 항목은 새 UI에서 필수 또는 조건부 필수로 다루는 것이 안전합니다.

| 항목 | 권장 필수성 | 이유 |
| --- | --- | --- |
| 상차지 | 필수 | 거리, 운임, 배차의 기준 |
| 하차지 | 필수 | 거리, 운임, 배차의 기준 |
| 톤수/차종/대수 | 필수 | 배차 가능 차량 결정 |
| 운송료 | 필수 | 정산과 차주 모집에 직접 영향 |
| 화물정보 | 필수 | 차주가 운송 조건을 이해하는 핵심 정보 |
| 상차일시 | 필수 또는 조건부 필수 | 즉시/예약 배차 판단 |
| 하차일시 | 조건부 필수 | 당착/내착/월착 조건 판단 |
| 의뢰자구분 | 필수 | 실적신고와 사업자번호 검증에 영향 |
| 의뢰자 사업자번호 | 조건부 필수 | 의뢰자가 운수사인 경우 필요하다는 경고가 있음 |
| 차주/차량번호 | 배차 후 필수 | 완료/배차 상태 관리 |

## 입력 컴포넌트 재해석

| 현재 컨트롤 | 문제 | 권장 컴포넌트 |
| --- | --- | --- |
| `즉시표시`, `차주앱 표시일시` 라디오 | 노출 시점의 차이가 작게 보임 | `노출 방식` 세그먼트 컨트롤 |
| `독차`, `혼적`, `선택` | 상호 배타인지 복수 선택인지 불명확 | 독차/혼적은 세그먼트, 추가 조건은 드롭다운 |
| 상하차 방법 체크박스 | 장비 선택은 복수 가능해 보임 | 체크박스 그룹 유지, 아이콘 보조 |
| 상하차 일시 체크박스 | 날짜/상대일/시간 단위가 섞임 | 날짜 선택기 + 빠른 선택 칩 + 시간 선택 |
| 운송료/수수료/합계 | 계산 관계가 시각적으로 약함 | 금액 입력 + 계산 요약 카드 |
| 화물정보 긴 텍스트 | 중량 입력 금지 안내가 별도 텍스트 | 중량은 별도 필드로 강제, 메모는 placeholder 안내 |
| 목록 상태 색상 | 색상만으로 상태 구분 | 상태 배지 + 색상 + 텍스트 |

| `보조 정보` | 메모, 금액 로그, 운송 구간 지도 | 선택 오더의 운영 판단 정보를 입력 폼과 분리 |

## 2026-06-18 HiFi 디자인 정리 기준

최신 master는 기능 추가뿐 아니라 화면 밀도와 시각 계층을 정리한 상태를 기준으로 합니다.

| 영역 | 기준 |
| --- | --- |
| 그림자 체계 | `--shadow-xs`를 보강하고 카드, 영수증, 메모, dialog의 shadow 계층을 얕고 일관되게 정리 |
| 버튼 위계 | primary, secondary, ghost, danger가 시각적으로 구분되도록 secondary에 경계와 미세 shadow를 부여 |
| 입력 focus | 과한 4px ring 대신 저투명 3px focus ring을 사용 |
| dialog title | dialog 제목 스케일을 섹션 헤더와 같은 수준으로 통일 |
| warning token | `--warning-500`을 보강해 경고 계열 토큰 누락을 방지 |
| 운송 구간 | 운송 구간 섹션 내부의 중복 계산 메타는 삭제하고, header chip과 보조 정보 지도 탭으로 역할 분리 |
| 차주 섹션 | 데이터 적용 상태의 padding/margin을 다른 섹션과 맞춰 안쪽으로 밀려 보이지 않게 정리 |
| 라벨 토글 | `Aa` icon button을 header 상태 흐름 안에 배치하고 tooltip은 한글로 표시 |
| 배차 담당자 | 1번 배치안으로 확정. header 상태 chip 옆에 `배차 김민지` chip을 표시하고 독립 섹션은 추가하지 않음 |

## 운송 관련 최근 사용 리스트 기준

최근 사용 리스트는 기존 검색이나 입력폼을 대체하지 않는 보조 상태입니다. 사용자가 최근 항목을 눌러도 즉시 저장하지 않고, 다이얼로그 안의 미리보기 또는 입력폼만 갱신합니다.

| 대상 | 기준 |
| --- | --- |
| 상차지/하차지 주소 검색 | 다이얼로그 open 시 기존 검색 결과 영역에 최근 장소를 표시하고, `조회` 또는 Enter 후 기존 검색 결과로 전환 |
| 주소 미리보기 | 기존 `선택 미리보기` 구조를 유지하고, 최근 장소 선택도 적용 전 상태로만 표시 |
| 운송+품목 입력 | `톤수 + 차종 + 대수 + 실중량 + 품목` 조합 단위의 최근 후보를 표시 |
| 운송+품목 반영 | 최근 조합 클릭은 입력폼만 채우고, `운송+품목 적용` 전까지 `화물 정보` row에 반영하지 않음 |
| 개인정보 | 주소 row에는 연락처를 과하게 노출하지 않고, 상세 확인은 미리보기에서 제한적으로 처리 |

## 신규 접수 플로우 현대화 기준

신규 접수는 기존 화물 조회/수정과 다른 모드로 다룬다. 기존 화물을 선택해 보거나 수정할 때는 이미 본문 라벨과 값이 있으므로 섹션 헤더를 숨겨 명세서처럼 읽히게 한다. 반대로 `신규(F3)` 이후에는 입력 순서를 안내하기 위해 신규 접수 전용 섹션 헤더와 wizard 다이얼로그를 사용한다.

| 항목 | 기준 |
| --- | --- |
| 신규 접수 시작 | 전체 입력 상태 초기화 후 `new-reset` 진입, 화주 정보 focus |
| 최초 입력 | wizard 다이얼로그로 진행하고 왼쪽 프로세스 패널 표시 |
| 필수 단계 | 화주 정보, 상차지, 하차지, 화물 정보, 정산 정보 |
| 금액 후 분기 | `화물 등록 완료`는 API 없이 메인 화면 적용, `차주 정보로 이동`은 선택 단계 진입 |
| 등록 후 수정 | wizard를 재개하지 않고 기존처럼 독립 다이얼로그 사용 |
| 오더 요약 | 입력 단계가 아닌 확인용 섹션. 신규 접수 안내형 보기에서만 회색 `확인` 배지 사용 |
| 최종 API | 메인 화면의 `화물등록` 버튼만 실제 API 통신 실행. endpoint 연결은 보류 |

## 보조 정보 패널 현대화 기준

우측 `예약 영역`은 `보조 정보`로 이름을 정리합니다. 보조 정보는 신규 접수의 필수 입력을 늘리는 영역이 아니라, 선택된 오더를 확인하거나 수정할 때 운영자가 참고할 후속 정보를 모아 보여주는 영역입니다.

| 항목 | 기준 |
| --- | --- |
| 패널명 | `보조 정보` |
| 탭 순서 | `메모`, `금액 로그`, `운송 구간 지도` |
| 기본 탭 | `메모` |
| 메모 | 목록을 먼저 보여주고, 추가는 `메모 추가` 다이얼로그에서 처리 |
| 금액 로그 | `금액 명세서`처럼 기준 청구금, 청구 조정금, 기준 배차금, 배차 조정금, 수익을 컴팩트하게 표시 |
| 금액 용어 | 추가금/할인금을 나눠 부르지 않고 `조정금`으로 통일하며, `+`/`-`로 증감 방향을 확인 |
| 조정 대상 | 청구금과 배차금 모두 조정 가능. 최소 기준 청구금/배차금에 추가되는 증감 금액을 `조정금`으로 정리 |
| 상세 항목 | `상세 항목 보기`에서 사유, 대상, 금액, 담당자, 마지막 일시, 설명을 확인 |
| 운송 구간 지도 | 지도와 거리/예상 시간/경유 여부를 보여주고, 상차/하차 주소를 하단에 다시 중복 표시하지 않음 |
| 데이터 없음 | `신규(F3)` 클릭 후 데이터가 초기화되면 세 탭 모두 같은 톤의 빈 상태를 사용 |
| 부분 데이터 있음 | 화주 선택 시 메모 mock, 상차지+하차지 적용 시 지도 mock, 정산 정보 적용 시 금액 로그 mock을 표시 |
| 데이터 있음 | 기존 오더 조회/수정 기본 상태에서는 세 탭 모두 mock 데이터가 있는 상태로 확인 |

## Motion과 접근성 기준

attention motion은 다음 행동을 찾게 하는 보조 장치로만 사용한다. 흔들림이나 무한 반복처럼 업무 화면에서 산만해지는 표현은 피한다.

| 상황 | 기준 |
| --- | --- |
| 신규 접수 시작 | `화주 정보 입력` 버튼에 soft focus ring, glow pulse, highlight sweep 중 하나를 2회 이하로 적용 |
| 기본/등록 완료 명세서형 보기 | 빈 필수 입력 행은 좌측 2px 상태 바와 입력 버튼 reminder로 표시 |
| reminder | 최초 2회 pulse 후 멈추고, 값이 계속 비어 있으면 10초 간격으로 1회만 재실행 |
| 중단 조건 | 버튼 클릭, 값 입력 완료, 행 제거 시 motion 중단 |
| reduced motion | `prefers-reduced-motion: reduce`에서는 animation을 생략하고 정적 상태 표시만 유지 |

## 테이블 현대화 방향

| 개선 영역 | 권장안 |
| --- | --- |
| 핵심 컬럼 고정 | 상태, 화물ID, 처리시간, 상차지, 하차지, 운송료는 좌측 고정 |
| 컬럼 그룹 | `운송`, `운임`, `차주/차량`, `정산`으로 그룹화 |
| 상태 필터 | 전체/접수/완료/취소를 탭으로 분리 |
| 행 액션 | 수정, 복사, 취소, 배차취소는 행 선택 후 명확히 노출 |
| 검색 | 화물번호 외에도 차주전화, 차량번호, 상차지, 하차지 검색 가능성 검토 |
| 가독성 | 글씨 크기 조절은 `밀도: 보통/촘촘/크게` 설정으로 통합 |
| 내보내기 | 엑셀/인쇄는 목록 상단 우측 액션으로 배치 |

## 개선안 검수 체크리스트

| 체크 항목 | 통과 기준 |
| --- | --- |
| 필드 보존 | `02-field-inventory.md`의 항목이 새 UI에 모두 매핑됨 |
| 상태 보존 | 접수/완료/취소/선택 상태가 색상 외 텍스트로도 구분됨 |
| 금액 보존 | 운송료, 수수료, 합계, 수수료율, 계산서금액이 분리되어 보임 |
| 일정 보존 | 상차일시와 하차일시가 서로 다른 필드로 유지됨 |
| 조건부 검증 | `지금상` 시간 입력 금지, 운수사 사업자번호 필수 규칙이 UI에 반영됨 |
| 목록 기능 | 검색, 날짜, 상태 탭, 정렬, 페이징, 컬럼 너비 저장이 유지됨 |
| 후속 처리 | 계산서, 인수증, 정산, 서명확인, 위탁증 기능 진입점이 유지됨 |
| 보조 정보 | `메모`, `금액 로그`, `운송 구간 지도`의 목적이 겹치지 않고 데이터 있음/없음 상태가 모두 확인됨 |
| 최근 사용 | 상하차지와 운송+품목 최근 후보가 기존 검색/입력/적용 흐름을 대체하지 않고 보조 상태로 동작함 |
| 헤더 책임 정보 | 상태 chip, 거리/기준금액, 배차 담당자, 라벨 토글이 한 헤더 흐름에 있으며 본문 입력 섹션을 밀지 않음 |

## 의도적 미결정

| 항목 | 미결정 사유 | 다음 확인 |
| --- | --- | --- |
| 실제 필수 필드 | 캡처만으로 프로그램 검증 규칙을 확정할 수 없음 | 실제 저장 시 validation 확인 |
| 드롭다운 옵션 | 목록이 닫혀 있어 전체 옵션을 볼 수 없음 | 차종, 시간, 검색 기준 옵션 캡처 |
| 우측 끝 목록 컬럼 | 컬럼명이 잘려 있음 | 가로 스크롤 또는 전체 컬럼 캡처 |
| 권한별 버튼 활성화 | 비활성 사유를 알 수 없음 | 계정 권한/상태별 버튼 노출 확인 |
| 조정금 저장 단위 | 청구금/배차금 둘 다 조정되는 경우 단일 조정 item인지 대상별 분리 item인지 미확정 | 정산 데이터 모델과 감사 로그 정책 확인 |
| 보조 정보 권한 | 메모 작성/수정/삭제, 금액 로그 조회, 지도 조회 권한이 계정별로 다른지 미확정 | 운영 권한 정책 확인 |
| 최근 사용 정책 | 화주별/사용자별 scope, 저장 시점, 개인정보 표시 범위가 미확정 | `sections/transport-dialog-recent-lists/self-review.md` 기준으로 제품/개발 정책 확인 |
| 배차 담당자 데이터 | 현재는 HiFi 고정 예시이며 실제 담당자 배정/변경 데이터 모델은 미확정 | 배차 담당자 선택/변경 다이얼로그와 권한 정책 확인 |
