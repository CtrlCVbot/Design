# Component Design Audit

## 진단 요약

현재 clean HTML은 기획 설명이 제거되어 구현용 화면에 가까워졌지만, 컴포넌트 표현 방식은 섹션마다 아직 다릅니다. 특히 라벨 표기, 버튼 스타일, 입력 전 상태, 수정 가능 표시, 구분선 사용 방식이 통일되지 않았습니다.

| 항목 | 상태 | 영향 | 우선순위 |
| --- | --- | --- | --- |
| 라벨 표기 기준 | 불일치 | 같은 종류의 데이터가 다르게 읽힘 | 높음 |
| 라벨 표시 모드 | 없음 | compact와 상세 확인 요구를 동시에 만족하기 어려움 | 높음 |
| 버튼 역할 체계 | 불일치 | 사용자가 주요 action과 보조 action을 구분하기 어려움 | 높음 |
| 입력 전 상태 | 부분 불일치 | 섹션마다 시작 방법이 다르게 보임 | 높음 |
| 적용 후 상태 | 부분 불일치 | 어떤 값이 수정 가능한지 판단하기 어려움 | 중간 |
| 구분선/점선 | 부분 불일치 | row 경계와 수정 가능 영역이 혼재됨 | 중간 |
| hover/focus affordance | 불일치 | 마우스/키보드 진입 가능성이 섹션별로 다르게 보임 | 중간 |

## 섹션별 진단

### 1. 화주 정보

| 항목 | 현재 상태 | 문제 | 개선 방향 |
| --- | --- | --- | --- |
| 라벨 | `화주 업체명`, `사업자 번호`, `담당자명`, `담당자 연락처`, `담당자 이메일` 모두 표시 | 명확하지만 다른 섹션보다 라벨 밀도가 높음 | 신원/연락처 계열의 기준 라벨 패턴으로 삼기 |
| 버튼 | `화주/담당자 변경` 항상 표시 | 변경 action 위치는 명확함 | 다른 섹션의 변경 버튼도 이 위치/톤을 기준화 |
| 수정 가능 표시 | 연락처/이메일 inline edit 가능 | hover/focus 방식이 다른 섹션과 일부 다름 | inline edit 필드 전체에 같은 hover 규칙 적용 |
| row 구조 | 단일 row | 안정적 | 기준 row 구조로 유지 |

### 2. 운송 구간

| 항목 | 현재 상태 | 문제 | 개선 방향 |
| --- | --- | --- | --- |
| 라벨 | `상차`, `하차` kind만 있고 세부 필드는 값 중심 | 빠르게 읽히지만 초보 사용자는 필드 의미를 추론해야 함 | 위치/주소/상세/담당자/시간/방법의 암묵적 컬럼 규칙 문서화 |
| 버튼 | `상차지 변경`, `하차지 변경` row 끝에 표시 | 화주 변경 버튼과 기능상 유사하지만 시각 톤이 다름 | `변경 action` 스타일로 통일 |
| 수정 가능 표시 | 상세주소, 담당자, 연락처, 일시, 방법이 직접 수정/선택 가능 | 어떤 필드가 dialog인지 inline edit인지 한눈에 덜 보임 | 필드 유형별 affordance 규칙 필요 |
| 구분선 | 상차/하차 사이 dashed divider | row 구분에는 적절함 | 섹션 내부 row divider 기준으로 채택 |

### 3. 화물 운송정보

| 항목 | 현재 상태 | 문제 | 개선 방향 |
| --- | --- | --- | --- |
| 입력 전 상태 | `운송+품목 입력`, `금액 조건 선택` 버튼 중심 | 다른 섹션보다 점선과 버튼이 강하게 보임 | 입력 전 row 문법 통일 필요 |
| 라벨 | 입력 전에는 라벨이 적고, 적용 후에는 필드 라벨이 있음 | 입력 전/후 정보 밀도 차이가 큼 | 적용 후 라벨 기준을 다른 섹션에도 확장할지 결정 |
| 버튼 | `운송+품목 입력`, `금액 조건 선택` | 버튼 역할은 명확하나 디자인이 `변경` 버튼과 다름 | `입력 action`과 `변경 action`을 분리 정의 |
| 구분선 | 운송+품목과 금액 사이 dashed divider | 적절하지만 점선 스타일이 다른 구간과 달라 보일 수 있음 | divider 색/두께 기준 통일 |

### 4. 화물정보 요약

| 항목 | 현재 상태 | 문제 | 개선 방향 |
| --- | --- | --- | --- |
| 라벨 | `요약` kind만 표시, 세부 라벨은 숨김 | 현재 화면에서는 가장 간결함 | 단일 긴 텍스트 row의 기준으로 사용 가능 |
| 수정 버튼 | 기본 숨김, hover/focus 때 노출 | 다른 inline edit 필드보다 좋아 보임 | inline edit 가능 필드의 공통 규칙 후보 |
| 값 표현 | 긴 문장 ellipsis | 공간 효율은 좋지만 변경 감지는 약함 | stale 상태나 자동 생성 상태가 필요하면 별도 chip 사용 |

### 5. 차주 정보

| 항목 | 현재 상태 | 문제 | 개선 방향 |
| --- | --- | --- | --- |
| 기준 버전 | 화물맨 연동 버전만 표시 | 기준이 명확해짐 | 내부 DB 비교 UI는 별도 참고 자료로 분리 유지 |
| 입력 전 상태 | `화물맨 연동`, 안내문, `차주/차량 입력` | action이 2개라 역할 구분이 중요함 | 외부 연동 primary, 내부 입력 secondary로 명확화 |
| 하단 meta | 제거됨 | 중복 문구 감소 | 상태가 필요한 경우 row 안에 짧은 status만 유지 |
| 버튼 | 외부 연동과 일반 입력 버튼이 같은 row에 공존 | 시각 우선순위가 중요함 | 외부 연동 action 스타일 정의 필요 |

## 공통 불일치 항목

### CUA-001 라벨 노출 기준 불일치

| 구분 | 예시 | 문제 |
| --- | --- | --- |
| 명시 라벨 | 화주 정보의 `화주 업체명`, `사업자 번호` | 명확하지만 공간 사용량이 큼 |
| 암묵 컬럼 | 운송 구간의 주소, 상세주소, 담당자 | 빠르지만 기준이 없으면 구현자마다 다르게 표현 가능 |
| 혼합 라벨 | 화물 운송정보 적용 후 필드 라벨 | 같은 화면 안에서 라벨 규칙이 섞임 |

권장 판단:
기본은 compact 표시로 두되, 사용자가 `라벨 보기` 토글을 켜면 2행 구조로 전환합니다. 토글 OFF 상태에서도 특정 필드에 hover/focus하면 해당 필드 상단에 임시 라벨을 표시합니다.

```text
토글 OFF: 화주  로지스팩토리  123-45-67890  김민수
hover/focus: 해당 필드 상단에 `화주 업체명` 임시 라벨 표시
토글 ON:
화주   화주 업체명      사업자 번호      담당자명
       로지스팩토리     123-45-67890     김민수
```

### CUA-002 버튼 역할 체계 불일치

| 버튼 유형 | 현재 예시 | 필요한 기준 |
| --- | --- | --- |
| 변경 | `화주/담당자 변경`, `상차지 변경` | 이미 적용된 데이터를 다른 데이터로 바꿈 |
| 입력 | `운송+품목 입력`, `금액 조건 선택`, `차주/차량 입력` | 비어 있거나 미완성인 데이터를 채움 |
| 외부 연동 | `화물맨 연동` | 외부 서비스 action이며 primary 강조 필요 |
| 위험 | `연동 취소` | 취소/API 요청이므로 danger 필요 |
| 수정 | `수정` hover 버튼 | 같은 row 안에서 값만 수정 |

### CUA-003 입력 전 상태 문법 불일치

| 현재 경향 | 문제 |
| --- | --- |
| 어떤 섹션은 버튼만 있음 | 무엇을 입력하는지 약할 수 있음 |
| 어떤 섹션은 안내문이 있음 | 화면 밀도를 높임 |
| 어떤 섹션은 chip이 있음 | 정보인지 action인지 혼동 가능 |

권장 판단:
입력 전 상태는 `kind + 짧은 placeholder + primary/secondary action` 구조로 통일합니다. placeholder는 한 문장보다 짧은 구문을 사용합니다.

### CUA-004 수정 가능 표시 기준 불일치

| 방식 | 문제 |
| --- | --- |
| 항상 버튼 노출 | 반복 화면에서 시각 소음 증가 |
| hover/focus 때만 노출 | 깔끔하지만 발견성이 낮을 수 있음 |
| 점선만 표시 | 수정 가능성은 보이나 action이 약함 |

권장 판단:
inline edit 필드는 hover/focus 때만 `수정` 또는 시각 highlight를 보여주고, dialog 변경이 필요한 항목은 row 끝의 변경 버튼을 항상 보여줍니다.

## 우선순위

| 우선순위 | 항목 | 이유 |
| --- | --- | --- |
| 1 | 버튼 역할 체계 통일 | 사용자가 가장 먼저 액션을 판단함 |
| 2 | `라벨 보기` 토글과 2행 라벨 모드 확정 | compact와 상세 확인 요구를 동시에 해결 |
| 3 | hover/focus 임시 라벨 규칙 통일 | 토글 OFF 상태에서도 필드 의미를 확인 가능 |
| 4 | 입력 전 상태 문법 통일 | 데이터 없는 상태가 섹션마다 다르게 보이는 문제 해결 |
| 5 | hover/focus 수정 규칙 통일 | 실제 사용성과 접근성에 영향 |
| 6 | 구분선/점선 기준 정리 | 시각 완성도 개선 |
