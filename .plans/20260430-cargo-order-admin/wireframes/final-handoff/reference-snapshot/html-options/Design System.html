화물 오더 접수/수정 — 디자인 시스템 기반

고도화 직전 단계의 정리 + 기반 세우기입니다. 흩어진 이중 토큰(accent/ink + paper/pen)을 의미 기반 단일 세트로 통합하고, 섹션마다 중복 정의된 버튼·칩·필드를 6종 공통 컴포넌트로 추상화했습니다. 손글씨 폰트는 제거하고 차분한 고밀도 업무용 산세리프(Pretendard)로 통일합니다. 화면 구조와 필드는 그대로 둡니다.

색상 타이포 스페이싱·라운드·섀도 섹션 헤더 Button Field SummaryRow Chip · Badge Dialog EmptyState 확인 필요 결정

05 Button

primary / secondary / ghost / danger. 위험 액션(화물취소·배차취소·연동취소·정산처리)은 danger로 시각 분리하고, 액션바에서는 구분선으로 떼어 놓습니다.

danger 버튼은 평상시 빨강 외곽선·옅은 배경으로 두고, hover 시에만 솔리드로 채워 "한 번 더 생각" 신호를 줍니다. 정산처리·연동취소도 같은 계열로 통일합니다.

11 다음 단계 전 확인이 필요한 결정

고도화로 넘어가기 전, 디자인에서 임의로 확정하지 않고 확인이 필요한 항목입니다.

1. 어드민 셸(페이지 헤더 + 전역 액션바 + 하단 목록 자리표시) 추가 여부

현재는 입력 폼만 존재합니다. cargo-list는 이번 범위 제외이므로, 목록은 자리표시 영역만 두고 실제 테이블은 별도 작업으로 두는 것을 권장합니다.

A. 헤더+액션바만 추가B. 목록 자리표시까지C. 폼만 유지

2. 요약(4섹션) 긴 문장 처리 — 말줄임 vs 줄바꿈

문서와 HTML이 불일치 상태입니다(최신 규칙은 줄바꿈 보존 white-space:normal). 디자인 확정 전 1개로 통일 필요.

A. 줄바꿈 보존(권장)B. 1줄 말줄임 + 펼치기

3. 수수료(금액) — 인수증 결제 시 숨김 vs 비활성

수수료는 좌측 "금액" 영역의 금액이며 수수료율(%)과 다릅니다. 인수증일 때 셀을 완전히 숨길지, 비활성으로 남길지 결정 필요.

A. 숨김B. 비활성 표시

4. 상태 chip 상시 노출 여부 / 화물맨 중복·기존차주 판정 / 실제 API 계약

화물맨 Phase 2의 중복 후보·기존 차주 판정 규칙과 API 계약은 미확정입니다. Phase 1(내부 DB) 먼저 적용하고 Phase 2는 계약 확정 후 확장합니다.
