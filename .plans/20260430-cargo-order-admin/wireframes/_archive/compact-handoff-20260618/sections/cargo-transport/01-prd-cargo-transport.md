# PRD: 화물 운송정보 섹션

## 목적

`화물 운송정보` 섹션은 화물 오더의 차량/화물 조건과 정산 금액 조건을 빠르게 확인하고 수정하는 영역입니다.

이번 기준에서는 운송구간과 겹치는 정보를 줄입니다. `광폭`은 핵심 기준 필드에서 제외하고, `도착`은 운송구간의 상하차 일정/조건에 포함된 값으로 보아 화물 운송정보에서 제외합니다.

## 사용자 문제

| 문제 | 영향 |
| --- | --- |
| 운송구간과 화물 운송정보에 비슷한 조건이 반복됨 | 사용자가 같은 정보를 두 번 확인하는 느낌을 받음 |
| `청구비용`, `운송비용`, `수수료`, `합계`의 의미가 섞임 | 화주 청구 금액과 차주 정산 금액을 혼동할 수 있음 |
| 선착불 수수료 케이스가 일반 인수증 케이스와 같은 방식으로 보임 | 차주에게 수수료를 청구하는 특수 흐름을 놓칠 수 있음 |
| 기존 `합계` 명칭이 계산 의미를 충분히 설명하지 못함 | 조건별로 수익인지 차주에게 걸리는 운임인지 판단하기 어려움 |

## 사용자 목표

| 사용자 | 목표 |
| --- | --- |
| 접수/배차 운영자 | `톤수`, `차종`, `대수`, `실중량`과 `품목`을 `운송+품목` 2행에서 빠르게 확인 |
| 정산 담당자 | 화주에게 받을 `청구비용`과 차주에게 지급/청구할 `운송비용`, `수수료`를 구분 |
| 수정 담당자 | 결제방법이 바뀌어도 같은 위치에서 금액을 수정 |

## 기능 요구사항

| ID | 요구사항 | 상태 |
| --- | --- | --- |
| `REQ-CT-001` | 운송 조건은 `톤수`, `차종`, `대수`, `실중량`을 기준 필드로 둔다 | 확정 방향 |
| `REQ-CT-002` | `광폭`은 이번 섹션 기준 필드에서 제외한다 | 확정 방향 |
| `REQ-CT-003` | `도착`은 운송구간과 중복되므로 이번 섹션 기준 필드에서 제외한다 | 확정 방향 |
| `REQ-CT-004` | 금액 조건에는 `결제방법`을 포함하고 선택값은 `인수증`, `선불`, `착불`, `선착불`로 둔다 | 확정 방향 |
| `REQ-CT-005` | 금액 입력 필드는 `청구비용`, `운송비용`, `수수료`, 결과 금액 순서로 둔다 | 확정 방향 |
| `REQ-CT-006` | `품목`은 `운송+품목` 박스의 2번째 row로 둔다 | 확정 방향 |
| `REQ-CT-007` | `청구비용` 입력 시 `운송비용`도 같은 금액으로 자동 입력한다 | 확정 방향 |
| `REQ-CT-008` | 사용자가 `운송비용`을 수정하면 `청구비용`은 유지하고 `운송비용`만 바뀐다 | 확정 방향 |
| `REQ-CT-009` | `수수료` 입력 시 `결제방법`은 자동으로 `선착불`로 바뀐다 | 확정 방향 |
| `REQ-CT-010` | `수수료` 입력 시 `청구비용`은 `0원`으로 고정하고 포커스는 `운송비용`으로 이동한다 | 확정 방향 |
| `REQ-CT-011` | `인수증` 조건에서는 `수수료`를 숨기거나 비활성 상태로 처리한다 | 확정 방향 |
| `REQ-CT-012` | 기존 `합계` 명칭은 결제방법에 따라 `수익` 또는 `차주운임`으로 바꾼다 | 확정 방향 |
| `REQ-CT-013` | `인수증`과 `선착불` 상태는 같은 컴포넌트 위치를 유지한다 | 확정 방향 |
| `REQ-CT-014` | 운송구간의 거리/기준 금액은 `톤수`, `차종` 선택 이후 더 정확하게 출력된다는 상태를 표시한다 | 확정 방향 |
| `REQ-CT-015` | `실중량`은 톤수 선택 시 해당 톤수의 110%로 자동 입력하고, 사용자가 `0.00` 단위로 수정할 수 있게 한다 | 확정 방향 |
| `REQ-CT-016` | 값 없음 상태에서 `운송+품목 입력` 또는 `금액 조건 선택` 버튼을 누르면 버튼이 해당 라벨로 흡수되고 다이얼로그 적용 후 항상 노출 row로 전환된다 | 확정 방향 |

## 금액 조건 정의

| 필드 | 의미 | 표시/동작 |
| --- | --- | --- |
| `결제방법` | 정산 방식 선택값 | `인수증`, `선불`, `착불`, `선착불` 중 선택. 수수료 표시와 계산식을 제어 |
| `청구비용` | 화주에게 청구할 금액 | 금액 조건의 첫 번째 필드 |
| `운송비용` | 차주에게 지급하거나 차주 운임 계산의 기준이 되는 금액 | `청구비용` 입력 시 같은 금액으로 자동 입력 |
| `수수료` | 선불/착불/선착불 조건에서 차주에게 청구하는 주선 수수료 | 인수증에서는 숨김/비활성, 입력 시 자동 선착불 전환 |
| `수익/차주운임` | 기존 `합계` 대체 라벨 | `인수증`은 `수익`, 선불/착불/선착불은 `차주운임`으로 표시 |

## 결제방법별 계산

| 조건 | 표시 규칙 | 계산 |
| --- | --- | --- |
| `인수증` | `수수료` 숨김 또는 비활성 | `청구비용 - 운송비용 = 수익` |
| `선불/착불/선착불` | `수수료` 표시 | `운송비용 + 수수료 = 차주운임` |

인수증에서는 결과 라벨을 `차주운임`이 아니라 `수익`으로 표시합니다. 선착불에서는 화주에게 받는 금액이 아니라 차주에게 청구되는 운임/수수료 합산값이므로 `차주운임` 라벨을 유지합니다.

## 비기능 요구사항

| ID | 요구사항 | 설명 |
| --- | --- | --- |
| `NFR-CT-001` | 기존 레이아웃과 큰 차이를 만들지 않는다 | 사용자가 갑작스러운 화면 변화에 불편함을 느끼지 않도록 함 |
| `NFR-CT-002` | 결제방법이 바뀌어도 필드 위치를 바꾸지 않는다 | 같은 위치에서 상태와 값만 달라져야 함 |
| `NFR-CT-003` | 수수료 조건부 표시가 명확해야 한다 | 인수증과 선착불의 차이를 혼동하지 않게 함 |
| `NFR-CT-004` | 계산 상태를 과장하지 않는다 | 아직 확정되지 않은 금액이나 기준값은 `확인 필요` 상태로 표현 |
| `NFR-CT-005` | 빈 값 전환은 기존 placeholder 구조를 유지하며 부드럽게 상태만 바꾼다 | 사용자가 갑작스러운 화면 교체로 느끼지 않게 함 |
| `NFR-CT-006` | 입력 전 CTA와 적용 후 row의 밀도 차이를 줄인다 | 입력 전 row는 CTA 중심, 적용 후 row는 정보 중심으로 보이되 높이와 폭 리듬을 맞춤 |
| `NFR-CT-007` | 보조 설명은 화면 안에서 과하게 강조하지 않는다 | 입력 전 chip은 muted dashed 스타일, 적용 후 계산식은 보조 정보로 낮춰 실제 값이 먼저 보이게 함 |

## Acceptance Criteria

| ID | 기준 | 상태 |
| --- | --- | --- |
| `AC-CT-001` | 기준 운송 조건에서 `광폭`, `도착`이 제외되어 있다 | 검증 완료 |
| `AC-CT-002` | `품목`은 `운송+품목` 박스의 2번째 row로 문서화되어 있다 | 검증 완료 |
| `AC-CT-003` | 금액 조건에 `결제방법`이 포함되어 있고 선택값이 문서화되어 있다 | 검증 완료 |
| `AC-CT-004` | 금액 입력 필드의 첫 번째 필드는 `청구비용`이다 | 검증 완료 |
| `AC-CT-005` | `청구비용` 입력 후 `운송비용` 자동 입력 규칙이 문서화되어 있다 | 검증 완료 |
| `AC-CT-006` | `운송비용` 수정 시 `청구비용`을 유지하는 규칙이 문서화되어 있다 | 검증 완료 |
| `AC-CT-007` | `수수료` 입력 시 자동 `선착불`, `청구비용 0원`, `운송비용 포커스` 규칙이 문서화되어 있다 | 검증 완료 |
| `AC-CT-008` | 기존 `합계`는 인수증에서 `수익`, 선착불에서 `차주운임`으로 재정의되어 있다 | 검증 완료 |
| `AC-CT-009` | 인수증과 선착불에서 컴포넌트 위치가 동일하다는 기준이 문서화되어 있다 | 검증 완료 |
| `AC-CT-010` | `실중량`은 대수 오른쪽에 있고 톤수 110% 자동 입력 및 직접 수정 기준이 문서화되어 있다 | 검증 완료 |
| `AC-CT-011` | 값 없음 상태에서 `운송+품목 입력`, `금액 조건 선택` CTA 흡수 전환 기준이 문서화되어 있다 | 검증 완료 |
| `AC-CT-012` | B 통합본에서 입력 전 CTA row와 적용 후 정보 row의 높이/폭 리듬이 맞춰져 있다 | 검증 완료 |
| `AC-CT-013` | B 통합본에서 적용 후 다이얼로그가 정상적으로 닫히고 다시 열리지 않는다 | 검증 완료 |

## 제외 범위

- 실제 금액 계산 로직 구현
- 실제 저장 API 연동
- 상차/하차 일시와 방법 선택 UI 구현
- 기존 파일 삭제
