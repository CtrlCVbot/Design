# Source Comparison Review: 화물 운송정보

## 리뷰 목적

기존 화면의 정보를 빠뜨리지 않으면서 `화물 운송정보` 섹션을 더 읽기 쉬운 구조로 바꾸기 위한 비교 리뷰 문서입니다.

2026-05-28 기준으로 사용자의 정산 규칙, 실중량 입력, 품목 row 이동 기준을 반영했습니다. 이후 B 통합본 F안에서는 운송구간 D안 컨셉을 유지하되, 운송 조건은 `톤수`, `차종`, `대수`, `실중량`으로 정리하고, `품목`은 `운송+품목` 박스의 2번째 row로 배치합니다.

## 비교 기준

| 기준 | 확인 결과 |
| --- | --- |
| 정보 누락 | B 통합본 source field를 검토하되 target field는 새 기준으로 재정의 |
| 중복 제거 | `광폭`, `도착`은 기준 운송 조건에서 제외 |
| 그룹 적합성 | `운송+품목`, `금액` 2그룹 구성. `품목`은 `운송+품목`의 2번째 row |
| 계산 의존성 | `톤수`, `차종`은 운송구간 기준 금액과 연결, `실중량`은 톤수 110% 자동 입력 |
| 정산 명확성 | `결제방법`, `청구비용`, `운송비용`, `수수료`, `수익/차주운임`의 의미를 조건별로 분리 |

## Source별 차이

| Source | 강점 | 한계 | 다음 설계 반영 |
| --- | --- | --- | --- |
| B 통합본 | 현재 운영 화면의 밀도와 위치감이 가장 잘 남아 있음 | `광폭`, `도착`, `합계`가 새 기준과 맞지 않음 | 시각 톤만 유지, field는 재정의 |
| 최초 wireframe | 상하차/배차/차량 조건의 원래 범위가 넓게 남아 있음 | 현재 운송구간 D안과 중복되는 필드가 많음 | 이동/제외 근거로 사용 |
| 명세서형 wireframe | 값 없음 상태와 계산표가 명확함 | 새 금액 row와 직접 일치하지 않음 | empty state와 금액 계산 참고 |

## 현재 관찰 사항

| 항목 | 관찰 | 판단 |
| --- | --- | --- |
| `광폭` | B 통합본에 있으나 source 대응은 최초 `옵션` 일부에 가까움 | 이번 기준 필드에서 제외 |
| `도착` | B 통합본에 있으나 운송구간 상하차 일정/조건과 겹침 | 이번 기준 필드에서 제외 |
| `수수료` 위치 | B 통합본에는 항상 보이는 금액처럼 있음 | 조건부 표시로 변경 필요 |
| `결제방법` | B 통합본 `운송비구분`, 최초 wireframe `결제 조건`이 존재 | 금액 조건 selector로 복구 필요 |
| `합계` 명칭 | B 통합본의 `합계`는 의미가 모호함 | 인수증은 `수익`, 선착불은 `차주운임`으로 변경 |
| 금액 계산 | 기존 문서는 `운송료 + 수수료` 중심 | 인수증과 선착불 계산식을 분리해야 함 |
| 운송구간 계산 메타 | B 통합본에 `톤수·차종 변경 시 재계산` 표시 | `톤수`, `차종` 옆 상태 badge로 연결 가능 |

## 다음 리뷰 액션

1. `수수료` 입력 시 자동 선착불 전환, `청구비용 0원`, `운송비용 포커스`가 실제 구현 단계에서도 유지되는지 확인합니다.
2. 인수증 `수익`, 선착불 `차주운임` 라벨이 충분히 명확한지 검토합니다.
3. validation message와 `자동/수정됨` chip의 필요 여부를 후속 단계에서 결정합니다.
4. 차종 축약어와 일정 토큰의 운영 표기를 확정합니다.
