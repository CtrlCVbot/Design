# 예약 영역 탭 전체 기획

## 목표

현재 우측 `예약 영역`은 `여기에 섹션이 추가됩니다`라는 placeholder만 있는 상태입니다. 이 영역을 화면에서는 `보조 정보`로 노출하고, 운송건의 보조 판단 정보를 담는 3개 탭으로 확장합니다.

핵심 목표는 운영자가 좌측의 접수/수정 입력값을 보면서, 우측에서 메모, 금액, 지도 맥락을 빠르게 확인하게 만드는 것입니다.

2026-06-17 피드백 반영 후 `보조 정보`는 `업무 요약 우선` 원칙을 따릅니다. 기본 첫 화면은 `메모` 리스트를 보여주고, 금액 로그는 청구금/배차금/수익을 금액 명세로 보여줍니다. 운송 구간 지도는 위치와 계산 상태 중심으로 정리하고, 상세 확인이나 작성은 dialog 또는 펼침(expand)으로 분리합니다.

## 현재 화면 기준

| 항목 | 현재 상태 | 기획 방향 |
| --- | --- | --- |
| 위치 | `aside.split__right` 내부 placeholder | 같은 위치를 탭 패널로 승격 |
| 접근성 | 기존 placeholder는 `aria-hidden="true"`로 숨겨진 상태 | 실제 기능 후보에서는 `aria-label="보조 정보"`와 탭 semantics 적용 필요 |
| 레이아웃 | 좌측은 `3. 화물 운송정보`, `4. 화물정보 요약`, `5. 차주 정보`, 우측은 기존 예약 영역 | 좌측 입력 흐름은 유지, 우측은 `보조 정보` 확인 패널 |
| 화면 헤더 | 기존 `예약 영역`, `여기에 섹션이 추가됩니다` | 노출명: `보조 정보`, 탭: `메모`, `금액 로그`, `운송 구간 지도` |
| shadcn 디자인 기준 | `results/html/Design System.html` | 해당 파일의 shadcn 토큰, 버튼, chip, badge, section 구조 기반으로 확장 |

## shadcn 디자인 시스템 적용 원칙

예약 영역 탭은 `C:\Work\Dev\Design\.plans\wireframes\cargo-order-admin-20260430\results\html\Design System.html`의 shadcn 기준으로 설계합니다. 이 파일은 현재 프로젝트의 shadcn 기반 색상, 타이포그래피, 간격, 버튼, chip, badge, empty state, dialog 감각을 정리한 기준 파일입니다.

| 영역 | 적용 기준 |
| --- | --- |
| 탭 외곽 | 기존 `ds-section` 수준의 경계, radius, shadow 밀도를 유지합니다. |
| 탭 버튼 | 별도 브랜드 색을 만들지 않고 `btn` 또는 segmented control 감각을 사용합니다. |
| 상태 chip | 거리, 예상 시간, 경유 여부, 메모 수, 정산 상태는 `chip`과 `chip--metric` 패턴을 우선합니다. |
| 상태 badge | 접수/완료/취소처럼 강한 상태는 기존 `badge` 패턴을 사용합니다. |
| 빈 상태 | `empty` 패턴처럼 짧은 안내와 1개 주요 액션만 둡니다. |
| 오류/주의 | `notice`, `notice--danger`, `notice--info` 계열을 사용합니다. |
| 밀도 | 운영 화면이므로 넓은 hero나 마케팅형 설명 대신 좁고 읽기 쉬운 정보 구성을 유지합니다. |

## 사용자 가치

| 사용자 상황 | 필요한 정보 | 탭으로 해결하는 방식 |
| --- | --- | --- |
| 운송 경로가 적절한지 확인 | 상차지/하차지, 거리, 예상 시간, 경유 여부 | `운송 구간 지도`에서 위치와 계산 메타를 한 번에 확인 |
| 고객 요청이나 배차 주의사항 확인 | 운영자 메모, 고객 요청, 배차 특이사항 | `메모`에서 유형별 메모를 시간순으로 확인 |
| 금액 조정 내역 확인 | 청구금 또는 운송비(배차금)에 반영된 대기비, 경유비, 할인 등 조정금의 +/-, 담당자, 마지막 일시 | `금액 로그`에서 기준금액, 조정금, 합계, 수익을 먼저 확인하고 항목별 담당자/일시/사유는 필요할 때 펼침 |

## 운영자 주요 사용 시나리오

| 시나리오 | 흐름 | 성공 기준 |
| --- | --- | --- |
| 주소 입력 후 경로 확인 | 운영자가 좌측 `2. 운송 구간`에서 상차지/하차지를 적용한 뒤 우측 `운송 구간 지도`를 확인합니다. | 상차/하차 핀, 거리, 예상 시간, 경유 여부가 같은 오더 기준으로 표시됩니다. |
| 배차 전 특이사항 확인 | 운영자가 차주를 적용하기 전에 `메모` 탭에서 고객 요청과 배차 특이사항을 확인합니다. | 최신 메모와 유형 chip을 보고 현장 주의사항을 빠르게 파악합니다. |
| 고객 문의 대응 | 고객이 대기비나 경유비 등 조정금 이유를 문의하면 운영자가 `금액 로그` 탭에서 금액 명세와 명세 내부 상세 row를 확인합니다. | 항목별 +/- 금액, 담당자, 처리 시각, 사유를 기준으로 설명할 수 있습니다. |
| 추가 메모 작성 | 통화 중 확인한 요청사항을 `메모` 탭의 `추가` 버튼으로 dialog를 열어 저장합니다. | 기본 탭 화면은 리스트 중심으로 유지되고, 성공 시 새 메모가 리스트 상단에 추가됩니다. |
| 정산 전 확인 | 정산 처리 전에 `금액 로그`에서 청구금 합계, 배차금 합계, 수익, 정산 상태를 확인합니다. | 좌측 금액 표시와 금액 로그의 금액 명세 기준이 서로 충돌하지 않습니다. |

## 정보 구조

우측 `보조 정보` 패널은 아래 계층으로 구성합니다.

1. 섹션 헤더
2. 탭 목록
3. 탭별 상태 요약
4. 탭별 본문
5. 탭별 주요 액션

섹션 헤더의 화면 노출명은 `보조 정보`로 확정합니다. 이 이름은 메모, 금액 로그, 운송 구간 지도가 기존 입력 영역을 대체하지 않고 오더 판단을 돕는 보조 패널이라는 의미를 전달합니다.

권장 기본 탭은 `메모`입니다. 이유는 운영자가 배차 전후에 고객 요청, 배차 특이사항, 내부 주의사항을 가장 먼저 확인해야 하고, 메모가 없을 때도 바로 추가 액션을 제공할 수 있기 때문입니다.

## 탭 역할 분리

| 탭 | 담당하는 질문 | 담당하지 않는 것 |
| --- | --- | --- |
| `메모` | 이 오더에 대해 사람이 남긴 주의사항은 무엇인가? | 금액 변경 이력, 시스템 이벤트 전체 로그 |
| `금액 로그` | 청구금 또는 운송비(배차금)에 어떤 조정금이 붙거나 빠졌는가? 마지막 담당자와 일시는 언제인가? | 금액 직접 수정, 고객 메모, 전체 회계 감사 화면 |
| `운송 구간 지도` | 이 오더가 어디에서 어디로 이동하는가? 거리와 예상 시간은 어떤가? | 주소 직접 수정, 운송 조건 입력 |

이 분리를 유지해야 탭 간 정보가 겹치지 않고, 운영자가 탭 이름만 보고도 어디를 열어야 할지 알 수 있습니다.

## 기존 접수/수정 섹션과의 연결 방식

| 기존 섹션 | 연결되는 탭 | 연결 원칙 |
| --- | --- | --- |
| `2. 운송 구간` | `운송 구간 지도` | 상차지/하차지 주소와 계산 메타를 읽어 지도, 핀, 상태 chip으로 표시합니다. 하단에 상차/하차 주소를 다시 길게 반복하지 않고, 주소 수정은 기존 운송 구간 섹션에서 처리합니다. |
| `3. 화물 운송정보`의 금액 영역 | `금액 로그` | `금액 입력`, `금액 조건 적용`, `정산 처리` 결과 중 대기비, 경유비 등 조정금의 +/- 영향을 요약합니다. 금액 수정은 기존 다이얼로그에서 처리합니다. |
| `4. 화물정보 요약` | 전체 탭 | 탭 본문은 요약의 보조 근거 역할을 합니다. 요약 섹션의 대표값과 충돌하면 안 됩니다. |
| `5. 차주 정보` | `메모`, `운송 구간 지도` | 배차 특이사항 메모와 경유/상하차 주의사항을 보조 정보로 연결합니다. |
| 하단 actionbar | `금액 로그` | `화물 취소`, `배차 취소`, `정산 처리` 같은 위험 액션 결과는 금액 로그 또는 별도 시스템 로그 정책 확인 후 표시합니다. |

## 레이아웃 원칙

| 원칙 | 설명 |
| --- | --- |
| 운영 화면 우선 | 홍보형 설명이나 큰 hero는 두지 않고, 바로 스캔 가능한 정보 밀도를 유지합니다. |
| 기존 톤 유지 | `Design System.html`의 `ds-section`, `sec-head`, `chip`, `badge`, `notice` 계열을 재사용하는 방향으로 설계합니다. |
| 보조 패널 역할 | 우측 탭은 기존 입력을 대체하지 않고 확인, 기록, 맥락 제공에 집중합니다. |
| 요약 우선 | 첫 화면은 compact 요약과 리스트 중심으로 구성하고, 상세 확인이나 작성은 dialog 또는 펼침으로 분리합니다. |
| 상태 우선 표시 | 빈 상태, 로딩, 오류, 데이터 있음 상태가 탭마다 명확히 보여야 합니다. |
| 탭 간 독립성 | 지도, 메모, 금액 로그는 데이터 소스와 목적이 다르므로 서로의 상태에 영향을 주지 않습니다. |

## 탭 전환 규칙

| 규칙 | 설명 |
| --- | --- |
| 기본 탭 | 최초 진입 시 `메모`를 엽니다. |
| 상태 유지 | 같은 오더 안에서 사용자가 마지막으로 연 탭을 유지할 수 있습니다. 구현이 부담되면 MVP에서는 `메모` 기본 탭 고정으로 시작합니다. |
| 오더 변경 | 다른 오더를 열면 탭 본문 데이터는 새 오더 기준으로 다시 로딩합니다. |
| 메모 작성 dialog | 메모 작성은 탭 본문이 아니라 dialog에서 처리합니다. 작성 중에는 dialog를 닫기 전까지 탭 전환을 막거나 이탈 확인을 띄웁니다. MVP에서는 dialog 내부 입력값을 유지합니다. |
| 오류 격리 | 한 탭의 오류가 다른 탭 전환을 막지 않습니다. |

## MVP 범위

| 영역 | MVP 포함 |
| --- | --- |
| 탭 구조 | 3개 탭 표시, 선택 탭 본문 전환 |
| 지도 | 지도/핀 표시, 거리/예상 시간/경유 여부 요약, 주소 미입력 상태, 하단 주소 반복 표시 제외 |
| 메모 | 메모 리스트, 유형 표시, 메모 추가 dialog |
| 금액 로그 | 기준 청구금, 청구 조정금, 청구금 합계, 기준 배차금, 배차 조정금, 배차금 합계, 수익, 담당자, 마지막 처리 시각, 정산 상태 |
| 상태 | 빈 상태, 로딩 상태, 오류 상태, 데이터 있음 상태 |
| 연결 | 기존 운송 구간/금액 입력 결과를 읽어 탭 본문에 반영 |

## Acceptance Criteria

| ID | 기준 |
| --- | --- |
| `RAT-AC-01` | 우측 패널의 화면 노출명은 `보조 정보`이며, 내부 작업명 `예약 영역`은 `메모`, `금액 로그`, `운송 구간 지도` 3개 탭으로 정의되어야 합니다. |
| `RAT-AC-02` | 각 탭은 목적과 표시 정보가 문서상 분리되어야 하며, 같은 정보가 중복 주 역할을 맡지 않아야 합니다. |
| `RAT-AC-03` | MVP 범위와 후속 확장 범위가 탭별로 분리되어야 합니다. |
| `RAT-AC-04` | 기존 `2. 운송 구간`, `3. 화물 운송정보`, `4. 화물정보 요약`, `5. 차주 정보`의 입력/수정 역할을 침범하지 않아야 합니다. |
| `RAT-AC-05` | 빈 상태, 로딩 상태, 오류 상태, 데이터 있음 상태가 모든 탭에 정의되어야 합니다. |

| `RAT-AC-07` | `보조 정보` 첫 화면은 요약 중심이어야 하며, 메모 작성과 금액 상세는 dialog 또는 펼침으로 분리되어야 합니다. |

## 다음 단계 실행 계획

1. 데이터 있음 후보와 빈 상태 후보를 브라우저에서 최종 QA합니다.
2. 지도 provider, 예상 시간 산정 기준, 경유 데이터 출처를 정책으로 확정합니다.

4. 메모 작성/수정/삭제 권한과 공개 범위를 확정합니다.
5. 확정된 정책을 기준으로 데이터 모델과 API 응답 필드를 설계합니다.
