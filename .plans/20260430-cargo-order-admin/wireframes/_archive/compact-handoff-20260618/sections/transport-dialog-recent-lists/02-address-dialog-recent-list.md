# 상하차지 다이얼로그 최근 사용 리스트 기획

## 대상 흐름

기존 `운송 구간`의 주소 검색/주소 적용 흐름을 대상으로 합니다.

| 기준 | 현재 흐름 |
| --- | --- |
| 호출 | `openAddrSearch('load')`, `openAddrSearch('unload')` |
| 선택 | 조회 결과 row 클릭 후 `선택 미리보기` 갱신 |
| 확정 | `상차지 적용` 또는 `하차지 적용` |
| 반영 | `applyAddr()` 후 `운송 구간` row와 요약 갱신 |

최근 사용 리스트는 기존 검색 결과 리스트 영역의 초기 상태로 표시합니다. 최근 항목을 클릭해도 `applyAddr()`가 실행되지 않고, 미리보기만 갱신합니다.

## 최근 리스트 구분

| 구분 | 설명 | MVP 판단 |
| --- | --- | --- |
| `상차 최근` | 최근 상차지로 선택된 장소 | 상차지 다이얼로그 기본 탭 |
| `하차 최근` | 최근 하차지로 선택된 장소 | 하차지 다이얼로그 기본 탭 |
| `전체 최근` | 상차/하차 구분 없이 최근 선택된 장소 | 보조 탭 또는 fallback |

기본 제안은 현재 다이얼로그 역할에 맞는 탭을 먼저 여는 것입니다. 예를 들어 `openAddrSearch('load')`는 `상차 최근`, `openAddrSearch('unload')`는 `하차 최근`을 기본으로 둡니다.

## 화주별 최근과 사용자별 최근

| 기준 | 장점 | 주의 |
| --- | --- | --- |
| 화주별 최근 | 같은 화주가 반복 사용하는 창고/현장 후보 정확도가 높음 | 신규 화주 또는 데이터가 적은 화주에서는 비어 있을 수 있음 |
| 사용자별 최근 | 운영자가 방금 처리한 패턴을 이어서 사용하기 쉬움 | 다른 화주의 장소가 섞이면 오선택 위험이 있음 |

MVP 기본은 `화주별 최근`을 우선합니다. 화주별 최근이 없으면 `전체 최근` 또는 `사용자별 최근`을 보조로 보여줄 수 있습니다. 이때 다른 화주의 장소는 화주명 대신 `다른 화주에서 사용` 정도의 낮은 힌트만 검토하고, 업체명과 연락처는 과하게 노출하지 않습니다.

## 최근 장소 항목 표시 정보

| 정보 | 표시 위치 | 개인정보 기준 |
| --- | --- | --- |
| 장소명 | row 주요 값 | 표시 |
| 행정주소 | row 주요 값 | 표시 |
| 상세주소 | 보조 값 또는 미리보기 | row에서는 짧게, 미리보기에서 전체 |
| 사용 맥락 | `상차 최근`, `하차 최근`, `최근 2회`, `어제 사용` chip | 표시 |
| 담당자 | row에서는 이름 일부 또는 직책만 검토 | 미리보기에서 표시 |
| 연락처 | row에서는 원칙적으로 숨김 | 미리보기에서만 표시 검토 |
| 출처 | `화주 등록`, `Kakao`, `최근` badge/chip | 표시 |

## 다이얼로그 배치

```text
[상차지 주소 검색]

검색어 [도로명 · 지번 · 건물명 검색________] [조회]
[화주 등록 우선] [Kakao 포함] [중복 가능]

조회 결과 영역                    선택 미리보기
[초기: 최근 사용 장소]             장소명, 주소, 상세주소, 담당자, 연락처
○ 코덱트 후진입차 | 경기 여주시 산북면 후리 | 최근 3회
○ 산북 물류창고   | 경기 여주시 산북면 상품리 | 최근 1회

[검색 실행 후: 기존 검색 결과]
○ [화주 등록] 무갑리 현장 | 경기 광주시 초월읍 무갑리
○ [Kakao] 산북면 후리 18-1 | 경기 여주시 산북면 후리 18-1

[취소] [상차지 적용]
```

최근 리스트는 검색 영역 위에 별도 패널로 추가하지 않습니다. 다이얼로그 최초 진입 시 기존 검색 결과 영역에 `상차 최근` 또는 `하차 최근` 항목을 표시하고, 사용자가 검색어 입력 후 `조회` 또는 Enter로 검색을 실행하면 같은 영역을 기존 검색 결과 리스트로 전환합니다.

`선택 미리보기`는 현재 샘플의 내부 UI 톤을 유지할 수 있습니다. 다만 위치와 전체 레이아웃은 `cargo-order-admin-hifi-master.html`의 기존 주소 검색 다이얼로그 구조를 우선합니다. 기존 구조와 충돌하는 경우에는 미적용 사유를 `self-review.md`에 남깁니다.

## 선택 시 미리보기/입력값 반영

| 이벤트 | 처리 |
| --- | --- |
| 최근 장소 클릭 | 해당 row selected, `선택 미리보기`를 최근 장소 값으로 갱신 |
| `조회` 또는 Enter 검색 실행 | 최근 리스트를 숨기고 같은 영역에 기존 검색 결과 리스트 표시 |
| 검색 결과 클릭 | 최근 selected를 해제하거나 secondary 상태로 낮추고, `선택 미리보기`를 검색 결과 값으로 갱신 |
| 미리보기 수정 | 상세주소, 지명, 담당자, 연락처만 수정 가능 |
| 적용 클릭 | 현재 미리보기 값을 `상차지` 또는 `하차지` row에 반영 |
| 취소 클릭 | 기존 row 값 유지. 최근 클릭만으로 저장하지 않음 |

## 검색 결과 선택과 최근 항목 선택 우선순위

마지막으로 사용자가 명시 선택한 후보가 우선입니다.

| 순서 | 결과 |
| --- | --- |
| 최근 항목 선택 후 바로 적용 | 최근 항목 미리보기 값 적용 |
| 최근 항목 선택 후 검색 결과 선택 | 검색 결과 값 적용 |
| 검색 결과 선택 후 최근 항목 선택 | 최근 항목 값 적용 |
| 최근 항목 선택 후 검색어만 변경 | 미리보기는 유지 |
| 최근 항목 선택 후 검색 실행 | 리스트 영역은 검색 결과로 전환되지만, 검색 결과를 선택하기 전까지 미리보기는 유지 |

## Empty / Loading / Error

| 상태 | 표시 | 동작 |
| --- | --- | --- |
| empty | 기존 검색 결과 영역에 `최근 사용 장소가 없습니다.` empty 표시 | 검색 영역은 그대로 사용 가능 |
| loading | skeleton row 2~3개 또는 `최근 사용을 불러오는 중` | 검색 입력은 가능, 최근 리스트만 대기 |
| error | `최근 사용을 불러오지 못했습니다.` + `다시 시도` | 검색 결과 흐름은 차단하지 않음 |

최근 리스트 오류는 주소 검색 자체의 오류가 아닙니다. 따라서 최근 리스트 오류가 있어도 검색 결과 선택과 적용은 계속 가능해야 합니다.

## 데이터 정책 초안

| 항목 | 기준 |
| --- | --- |
| key | `shipperId + side(load/unload) + normalizedAddress` |
| dedupe | 같은 `normalizedAddress`와 같은 `side`는 최신 사용 1개 |
| 전체 최근 | `side` 구분 없이 `shipperId + normalizedAddress` 기준 dedupe |
| 정렬 | `lastSelectedAt desc` |
| limit | 10개 |
| 저장 시점 | 사용자가 `적용` 버튼으로 실제 상차/하차 row에 반영한 뒤 기록 |

## 구현 인계 메모

| 항목 | 메모 |
| --- | --- |
| 기존 함수 보호 | `openAddrSearch()`의 검색 결과와 `applyAddr()` 확정 책임을 유지 |
| 접근성 | 최근 row는 button 역할로 두고 selected 상태를 `aria-pressed` 또는 `aria-selected`로 표현 |
| 개인정보 | row에는 연락처를 기본 노출하지 않음 |
| wizard | wizard 단계에서는 `다음: 하차지`, `다음: 운송+품목` 버튼이 적용 역할을 수행할 수 있으므로 문구만 단계에 맞춤 |
