# 신규 접수 플로우 개요

## 목표

`신규 접수`는 기존 화물 수정과 다른 진입점이다.

사용자가 새 화물을 등록할 때는 빈 상태에서 시작하고, 필수 입력을 순서대로 완료하도록 wizard가 안내한다. 반대로 기존 화물을 수정하는 경우에는 현재 화면처럼 각 섹션의 독립 다이얼로그를 열어 필요한 항목만 수정한다.

섹션 헤더 정책의 의도는 보기 모드를 분리하는 것이다. 화물을 선택해 수정하거나 화물 자체를 조회할 때는 각 항목의 라벨이 이미 본문에 있으므로, 번호가 붙은 섹션 헤더를 숨겨 명세서처럼 한 번만 읽히는 화면으로 만든다. 신규 접수에서는 사용자가 아직 입력 순서를 따라가야 하므로 번호가 붙은 안내형 헤더를 표시한다.

다만 `오더 요약`은 입력 단계가 아니라 확인용 섹션이다. 신규 접수 안내형 보기에서도 단계 번호를 부여하지 않고, 기존 섹션 디자인을 유지한 상태에서 회색 `확인` 배지로 “검토 영역”임을 표시한다.

## 분기 기준

| 진입 조건 | 상태 | UI 정책 | 다이얼로그 정책 |
| --- | --- | --- | --- |
| 기본 조회 또는 기존 화물 수정 | `idle-edit` | 명세서형 보기. 섹션 헤더 숨김 | 독립 다이얼로그 |
| `신규 접수` 클릭 직후 | `new-reset` | 안내형 보기. 번호 헤더 표시 | 아직 다이얼로그 열기 전 |
| 신규 접수 중 최초 입력 시작 | `new-wizard-active` | 안내형 보기. 번호 헤더 표시 | wizard 다이얼로그 |
| 금액까지 필수 입력 완료 | `new-required-complete` | 안내형 보기. 번호 헤더 표시 | 금액 다이얼로그에서 분기 제공 |
| 차주 정보 입력 선택 | `new-driver-optional` | 안내형 보기. 번호 헤더 표시 | wizard의 선택 단계 |
| 메인 화면 적용 완료 | `new-submitted` | 명세서형 보기. 섹션 헤더 숨김, 메인 `화물 등록` 버튼 표시 | 이후 독립 다이얼로그 |

## 전체 흐름

1. 사용자가 `신규 접수`를 클릭한다.
2. 화면은 기존 입력값과 선택 상태를 모두 초기화한다.
3. 상태는 `new-reset`으로 바뀐다.
4. 신규 접수 전용 섹션 헤더가 표시된다.
5. `화주 정보` 섹션 또는 `화주 정보 입력` 버튼으로 focus가 이동한다.
6. 사용자가 최초 입력을 시작하면 wizard 다이얼로그가 열린다.
7. wizard는 `화주 정보 -> 상차지 -> 하차지 -> 운송+품목 -> 금액` 순서로 필수 입력을 진행한다.
8. 금액 단계 완료 후 `화물 등록 완료`와 `차주 정보로 이동` 중 하나를 선택한다.
9. `화물 등록 완료`를 선택하면 API 통신 없이 wizard 전체 데이터를 메인 화면에 적용하고 신규 접수 UI를 종료한다.
10. `차주 정보로 이동`을 선택하면 선택 단계인 차주 정보 입력으로 이어진다.
11. 데이터가 메인 화면에 적용되면 실제 API 통신 전 최종 CTA인 `화물 등록` 버튼을 표시한다.
12. 차주 정보 입력을 완료하거나 건너뛰면 같은 방식으로 메인 화면에 데이터를 적용한다.
13. 이후 개별 수정은 기존처럼 독립 다이얼로그로 처리한다.
14. 메인 `화물 등록` 버튼을 누르면 최종 validation 후 API 통신을 시작한다.
15. API 성공 후 실제 등록 완료 상태로 전환한다.

## 오더 요약 위치와 표현

`오더 요약`은 입력 단계 목록에 포함하지 않는다.

위치는 `정산 정보` 다음, `차주 정보` 이전을 기본값으로 한다. 이 위치는 금액까지 입력한 뒤 사용자가 최종 적용 전에 전체 입력값을 확인하기 좋고, 선택 항목인 차주 정보와도 자연스럽게 분리된다.

| 항목 | 정책 |
| --- | --- |
| 역할 | 입력값 확인과 정리 |
| 위치 | `정산 정보` 다음, `차주 정보` 이전 |
| 신규 접수 중 헤더 | 기존 섹션 헤더 구조 유지, 번호 대신 회색 `확인` 배지 |
| 기본/수정/등록 완료 후 | 다른 섹션과 동일하게 헤더 숨김 |
| 금지 | 별도 카드나 커스텀 요약 블록처럼 과도하게 분리하지 않음 |

## wizard 단계

| 순서 | 단계 | 필수 여부 | 완료 조건 |
| --- | --- | --- | --- |
| 1 | 화주 정보 | 필수 | 화주 선택 또는 필수 화주 정보 입력 완료 |
| 2-1 | 상차지 | 필수 | 상차지 주소와 필수 연락/일시 정보 입력 완료 |
| 2-2 | 하차지 | 필수 | 하차지 주소와 필수 연락/일시 정보 입력 완료 |
| 3 | 운송+품목 | 필수 | 톤수, 차종, 수량, 실중량 등 필수 운송 정보 입력 완료 |
| 4 | 금액 | 필수 | 결제방법, 청구비용, 운송비용 등 필수 금액 입력 완료 |
| 5 | 차주 정보 선택 | 선택 | 차주 선택 완료 또는 건너뛰기 선택 |

## 금액 후 분기 정책

금액 단계는 필수 입력의 마지막 단계다. 따라서 `적용`만 제공하면 사용자가 다음 행동을 놓칠 수 있다.

금액 다이얼로그의 마지막 화면 또는 footer에는 아래 두 선택지를 제공한다.

| 선택지 | 결과 |
| --- | --- |
| `화물 등록 완료` | API 통신 없이 전체 데이터를 메인 화면에 적용하고 신규 접수 UI 종료. 상태는 `new-submitted` |
| `차주 정보로 이동` | 같은 wizard 안에서 차주 정보 선택 단계로 전환. 상태는 `new-driver-optional` |

## 메인 화면 적용과 최종 등록 CTA

`화물 등록 완료`는 서버 저장 완료를 뜻하지 않는다.

이 선택지는 wizard에서 입력한 전체 데이터를 메인 화면에 반영하는 apply 동작이다. 적용이 끝나면 화면은 `new-submitted`가 되고, 사용자는 메인 화면에서 데이터를 최종 확인한 뒤 `화물 등록` 버튼으로 실제 API 통신을 시작한다.

| 항목 | 정책 |
| --- | --- |
| wizard 버튼 | `화물 등록 완료` |
| 실제 의미 | 전체 데이터를 메인 화면에 적용 |
| API 통신 | 발생하지 않음 |
| 적용 후 표시 | 메인 화면의 `화물 등록` 버튼 |
| 사용자 목적 | 최종 확인 후 실제 등록 요청 |

## 메인 `화물 등록` API 정책

메인 `화물 등록` 버튼은 실제 API 통신을 시작하는 유일한 버튼이다.

이 버튼을 클릭하면 먼저 최신 메인 화면 데이터 기준으로 최종 validation을 실행한다. validation을 통과하면 중복 등록 방지용 `idempotencyKey`를 만들고 API 요청을 보낸다. 실패하면 데이터는 유지하고 재시도할 수 있어야 한다.

| 상태 | 의미 |
| --- | --- |
| `submit-validating` | 최종 validation 실행 중 |
| `submit-pending` | API 통신 중 |
| `submit-failed` | API 실패 또는 서버 validation 실패 |
| `submit-complete` | API 성공 후 실제 등록 완료 |

## 화물 수정과의 충돌 방지

화물 수정은 `idle-edit` 상태에서만 처리한다.

기존 화물 수정, 메인 화면 적용 후 개별 수정, 조회 상태에서의 필드 수정은 모두 기존 독립 다이얼로그를 사용한다. 이때 wizard, 왼쪽 프로세스 패널, 신규 접수 섹션 헤더는 표시하지 않는다.

이 모드는 사용자가 이미 만들어진 화물을 확인하거나 고치는 상황이므로, 화면은 입력 wizard가 아니라 명세서에 가깝게 보여야 한다. 섹션 제목과 본문 라벨이 이중으로 반복되어 보이지 않도록 번호형 섹션 헤더를 숨기고, 본문 라벨과 값이 바로 읽히는 구조를 우선한다.

## 기본 화면 필수 입력 보조

명세서형 보기에서는 섹션 헤더가 숨겨지므로, 필수 입력이 비어 있을 때는 본문 행 자체가 입력 필요 상태를 알려야 한다.

| 대상 | 정책 |
| --- | --- |
| 빈 필수 행 | 좌측 2px 상태 바 표시 |
| 입력 버튼 | 최초 표시 시 soft glow pulse 최대 2회 |
| 장기 미입력 | 10초 간격으로 1회 reminder pulse만 다시 실행 |
| 중단 조건 | 버튼 클릭, 값 입력 완료, 행 제거, wizard 진입 |
| 접근성 | `prefers-reduced-motion`에서는 animation 생략, 정적 상태 표시 유지 |

이 보조 장치는 `idle-edit`과 `new-submitted` 같은 명세서형 보기에서만 사용한다. 신규 접수 안내형 보기에서는 섹션 헤더, focus, wizard가 이미 입력 순서를 안내하므로 중복 적용하지 않는다.

## MVP 범위

- 신규 접수 상태 모델 정의
- wizard 단계 순서와 완료 조건 정의
- 섹션 헤더 표시 조건 정의
- `오더 요약`은 확인용 섹션으로 분리하고 회색 `확인` 배지 정책 정의
- 금액 후 분기 정책 정의
- 차주 정보 선택 단계의 선택 사항 처리
- 기본 화면 필수 입력 보조 UI와 reminder motion 정의
- 데이터 메인 화면 적용 후 `화물 등록` CTA 표시
- 메인 `화물 등록` 클릭 후 최종 validation, loading, 실패, 재시도, 중복 방지 정책
- 메인 화면 적용 후 일반 수정/조회 상태 복귀

## 후속 확장 범위

- 단계별 임시 저장
- 중간 이탈 후 이어서 작성
- 단계별 validation error copy
- 실제 API endpoint와 서버 validation code 매핑
- 차주 추천/화물맨 연동을 wizard 안에서 어느 수준까지 노출할지 결정
- 사용자 권한별 `신규 접수` 버튼 노출 정책
