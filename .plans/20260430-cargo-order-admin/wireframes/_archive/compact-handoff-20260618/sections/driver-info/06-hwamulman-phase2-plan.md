# Phase 2 Plan - 화물맨 연동 차주 배차

## 1. 목적

`5. 차주 정보` 섹션에 외부 서비스 `화물맨` 연동 배차 흐름을 추가합니다.

Phase 1은 내부 DB 차주를 조회하고 선택하는 구조입니다. Phase 2는 실제 배치 row 안에서 `화물맨 연동`을 시작하고, `화물맨 화물 연동중`/`화물맨 화물 배차완료` 상태를 같은 row에서 확인한 뒤, 기존 `차주/차량 통합 조회 다이얼로그`로 배차 차주를 적용하는 구조입니다.

## 2. Phase 구분

| 구분 | 범위 | 설명 |
| --- | --- | --- |
| Phase 1 | 내부 DB 차주 배차 | 등록된 차주 조회, 선택, 적용, 차주 등록, 연락처 input, 톤수/차종 select 변경 |
| Phase 2 | 화물맨 연동 배차 | 화물맨 연동 CTA, 연동중 상태, 외부 배차 결과 수신, 화물맨 차주 적용, 검색 리스트 상단 고정, 수동 DB 저장 |

## 3. 확정 기준

| 항목 | 결정 |
| --- | --- |
| Phase 1 입력 전 상태 | `차주` 라벨 옆 `화물맨 연동`, row 우측 `차주/차량 입력`을 공존 |
| 진입 UI 추천 | `차주 [화물맨 연동] ... [차주/차량 입력]` 실제 배치 row 구조 |
| 연동 확인 | `화물맨 연동` 클릭 시 `화물맨에 화물을 공유하시겠습니까?` 확인 후 `YES`에서만 연동 시작 |
| 연동 취소 확인 | `연동 취소` 클릭 시 `화물맨 연동을 취소하시겠습니까?` 확인 후 `YES`에서 화물맨 API 취소 요청 |
| 연동중 표시 | row 중앙에 `화물맨 화물 연동중`, 우측에 `연동 취소`와 `차주/차량 입력` 표시 |
| 배차완료 표시 | row 중앙에 `화물맨 화물 배차완료`, 우측에 `연동 취소`와 `화물맨 차주 적용` 표시 |
| 적용 후 row | 기존 label/value 배치 유지, `출처` 컬럼에 `화물맨` 표시 |
| 연동중 다른 차주 적용 | 화물맨 연동은 살아 있고 내부 DB 차주가 적용된 경우 `차주/차량 변경` 옆 `연동 취소`를 유지 |
| 화물맨 배차 행 노출 | 해당 화물이 화물맨에서 배차된 차주 정보가 있으면 검색어와 무관하게 조회 리스트 상단에 항상 고정 |
| 다이얼로그 레이아웃 | 기존 `차주/차량 통합 조회 다이얼로그` 구조를 유지하고 화물맨 요소만 추가 |
| DB 저장 방식 | 자동 저장하지 않고 사용자가 수동으로 저장 |
| 신규 차주 배차 규칙 | 내부 DB 미등록 차주는 `차주 등록` 완료 후에만 배차 적용 가능 |
| 일반 차주 등록 버튼 | 검색 전에는 결과 영역 중앙 CTA, 검색/조회 후에는 결과 리스트 우측 상단 플로팅 CTA로 제공 |
| 화물맨 신규 저장 버튼 | 화물맨 신규 차주 행 선택 시 같은 등록 폼을 사용하되, 등록 폼 하단 CTA에 `차주 등록` primary 버튼으로 노출 |
| 기존 HTML | `driver-info-section-wireframe.html` 유지 |
| 비교 시안 | `driver-info-hwamulman-phase2-wireframe.html` 별도 추가 |
| 일반 차주와 구분 | 다이얼로그 안에서는 `화물맨 배차`, `저장 필요`, `DB 저장됨` 배지로 구분하고, 적용 후 row에서는 `출처` 컬럼으로 구분 |

## 4. Phase 2 사용자 가치

| 사용자 가치 | 설명 |
| --- | --- |
| 외부 배차 상태 확인 | 운영자가 차주 정보 섹션 안에서 화물맨 연동/배차 상태를 바로 확인 |
| 외부 배차 결과 재사용 | 화물맨에서 받은 차주 정보를 내부 DB에 저장해 다음 배차에 재사용 |
| 검색 흐름 유지 | 기존 내부 DB 차주 검색 흐름을 유지하면서 외부 배차 결과만 상단 고정 |
| 운영 통제 | 자동 저장 대신 수동 저장으로 중복/오등록 리스크 감소 |

| 화물맨 취소 요청중 | `화물맨 API 취소 요청중` 상태 문구 표시 | API 응답 대기, 성공 시 입력 전 상태 복귀 |
| 화물맨 취소 실패 | `화물맨 취소 실패` 알림 표시 | `연동 취소` 버튼 유지, 확인 후 API 취소 재시도 |
| 내부 DB 차주 적용 + 화물맨 연동중 | 차주 row는 내부 DB 차주 값으로 표시 | `차주/차량 변경` 옆 `연동 취소` 유지, 취소 성공 시 차주 row 유지 |
| 기존 등록 차주 | 화물맨 배차 행에 `기존 등록` 표시 | 선택 미리보기 후 바로 적용 |
| 신규 차주 | 화물맨 배차 행에 `신규 차주`, `등록 후 배차 가능` 표시 | `차주 등록` 완료 후 적용 |
| 저장 필요 | 화물맨 배차 행에 `저장 필요` 표시 | 신규 차주 등록 또는 DB 저장 |
| DB 저장됨 | 화물맨 배차 행에 `DB 저장됨` 표시 | 일반 차주처럼 선택/적용 |
| 배차 실패 | 실패 상태와 재연동 액션 표시 | 재시도 |
| DB 저장 실패 | 저장 실패 배지와 재시도 액션 표시 | 저장 재시도 |
| 중복 발견 | 기존 차주/차량과 연결 안내 | 기존 차주 연결 또는 신규 저장 보류 |

## 6. 조회 리스트 정책

화물맨 배차 차주 정보는 일반 검색 결과와 별도 우선 영역으로 표시합니다.

| 항목 | 정책 |
| --- | --- |
| 위치 | 조회 리스트 최상단 |
| 검색어 영향 | 검색어와 무관하게 항상 표시 |
| 개수 | 해당 화물 기준 최신 화물맨 배차 차주 1건 우선 |
| 행 타입 | `external-dispatch-row` 개념으로 관리 |
| 선택 가능 여부 | 기존 등록 차주는 바로 적용 가능, 신규 차주는 등록 완료 후 적용 가능 |
| 저장 후 | 일반 DB 차주와 동일하게 재사용 가능 |
| 검색 결과 없음 | 일반 결과가 없어도 화물맨 배차 행은 유지 |

## 7. 수동 DB 저장 정책

자동 저장은 하지 않습니다. 운영자가 화물맨 배차 차주 정보를 확인한 뒤 수동으로 내부 DB에 저장합니다.

| 단계 | 처리 |
| --- | --- |
| 배차 완료 수신 | 차주명, 차량번호, 연락처, 톤수, 차종, 화물맨 배차 식별값 표시 |
| 저장 전 | `저장 필요` 배지 표시 |
| 저장 액션 | 신규 차주는 `차주 등록`, 기존 등록 차주는 선택 미리보기 후 적용 |
| 중복 확인 | 차량번호, 연락처 기준으로 내부 DB 중복 후보 확인 |
| 저장 완료 | `DB 저장됨` 배지 표시 |
| 재사용 | 이후 내부 DB 차주 조회 결과로 검색 가능 |

## 9. 다이얼로그 레이아웃 유지 기준

화물맨 Phase 2는 기존 `차주/차량 통합 조회 다이얼로그`의 큰 레이아웃을 변경하지 않습니다.

| 구분 | 기준 |
| --- | --- |
| 유지 | 검색 영역, 조회 전 안내, 조회 리스트, 선택 미리보기, 차주 등록 폼, 하단 CTA 위치 |
| 추가 | 조회 리스트 최상단 `화물맨 배차 결과` 행, 상태 배지, 신규 차주 적용 제한 |
| 금지 | 새 다이얼로그 신설, 검색 input 위치 변경, 리스트 열 구조 재설계, 등록 폼 재배치 |
| 상세 기준 | `09-hwamulman-coexistence-ux-plan.md`의 `다이얼로그 레이아웃 유지 원칙` 참조 |
