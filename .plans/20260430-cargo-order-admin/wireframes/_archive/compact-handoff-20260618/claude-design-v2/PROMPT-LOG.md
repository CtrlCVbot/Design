## Step 1 — prompt-01-organize.md

- **목적**: 정리 + 디자인 시스템 기반(토큰·6컴포넌트)
- **첨부 자료**: 기준 구현물 `Cargo Order B Implementation Clean.html`(필수), 원본 화면 캡처 `source-full.png`, (선택) 01-section-digest.md·02-design-context.md 본문 첨부
- **산출물**: Design System.html + ds.css + Cargo Order Clean.html

### 붙여넣기 프롬프트 (전문)

```md
첨부한 "Cargo Order B Implementation Clean.html"은 화물 오더 접수/수정 관리 화면
(SCR-CARGO-ORDER-001)의 현재 구현물입니다. 5개 섹션(화주 정보 → 운송 구간 →
화물 운송정보 → 화물정보 요약 → 차주 정보)을 세로로 쌓은 고밀도 업무용 입력 화면입니다.

지금은 "디자인 고도화"가 아니라 그 직전 단계인 "정리 + 디자인 시스템 기반 세우기"를 해주세요.
구조와 필드는 그대로 두고, 흩어진 시각 언어를 하나로 정돈하는 것이 목표입니다.

[이 화면의 성격]
- 운영자가 하루 종일 빠르게 입력/수정/배차/정산하는 고밀도 데스크톱 업무 화면.
- SaaS 랜딩페이지, 큰 히어로, 카드형 마케팅 대시보드가 아닙니다.
- 와이어프레임에 남아있던 "종이/손글씨 스케치" 느낌은 저충실도 신호였을 뿐이므로,
  차분하고 업무적인 톤으로 정리해주세요. (손글씨 폰트는 사용하지 않습니다)

[현재 구현물의 정리해야 할 부채]
1. 디자인 토큰이 두 체계로 섞여 있습니다: accent/ink/line/muted 계열과
   pen/paper/note/title/rule 계열. → 의미 기반 단일 토큰 세트로 통합해주세요.
2. 토큰 밖 하드코딩 색(hex 약 60종)과 비표준 폰트 크기(11~30px 산발)가 많습니다.
   → 색 팔레트 정규화 + 모듈러 타입 스케일 + 4/8 기반 스페이싱 스케일로 토큰화.
3. 같은 역할의 버튼/칩/카드가 섹션마다 다른 클래스로 중복 정의돼 있습니다.
   → 공통 컴포넌트로 추상화.
4. 가시적 섹션 제목/번호가 없고 aria-label만 있습니다. → 섹션 헤더 체계를 정의.
5. 로드만 하고 쓰지 않는 손글씨 폰트(Gaegu/Caveat/Nanum)는 제거 전제로 정리.

[만들어 줄 디자인 시스템 기반]
1) 색상 토큰 (의미 기반):
   - 브랜드/주요 액션: 파란색 계열(현재 #1f64d6 계열 유지)
   - 텍스트: 기본/보조/비활성 3단계
   - 표면/배경/경계선
   - 상태색: 접수=파랑, 완료=중립 회색·검정, 취소·위험=빨강, 선택=강조 배경,
     경고=경고색(아이콘+문구 병행), 성공/정보
2) 타이포그래피: 업무 화면에 맞는 명확한 산세리프. 모듈러 타입 스케일(제목/섹션/본문/보조/캡션).
3) 스페이싱: 4/8 기반 스케일. 고밀도지만 섹션 경계가 읽히도록.
4) 공통 컴포넌트(변형 포함):
   - Button: primary / secondary / ghost / danger(위험 액션 전용)
   - Field: text / select / segment / money / date·time / inline-editable(읽기↔편집 토글)
   - SummaryRow: 라벨 컬럼 + 값 + 액션 버튼 (5개 섹션 공통, 약 32px 행 리듬)
   - Chip/Badge: status(접수·완료·취소) / metric(거리·금액) / source(화물맨·내부DB·Kakao) / role(배차·정산·관리)
   - Dialog: lookup(검색→결과→미리보기→적용) / confirm(YES·NO) / input
   - EmptyState: placeholder + CTA(빈 상태→입력 흡수 전환)
5) 섹션 헤더 체계: 번호 + 타이틀 + 얇은 구분선으로 5섹션을 스캔 가능하게.

[반드시 지킬 것 — 깨면 안 됨]
- 5개 섹션의 순서와 모든 입력 필드를 그대로 보존(화물 목록 cargo-list는 이 화면에 없음, 추가하지 마세요).
- 금액 의미: 청구비용 / 운송비용 / 수수료(금액) / 수익·차주운임(자동계산)을 구분.
  수수료는 좌측 "금액" 영역의 금액이며, 비율(%) 수수료율과 다릅니다.
- 직접 편집 불가 필드(업체명·사업자번호·담당자명·차주명·차량번호·행정주소)에는
  편집 가능한 것처럼 보이는 어포던스를 주지 마세요.
- 상태를 색상만으로 구분하지 말고 텍스트 배지를 함께 쓰세요.
- 위험 액션(화물취소·배차취소·연동취소·정산처리)은 일반 액션과 시각적으로 분리.

[이번 단계 산출물]
1. 디자인 토큰 정의(색/타이포/스페이싱/라운드/섀도) — 한 곳에 정리된 토큰표 또는 스타일 가이드.
2. 공통 컴포넌트 라이브러리(위 6종)와 각 변형 미리보기.
3. 통합 토큰/컴포넌트를 적용한 화면의 "정리본"(레이아웃·필드는 현재와 동일, 시각 언어만 통일).
4. 다음 단계(고도화) 전 확인이 필요한 결정 사항 목록(예: 어드민 헤더/전역 액션바/하단 목록 자리표시 추가 여부).

아직 화려하게 꾸미지 말고, "일관된 기반"을 먼저 완성하는 데 집중해주세요.
```

## Step 2 — prompt-03-restore-and-dialogs.md

- **목적**: 기획 레이아웃(세로 1열) 복원 + 다이얼로그 작동
- **첨부 자료**: 디자인 시스템 `Design System.html` + `design-system/ds.css`(필수·계승), 교정 대상 `Cargo Order Clean.html`, 원본 화면 캡처 `source-full.png`, (권장) 01-section-digest.md·02-design-context.md(SSOT)
- **산출물**: Cargo Order Planned.html

### 붙여넣기 프롬프트 (전문)

```md
첨부한 "Design System.html"(+ design-system/ds.css)와 "Cargo Order Clean.html"은
화물 오더 접수/수정 관리 화면(SCR-CARGO-ORDER-001)의 직전 결과물입니다.

디자인 시스템(토큰 + 6개 공통 컴포넌트)과 ds.css는 아주 좋으니 그대로 계승하세요.
다만 "Cargo Order Clean.html"이 원래 기획된 레이아웃을 임의로 바꿨고, 다이얼로그가
실제로 열리지 않습니다. 이 두 가지를 교정한 새 화면을 만들어주세요.

[실행 전제 — 새로 시작]
- 이전 작업/캔버스를 이어받지 말고 "새 화면"으로 처음부터 만드세요.
- 지금 캔버스에 어떤 레이아웃이 남아 있더라도 그것을 기준으로 삼지 마세요.
- 유일한 기준(SSOT)은 첨부한 자료입니다: 레이아웃은 아래 [교정 1]과 section-digest,
  비주얼은 design-system/ds.css. 직전 Clean.html의 2열 구도는 "고쳐야 할 변형"일 뿐 기준이 아닙니다.

[중요 — 출력 방식]
- "Cargo Order Clean.html"은 비교용으로 남겨야 하니 삭제하거나 덮어쓰지 마세요.
- 교정본은 "Cargo Order Planned.html"이라는 새 파일로 추가해주세요.
- 같은 design-system/ds.css 를 그대로 링크해서 사용하세요. 새 디자인 언어/토큰을 만들지 마세요.

[교정 1 — 원래 기획 레이아웃으로 복원]
현재 Clean.html은 화면을 좌우 2열로 쪼개고 섹션을 흩어 놨습니다(좌: 1·3·4 / 우: 2·5).
이건 기획에서 벗어난 변형입니다. 다음 "세로 1열 스택"으로 되돌리세요:
- 전체는 max-width 1500px 단일 컬럼. 페이지 헤더 → (선택)지표 subbar → 섹션 스택 → 액션바 → 목록 자리표시.
- 섹션은 위에서 아래로 이 순서 그대로:
  1. 화주 정보
  2. 운송 구간   ← full-width, 상차/하차 "세로 2행"으로 (좁은 컬럼에 넣지 말 것)
  3. 화물 운송정보 ← 섹션 안에서만 "운송+품목 / 금액" 2블록 좌우 배치(F안). 이건 유지.
  4. 화물정보 요약
  5. 차주 정보   ← 내부 DB / 화물맨 연동 탭
- 페이지 자체를 좌우 2열로 나누지 말고, 섹션을 다른 컬럼으로 옮기거나 순서를 바꾸지 마세요.
- 각 섹션은 ds.css의 섹션 헤더(번호 + 타이틀 + 구분선)를 그대로 사용.

[교정 2 — 다이얼로그를 실제로 열리고 닫히게]
ds.css의 dialog / lookup / confirm 컴포넌트를 재사용해서, 아래 트리거 버튼을 누르면
실제 모달이 열리고 닫히도록 JS로 연결하세요(정적 표시만 하지 말 것):
- 화주 "변경" 또는 빈 상태 "화주 정보 입력" → 화주 통합 조회 다이얼로그
  (검색 input + 기준 필터 + 결과 리스트 + 행 선택 → 우측 미리보기 + 담당자 추가 + "화주 정보에 적용")
- 차주 "변경" → 차주 통합 조회 다이얼로그
  (검색 전 중앙 CTA / 조회 후 결과 + 우상단 플로팅 "차주 등록" / 미리보기 / "차주 정보에 적용")
- 차주 "화물맨 연동" → confirm("화물맨에 화물을 공유하시겠습니까?" 예/아니오)
  → 예 누르면 "연동중" → "배차완료" 상태로 바뀌고, "연동 취소"는 다시 confirm.
- 운송 구간 상차/하차 "펼치기" → 상세주소·담당자·연락처·특이사항 inline 편집 영역이 펼쳐짐.
  주소 자체 변경은 "주소 검색" → 주소 검색·적용 다이얼로그(화주 주소록 + Kakao, 출처 배지로 구분).
- 화물 운송정보 입력/수정 → "운송+품목" 입력 다이얼로그, "금액" 입력 다이얼로그(계산 미리보기 포함).
  일시·방법은 작은 팝오버.
- 위험 액션 "화물 취소" / "배차 취소" / "정산 처리" → danger confirm 다이얼로그.
- 다이얼로그는 ds.css 패턴 유지: 헤더(제목+닫기), 본문, 푸터(취소 왼쪽 / primary 우측 끝),
  role="dialog" + aria-modal, 배경 클릭/ESC로 닫기. lookup의 행 선택은 단일 선택으로 토글.
- 화물정보 요약 "수정"은 다이얼로그가 아니라 같은 자리 inline input 입니다.

[유지 — 깨면 안 됨]
- 디자인 시스템(ds.css 토큰·컴포넌트), Pretendard, 상태색 규칙, 손글씨 폰트 미사용.
- 섹션 필드·상태·편집 권한은 첨부한 section-digest 기준 그대로:
  직접 편집 불가 필드(업체명·사업자번호·담당자명·차주명·차량번호·행정주소)는 잠금 표면,
  연락처·이메일·실중량·톤수·차종 등만 inline 편집.
- 금액 의미: 청구비용 / 운송비용 / 수수료(금액, 좌측 금액 블록) / 수익·차주운임(자동계산).
  인수증=수익(청구−운송), 그 외=차주운임(운송+수수료).
- 화주≠차주, 차주의 톤수·차종(차량 스펙) ≠ 화물 운송정보의 톤수·차종(운송 요구 조건).
- 외부 email 난독화 스크립트(cloudflare __cf_email__) 넣지 말고, 이메일은 평범한 샘플값으로.

[임의로 확정하지 말 것]
- 요약 긴 문장은 줄바꿈 보존(말줄임 금지). 수수료 인수증 시 숨김/비활성, 상태 chip 노출,
  화물맨 중복·기존차주 판정, 실제 API는 미확정 → 화면은 상태 표현만.
```
