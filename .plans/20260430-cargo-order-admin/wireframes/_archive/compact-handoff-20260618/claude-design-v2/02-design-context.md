# 02. 디자인 컨텍스트 — 현황·고도화 방향·보존 규칙

> 클로드 디자인이 "무엇을 정리하고, 어디로 끌어올리며, 무엇을 절대 깨면 안 되는가"를 판단하는 기준.

## A. 현재 구현물의 디자인 현황 (Implementation Clean.html)

| 영역 | 상태 | 근거 |
| --- | --- | --- |
| 레이아웃 | 중 | 5섹션 수직 스택 정렬됨. 단, 어드민 셸(헤더/네비/저장 푸터/목록) 부재 |
| 시각 디자인 | 중 | 토큰화·`var()` 충실, 그러나 이중 토큰 명명 혼재·미사용 폰트 |
| 상태 표현 | 상 | `is-empty`/`is-unselected`/`applied-editable`/`aria-live`로 체계적 |
| 반응형 | 중 | 1180/720px 분기 + `prefers-reduced-motion`. 720px 모바일 깊이 미검증 |
| 접근성 | 중상 | `role=dialog`/`aria-modal`/tablist/`aria-label`. 가시 헤딩(`<h2>`+) 없음 |
| 컴포넌트 일관성 | 중하 | 섹션별 접두사로 버튼·카드·필드 **중복 정의**(재사용 컴포넌트 아님) |

### 토큰 현황 (정리 대상)
- CSS 변수 32종, `var()` 511회 사용 — 기반은 양호.
- **이중 토큰 체계 혼재**: `--accent #1f64d6` / `--ink` / `--muted` / `--line` / `--success` / `--danger` / `--warn` / `--selected #fff7cc` (UI 토큰) **+** `--paper` / `--pen`(28회) / `--note`(59회) / `--title` / `--rule` / `--ink-mute`(38회) (종이/필기 테마 토큰). 두 네이밍이 섞임.
- 토큰 밖 hex 리터럴 60종 + rgba 27회 — 셰이드 산발.
- **미사용 손글씨 폰트**: Gaegu / Nanum Pen Script / Caveat 로드만 하고 CSS 미적용. 실제는 시스템 산세리프(`--font`)만 적용.
- 타입 스케일 비표준: 11/12/13/14/15/16/17/26/28/30px 산발(본문 13px). spacing 토큰 없이 px 직접 지정.

## B. 고도화 목표 방향

> 와이어프레임의 종이/손글씨 톤은 **저충실도 스케치 신호**였다. 고도화에서는 제거하고
> **차분하고 고밀도인 전문 업무용 데스크톱 UI**로 정돈한다. (마케팅/카드형 대시보드 아님)

| 원칙 | 설명 |
| --- | --- |
| 기존 구조 우선 | 5섹션 순서·필드 위치 유지. 사용자가 큰 변화에 불편하지 않게 |
| 고밀도 유지 | 핵심 필드를 탭 뒤에 숨기지 않고 빠르게 스캔 가능하게 |
| 단일 시맨틱 토큰 | `accent/ink/paper/note` 이중 체계 → 단일 시맨틱 토큰 세트로 통합 |
| 상태색 규칙화 | 접수=파랑 / 완료=중립 회색·검정 / 취소·위험=빨강 / 선택=강조 배경. 경고는 아이콘+문구 |
| 컴포넌트 추상화 | Button·Field·Card·Chip·Dialog·SummaryRow를 공통 컴포넌트로 |
| 위험 액션 분리 | 화물취소·배차취소·연동취소·정산처리는 일반 액션과 시각 구분 |
| 시각 위계 | 섹션 번호·제목·구분선으로 5섹션 스캔 가능성 향상 |

### 정리해야 할 디자인 부채 (우선순위 순)
1. **단일 디자인 토큰 체계 확립** — 색/타이포/스페이싱/라운드/섀도를 시맨틱 토큰으로 재정의(이중 체계 통합).
2. **타입 스케일·스페이싱 스케일** — 모듈러 타입 스케일 + 4/8 기반 spacing scale 토큰화.
3. **공통 컴포넌트 추출** — 섹션별 중복 버튼(4~5종)·칩 변종(chip/status-chip/metric-chip/source-badge 등) 통일.
4. **시각 위계 부여** — 가시적 섹션 헤더(번호+타이틀+구분선). `aria-label`만 의존하는 현 구조 보강.
5. **다이얼로그 표준화** — lookup/b-driver/b-cargo 다이얼로그의 푸터·버튼 패턴 통일.
6. **포커스/키보드 가시성** — inline-edit·trigger의 포커스 링·가시 상태 강화.
7. **미사용 자산 정리** — 손글씨 폰트 로드 제거(또는 컨셉 명시적 채택 — 권장: 제거).
8. **반응형 깊이** — 720px에서 상하차 다열 행·다이얼로그 모바일 레이아웃 검증.

### 어드민 셸 결정 필요 (프롬프트에서 사용자 확인)
현재는 입력 폼만 존재. 고도화 시 **페이지 헤더 + 전역 액션바(등록/수정/취소) + (선택) 하단 목록 자리표시**를 추가할지 결정 필요.
이번 범위는 `cargo-list` 제외이므로, 목록은 **자리표시 영역만 두고 실제 테이블은 별도 작업**으로 두는 것을 권장.

## C. 컴포넌트 설계 가이드 (고도화 산출물 기대치)

| 컴포넌트 | 변형 | 비고 |
| --- | --- | --- |
| Button | primary / secondary / ghost / **danger** | 위험 액션은 danger로 시각 분리 |
| Field | text / select / segment / money / date·time / inline-editable | inline은 read↔control 토글 표현 |
| SummaryRow | label-col + value + action | 32px 행 리듬, 5섹션 공통 |
| Chip / Badge | status(접수·완료·취소) / metric(거리·금액) / source(화물맨·내부 DB·Kakao) / role(배차·정산·관리) | 색+텍스트 병행 |
| Dialog | lookup(검색→결과→미리보기→적용) / confirm(YES·NO) / input(운송+품목·금액·일시) | 푸터·버튼 패턴 통일 |
| EmptyState | placeholder + CTA(흡수 전환) | 전 섹션 공통 |

## D. 반드시 보존해야 할 것 (고도화로 깨지면 안 됨)

- **5개 섹션 구성·순서**: 화주 → 운송구간(상/하차) → 화물운송정보(운송+품목/금액) → 화물요약 → 차주(내부DB/화물맨). `cargo-list` 제외 원칙.
- **업무 필드 일체**: [01-section-digest.md](01-section-digest.md)의 모든 필드. 특히 금액 흐름(청구−운송=수익 자동계산)과 수수료의 위치.
- **상태 전이 로직**: 입력대기→적용 전환, 화물맨 phase 상태 머신.
- **인터랙션 흐름**: inline edit(read↔control), 통합 조회 다이얼로그, 주소검색(화주 주소록+Kakao), 일시/방법 팝오버.
- **편집 권한 구분**: 직접 편집 불가 필드(업체명·사업자·담당자명·차주명·차량번호·행정주소)에 클릭 어포던스 주지 말 것.
- **데이터 책임 분리**: inline 임시 수정은 현재 오더에만 반영(마스터 불변). 화주≠차주, 차량 스펙≠운송 요구 조건.
- **접근성 골격**: `role=dialog`+`aria-modal`, tablist/tab/tabpanel, `aria-live` 유지(가시 헤딩만 추가).
- **데이터 훅 속성**: `data-b-*`/`data-route-*`/`data-summary-*`/`data-preview-*`(JS 동작 의존) 보존.
- **라벨 보기 토글**(`.label-mode`)과 hover/focus 라벨 노출.
