# 03. 미결정 / 리스크 통합

> 각 섹션의 self-review·closeout·quality-gap에서 나온 보류·미확정 항목을 취합.
> 이 항목들은 **디자인에서 임의로 확정하지 말 것**. 디자인은 "어느 쪽으로도 확장 가능한 형태"로 두거나, 명시적 옵션으로 표현한다.

## 전 섹션 공통

| 항목 | 영향 | 상태 | 다음 확인 |
| --- | --- | --- | --- |
| 실제 validation 규칙 | medium | 보류 | 저장 시 검증 규칙 확정 후 필드별 정의 |
| 실제 검색/저장 API 계약 | medium | 보류 | wireframe은 샘플 데이터 — 개발 단계 API contract |
| 어드민 셸(헤더/네비/전역 액션바/목록) 추가 범위 | high | 결정 필요 | 고도화 1단계에서 사용자 확인 |

## 1. 화주 정보

| 항목 | 영향 | 비고 |
| --- | --- | --- |
| 사업자 번호 검증 액션 위치 | low | 우측 액션 vs 별도 상세 미정 |
| 임시 수정값 저장 정책 | medium | 현재 오더 적용값 vs 마스터 저장 범위, audit 기준 미정 |
| 중복 담당자 처리 | medium | 현재는 단순 신규 추가만, 중복 체크 정책 후속 |
| 이메일 형식 validation 문구 | low | 1차 제외 |

## 3. 화물 운송정보

| 항목 | 영향 | 비고 |

| `수익/차주운임` 라벨 전환 시 위치 인지 | medium | 같은 위치 유지로 완화, 시각 검토 필요 |
| `운송 상품`(일반화물 분류)과 `품목` 병합 | medium | 최종 위치 미확정 |

## 디자인 단계에서의 처리 원칙

- **미정 = 옵션으로 표현하거나 확장 여지를 남긴다.** 임의 확정 금지.
- 특히 **금액/수수료 의미**, **요약 파생 데이터**, **편집 권한 구분**, **위험 액션 분리**는 보존 규칙([02-design-context.md](02-design-context.md) D)을 우선한다.
- API·validation·저장 정책은 디자인 범위 밖 → 화면은 상태 표현(대기/성공/실패/검증중)만 준비한다.
