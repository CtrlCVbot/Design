# Component Unification Options

## 비교 기준

| 기준 | 설명 |
| --- | --- |
| 스캔성 | 운영자가 한눈에 오더 정보를 읽을 수 있는가 |
| 명확성 | 처음 보는 사용자도 값의 의미를 알 수 있는가 |
| 밀도 | 화면이 과하게 늘어나지 않는가 |
| 구현 난이도 | 현재 HTML 구조를 크게 바꾸지 않고 반영 가능한가 |
| 디자인 확장성 | high fidelity 디자인에서 일관되게 확장 가능한가 |

## Option A. 전체 라벨 명시안

모든 필드에 라벨을 붙이는 방식입니다.

| 항목 | 내용 |
| --- | --- |
| 라벨 | 모든 값 위 또는 옆에 라벨 표시 |
| 버튼 | 모든 action 버튼을 같은 크기와 위치 규칙으로 정리 |
| 입력 전 | placeholder와 입력 버튼을 명시 |
| 적용 후 | 라벨+값 구조를 유지 |
| 수정 가능 | hover/focus 때 수정 표시 |

장점:
- 초보 사용자에게 가장 명확합니다.
- AI가 high fidelity 디자인을 입힐 때 각 값의 의미를 놓치기 어렵습니다.
- 접근성 문서화가 쉽습니다.

단점:
- 운송 구간처럼 많은 필드가 있는 row에서 밀도가 높아집니다.
- 현재 hand-drawn compact tone보다 화면이 무거워질 수 있습니다.
- 모바일이나 좁은 영역에서 줄바꿈이 늘어날 수 있습니다.

적합한 경우:
- 사용자가 초보 운영자 위주이거나, 교육 비용을 낮추는 것이 최우선일 때.

## Option B. 역할 기반 통일안

필드의 성격에 따라 라벨 노출을 다르게 가져가는 방식입니다.

| 필드 성격 | 라벨 기준 |
| --- | --- |
| 신원 정보 | 라벨 표시: 업체명, 사업자번호, 담당자, 연락처, 차량번호 |
| 금액 정보 | 라벨 표시: 결제방법, 청구비용, 운송비용, 수수료, 수익/차주운임 |
| 운송 위치 정보 | kind와 컬럼 순서 중심: 상차/하차, 지명, 주소, 상세주소 |
| 운송 조건 정보 | 짧은 값 중심: 지금, 당일, 상차방법, 하차방법 |
| 긴 요약 | kind와 본문 중심: 요약 |
| 출처/연동 | chip 또는 source field: 화물맨, 내부 DB |

버튼 규칙:

| 역할 | 스타일 |
| --- | --- |
| primary input | accent soft 배경, row 우측 또는 주요 위치 |
| secondary input | white 배경, border 유지 |
| change | 항상 row 끝, compact button |
| inline edit | 기본 숨김, hover/focus 때 노출 |
| external link | primary보다 약간 더 강조, 문구는 `화물맨 연동` |
| danger | danger soft 배경과 danger text |

장점:
- 현재 화면의 compact함을 유지합니다.
- 업무 맥락상 이미 알 수 있는 값은 라벨을 줄일 수 있습니다.
- 구현 난이도가 낮습니다.
- high fidelity에서 디자인 시스템으로 확장하기 쉽습니다.

단점:
- 라벨이 없는 항목은 컬럼 순서 문서가 필요합니다.
- 초기 사용자에게는 일부 값 의미가 약할 수 있습니다.

적합한 경우:
- 현재 B 통합안의 사용감을 크게 바꾸지 않고 정리하려는 경우.

## Option C. 최소 표시안

값 중심으로 최대한 간결하게 보여주는 방식입니다.

| 항목 | 내용 |
| --- | --- |
| 라벨 | 대부분 숨김 |
| 버튼 | hover 또는 row focus 때만 노출 |
| 입력 전 | 거의 버튼 중심 |
| 적용 후 | 값만 나열 |
| 수정 가능 | 점선/hover highlight 중심 |

장점:
- 가장 가볍고 빠르게 보입니다.
- 반복 입력 화면에서 시각 소음이 적습니다.
- 고급 운영자에게는 효율적입니다.

단점:
- 값 의미를 모르는 사용자는 학습이 필요합니다.
- high fidelity 작업 시 AI가 필드 의미를 오해할 수 있습니다.
- 구현 문서와 ARIA label 의존도가 높아집니다.

적합한 경우:
- 숙련 운영자만 사용하는 내부 tool이고, 화면 밀도가 최우선일 때.

## Option D+. Compact 기본 + 2행 라벨 보기 + hover/focus 임시 라벨

기본은 Option C처럼 값 중심 compact row로 표시하고, 사용자가 `라벨 보기` 토글을 켜면 Option A처럼 라벨이 명확한 2행 구조로 전환하는 방식입니다. 토글 OFF 상태에서도 특정 필드에 hover/focus하면 해당 필드 상단에 임시 라벨을 보여줍니다.

### 구조

기본 OFF:

```text
화주  로지스팩토리  123-45-67890  김민수  010-1234-5678
```

OFF + hover/focus:

```text
      화주 업체명
화주  로지스팩토리  123-45-67890  김민수  010-1234-5678
```

ON:

```text
화주   화주 업체명      사업자 번호      담당자명      담당자 연락처
       로지스팩토리     123-45-67890     김민수        010-1234-5678
```

### 규칙

| 항목 | 기준 |
| --- | --- |
| 기본값 | OFF |
| 토글 명칭 | `라벨 보기` 또는 `필드명 보기` |
| 토글 위치 | 좌측 작업 영역 상단 또는 고정 toolbar |
| 토글 저장 | 사용자별 preference로 마지막 선택값 저장 |
| ON 상태 | 모든 대상 필드를 2행 구조로 표시 |
| OFF 상태 | 데이터만 표시 |
| OFF + hover/focus | 해당 필드 상단에 임시 라벨 표시 |
| 라벨 위치 | 항상 field 상단 |
| row height | hover/focus 시 높이가 변하지 않도록 고정 |
| 모바일/터치 | tap/focus 시 임시 라벨 표시 |

### 장점

- 숙련자는 compact 화면으로 빠르게 입력할 수 있습니다.
- 신규 사용자나 검수자는 라벨을 켜서 정확히 확인할 수 있습니다.
- 토글 OFF에서도 헷갈리는 값만 hover/focus로 확인할 수 있습니다.
- 라벨 위치가 항상 상단이라 ON/OFF 규칙이 흔들리지 않습니다.
- high fidelity 디자인에서 `compact mode`와 `labeled mode`라는 명확한 상태로 확장할 수 있습니다.

### 단점

- 2행 모드에서는 row 높이가 증가합니다.
- hover/focus 임시 라벨은 겹침과 row height jump를 조심해야 합니다.
- 사용자별 preference 저장이 필요합니다.
- 모든 필드에 적용하면 구현 범위가 커질 수 있어 우선순위가 필요합니다.

### 적용 우선순위

| 우선순위 | 섹션 | 이유 |
| --- | --- | --- |
| 1 | 화주 정보 | 라벨 의미가 명확하고 필드 수가 적절함 |
| 2 | 화물 운송정보 | 톤수/차종/금액 등 오해 위험이 큼 |
| 3 | 차주 정보 | 차량번호/연락처/톤수/차종/출처 구분 필요 |
| 4 | 운송 구간 | 컬럼 규칙이 강하므로 신중히 적용 |
| 5 | 화물정보 요약 | 단일 본문이라 라벨 모드 필요성이 낮음 |

## 추천안

추천은 **Option D+. Compact 기본 + 2행 라벨 보기 + hover/focus 임시 라벨**입니다.

| 판단 기준 | 평가 |
| --- | --- |
| 스캔성 | 높음 |
| 명확성 | 높음 |
| 밀도 | 높음 |
| 구현 난이도 | 중간 |
| 디자인 확장성 | 높음 |

## 추천 이유

1. 기본값은 compact라 현재 화면의 hand-drawn tone과 반복 입력 효율을 유지합니다.
2. 필요할 때만 A안처럼 2행 라벨을 켤 수 있어 신규 사용자와 검수 상황을 보완합니다.
3. 토글 OFF 상태에서도 hover/focus 임시 라벨로 특정 필드 의미를 즉시 확인할 수 있습니다.
4. 라벨 위치를 항상 상단으로 통일해 ON/OFF 상태의 디자인 문법이 같습니다.
5. 우측 빈공간을 나중에 채우더라도 좌측 row 규칙이 흔들리지 않습니다.

## 추천안 적용 원칙

| 원칙 | 내용 |
| --- | --- |
| 기본은 데이터만 표시 | compact mode를 기본값으로 유지 |
| 라벨 보기 ON 시 2행 구조 | 1행 라벨, 2행 데이터 |
| 라벨 보기 OFF + hover/focus | 해당 필드 상단에 임시 라벨 표시 |
| row height 고정 | hover/focus 시 layout jump 방지 |
| 버튼은 역할별로 통일 | 입력, 변경, 외부 연동, 위험, inline edit |
| 입력 전 상태는 짧게 | 긴 설명 문장 대신 placeholder 구문 사용 |
| 적용 후 상태는 row 안정성 유지 | 필드 위치가 상태별로 움직이지 않게 유지 |
| 수정 가능 필드는 hover/focus 일관화 | inline edit은 같은 affordance 사용 |

## 확정 전 확인 질문

| 질문 | 결정 필요성 |
| --- | --- |
| 토글 명칭을 `라벨 보기`로 할지 `필드명 보기`로 할지 | 사용자 이해에 영향 |
| 운송 구간에도 2행 라벨 모드를 적용할 것인가 | row 높이와 밀도에 직접 영향 |
| `화물맨 연동`은 primary action으로 계속 강조할 것인가 | 차주 정보의 주된 시작점 결정 |
| 입력 전 안내문은 어느 정도까지 허용할 것인가 | clean 화면의 밀도 결정 |
| hover/focus 임시 라벨을 모든 필드에 줄 것인가 | 발견성과 시각 소음 trade-off |
