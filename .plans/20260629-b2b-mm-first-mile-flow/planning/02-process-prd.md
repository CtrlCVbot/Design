# B2B MM first-mile 수출입 process PRD

## 1. 목적

B2B MM first-mile 수출입 운영자가 견적 문의부터 `B/L` 입력, 선사/콘솔사 작업, 적하목록/EDI, 선적/운송 완료, 정산까지 이어지는 업무를 `PMS` 중심으로 추적하고 처리할 수 있게 한다.

여기서 first-mile은 Figma 기준으로 국내 운송의 출발 지점만 뜻하지 않고, 국제 물류 오더가 실제 선적/운송과 정산으로 넘어가기 전까지의 초기 운영 구간을 포함한다.

## 2. 문제 정의

| 문제 | 영향 |
| --- | --- |
| 견적, 오더, 서류, 선사/콘솔사 요청이 `PMS`, `TOMS`, 메일, 카톡, 엑셀, 외부 사이트에 흩어짐 | 진행 상태 확인이 어렵고 누락 가능성이 높음 |
| 수출 `FCL/LCL`과 수입 flow가 `B/L` 이후 갈라지지만 공통 기준 상태가 약함 | 화면과 데이터 모델이 분기마다 중복될 수 있음 |
| 적하목록, EDI 전송, 오류 정정이 별도 작업처럼 보임 | 오류 발생 시 재처리와 선사 제출 상태 추적이 어려움 |
| 정산/청구/입금이 수출/수입별로 다르게 연결됨 | 실적정산 완료 기준이 모호해질 수 있음 |

## 3. 핵심 사용자

| 사용자 | 주요 책임 |
| --- | --- |
| 영업/운영 담당자 | 견적 문의 접수, 비용 확인, 견적 공유, 오더/서류 등록 |
| 수출 운영 담당자 | 선복 부킹, `S/R`, `VGM`, Master copy `B/L`, 적하목록/EDI 처리 |
| 수입 운영 담당자 | 수입 `B/L`, 적하목록/EDI, 운송 완료, `D/O` 관련 처리 |
| 정산 담당자 | 지출결의서, 해외 정산서, 화주 청구서, 세금계산서, 입금 확인, 실적정산 |
| 외부 파트너 | 선사, 항공사, 콘솔사, 화주, 운송사 |

## 4. 범위

### 4.1 포함 범위

| 영역 | 포함 |
| --- | --- |
| 거래처/기초 데이터 | 기초 데이터 등록, 신규 업체/담당자 등록 |
| 견적 | 견적 문의, 비용 확인, 견적서 작성/공유 |
| 오더/서류 | 오더/서류 등록, `CI / PL` 등록 여부, 오더 디테일 확인/수정 |
| `B/L` | `B/L` 입력, 수출/수입 분기, `M/BL`, `H/BL` 발행 |
| 수출 | `FCL`, `LCL`, 선복 부킹, 선사 승인, `S/R`, `VGM`, Master copy `B/L`, Master way bill |
| 수입 | 수입 `B/L` 입력, 운송 완료, `M/D/O`, `H/D/O` |
| EDI | 적하목록 발행/출력, EDI 전송, EDI 정정신고, 선사 제출 |
| 정산 | 지출결의서, 해외 정산서, 화주 청구서, 세금계산서, 입금 확인, 실적정산 |

### 4.2 제외 범위

| 제외 항목 | 이유 |
| --- | --- |
| 외부 선사/콘솔사 사이트 내부 화면 | 외부 시스템 상세 정책이 필요함 |
| 다우 회계/송금 내부 프로세스 | 회계 시스템 상세 계약이 필요함 |
| 세관 EDI 전문 규격 상세 | 별도 기술/법규 검토가 필요함 |
| API/DB 스키마 | 이번 패키지는 기획 분석 산출물임 |
| 1차/2차 섹션 차이의 완전 비교 | Figma MCP 호출 제한으로 직접 확인하지 못함 |

## 5. 핵심 상태 모델

| 상태 | 의미 | 주요 다음 액션 |
| --- | --- | --- |
| `draft` | 기초 데이터 또는 업체 정보가 준비 중 | 업체/담당자 등록 |
| `quote_requested` | 견적 문의가 접수됨 | 비용/스케줄 확인 |
| `quote_shared` | 견적서가 작성/공유됨 | 오더/서류 등록 |
| `order_registered` | 오더가 등록됨 | `CI / PL` 확인 |
| `document_ready` | `CI / PL` 등 기본 서류가 준비됨 | 오더 디테일 확인 |
| `bl_input_ready` | `B/L` 입력 가능 상태 | 수출/수입 분기 |
| `export_booking` | 수출 선복/콘솔 작업 중 | `S/R`, `VGM`, Master copy 확인 |
| `import_processing` | 수입 `B/L` 및 적하목록 작업 중 | EDI/운송 완료 처리 |
| `manifest_ready` | 적하목록 발행/출력 완료 | EDI 전송 |
| `edi_error` | EDI 오류 발생 | EDI 정정신고 |
| `edi_submitted` | EDI 정상 제출 또는 정정 후 제출 | 선적/운송 완료 |
| `shipment_completed` | 수출 선적완료 | 정산/청구 |
| `transport_completed` | 수입 운송완료 | 정산/청구/`D/O` |
| `settlement_in_progress` | 지출/청구/세금계산서/입금 처리 중 | 입금 확인 |
| `performance_settled` | 실적정산 완료 | 종료 |

## 6. 요구사항

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| `REQ-MM-001` | 운영자는 `PMS`에서 기초 데이터와 신규 업체/담당자를 등록할 수 있어야 한다. | Must |
| `REQ-MM-002` | 운영자는 견적 문의 채널이 `TOMS`인지 메일인지 구분해 기록할 수 있어야 한다. | Must |
| `REQ-MM-003` | 운영자는 선사/항공사/콘솔 비용 확인 결과와 견적서 산출물을 연결할 수 있어야 한다. | Must |
| `REQ-MM-004` | 운영자는 오더/서류 등록 단계에서 `CI / PL` 등록 여부를 확인하고 미등록 시 서류 등록으로 유도받아야 한다. | Must |
| `REQ-MM-005` | `B/L 입력`은 수출 `FCL`, 수출 `LCL`, 수입 flow의 공통 분기점으로 관리되어야 한다. | Must |
| `REQ-MM-006` | 수출 `FCL`은 선복 부킹 요청, 선사 승인, 부킹 번호 부여, `S/R`, `VGM`, Master copy `B/L` 수신을 추적해야 한다. | Must |
| `REQ-MM-007` | 수출 `LCL`은 콘솔사 선적 요청, `S/R` 출력/제출, 콘솔사 Master copy `B/L` 수신을 추적해야 한다. | Must |
| `REQ-MM-008` | 수입 flow는 `B/L 입력` 이후 적하목록 발생 전까지 필요한 추가 작업을 정의해야 한다. | Must |
| `REQ-MM-009` | `M/BL`, `H/BL` 발행과 적하목록 발행/출력은 EDI 전송 전 필수 단계로 추적되어야 한다. | Must |
| `REQ-MM-010` | EDI 전송 결과가 오류인 경우 `EDI 정정신고` 후 선사 제출로 이어지는 재처리 flow를 지원해야 한다. | Must |
| `REQ-MM-011` | EDI 전송, 확인, 정정 업무는 한 화면 또는 하나의 작업 묶음으로 처리 가능해야 한다. | Should |
| `REQ-MM-012` | 수출 선적완료와 수입 운송완료는 서로 다른 후속 정산 flow를 가져야 한다. | Must |
| `REQ-MM-013` | 지출결의서, 해외 정산서, 화주 청구서, 세금계산서, 입금 확인은 실적정산과 연결되어야 한다. | Must |
| `REQ-MM-014` | 유니패스, 트래킹 조회 같은 외부 조회는 어느 단계에서 필요한지 명확히 표시되어야 한다. | Should |

## 7. 수용 기준

| 기준 | 완료 조건 |
| --- | --- |
| 공통 flow | 견적 문의부터 `B/L 입력`까지 수출/수입 공통 단계가 하나의 상태 흐름으로 설명된다. |
| 수출 분기 | `FCL`과 `LCL`의 선사/콘솔사 차이가 화면/상태/서류 기준으로 구분된다. |
| 수입 분기 | 수입 `B/L` 입력 후 적하목록 발생 전까지의 빈 구간이 질문 또는 결정 항목으로 남는다. |
| EDI | EDI 전송, 오류 판단, 정정신고, 선사 제출이 하나의 업무 묶음으로 설계된다. |
| 정산 | 수출 선적완료와 수입 운송완료 이후 실적정산까지의 문서 흐름이 끊기지 않는다. |
| 추적성 | 각 요구사항이 Figma 요소 또는 미결 질문과 연결된다. |

## 8. MVP 제안

첫 구현 또는 상세 기획은 다음 범위부터 시작하는 것을 추천한다.

| 우선순위 | 범위 | 이유 |
| --- | --- | --- |
| 1 | 공통 오더/서류 등록과 `B/L 입력` | 모든 수출입 flow의 중심 게이트 |
| 2 | 적하목록/EDI 작업 묶음 | Figma에 명확한 개선 주석이 있음 |
| 3 | 수출 `FCL/LCL` 상태 추적 | 선사/콘솔사 분기와 외부 제출물이 많아 누락 리스크가 큼 |
| 4 | 정산 연결 상태 | 실적정산 완료 기준을 앞단 업무와 연결해야 함 |
| 5 | 수입 상세 보완 | Figma 주석상 미확정 구간이 있어 확인 후 상세화 필요 |
