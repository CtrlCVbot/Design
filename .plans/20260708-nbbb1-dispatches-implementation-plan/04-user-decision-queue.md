## 2. 결정 상세

### DEC-001 — 배차 상태 모델 통합·상태 전이 규칙·mutation 연결

- 통합 NB: **NB-063** (6단계 상태 모델 통합), **NB-046** (상태 버튼 6종 mutation 연결) — 같은 상태 도메인의 모델 결정과 실행 결정이므로 통합.

**질문**: nbbb1 6단계 상태 모델(배차중→배차완료→상차완료→하차완료→운행완료 + 취소)을 기존 `orders.flow_status`/dispatch 상태 enum과 어떻게 통합·매핑하는가? 비순차 전환(목업 전제 ⚠)을 허용하는가? 아코디언 상태 버튼 6종에 mutation을 어느 범위까지 연결하는가?

**배경 (Phase 3에서 왜 미결인가)**:
- 상태 데이터 모델과 변경 mutation은 legacy 상세 시트에 완비되어 있으나(IMPL-D5-058 배송 상태 dropdown, IMPL-D5-057 배차 취소 batch cancel, IMPL-D4-004 운송 마감/정산 생성 + server domain guard), Phase 3 콘솔 drawer의 `배차 취소`/`운행 마감` 버튼은 **조건부 노출만 구현되고 mutation 미연결** 상태다(IMPL-D5-022 코드 검증: deferred/disabled, 클릭 핸들러 없음; IMPL-D4-038: createSettlement 호출 경로 부재) — 후속 Feature 연결로 의도적으로 보류됐다(M2 NB-046).
- 명칭·단계 수가 불일치한다: 기존 enum은 운송요청/배차완료/운송마감/취소 + dispatch 상하차 상태이고, nbbb1은 6단계다(M5 NB-063 수정 포인트 (1)).
- 목업은 비순차 전환을 허용(⚠)하나 기존 server domain guard(마감/취소 잠금 등)와 충돌 가능성이 있어 전이표 정의가 선행돼야 한다(M5 NB-063 수정 포인트 (2)). legacy 쪽 잔여 gap(IMPL-D5-057 batch cancel의 confirm 없음·reason 고정, IMPL-D4-001 settlement-type 전환 서버 lock 부재 — 계획만)도 함께 걸려 있다.

**선택지**:

| 선택지 | 내용 | 영향 | 비용 |
|---|---|---|---|
| A | 기존 enum 유지 + 표시 계층 매핑(상차완료/하차완료는 dispatch 상하차 상태에서 파생) | DB/서버 무변경 — Phase 3 "backend 무변경" 원칙 유지. 매핑 불가능 상태가 있으면 표시 축소 | 낮음. 매핑표 정의 + selector 구현 |
| B | enum 자체를 6단계로 확장(서버 domain 변경) | nbbb1 모델과 1:1 정합. 기존 오더 데이터 migration·legacy 화면 영향 발생 | 높음. 서버 domain guard·legacy 화면 회귀 범위 큼 |
| C | 전이 규칙: 순차 강제(전진만 허용 + 취소는 마감 전 항상 가능) | server domain guard와 충돌 없음. 목업의 비순차 UX와 다름 | 낮음 |
| D | 전이 규칙: 비순차 허용(목업 그대로) | 잘못 누른 상태 복구가 쉬움. server guard(마감 잠금 등)와의 충돌 지점별 예외 설계 필요 | 중간. guard 예외 규칙 + 감사 로그 검토 필요 |

**권고안**: **A + C(단, 후진 1단계 되돌리기만 예외 허용 검토)**. 근거: 상태 변경 mutation 자산이 전부 기존 enum 전제로 완성돼 있고(IMPL-D5-057/058, D4-004), Phase 3 콘솔 미연결 지점(IMPL-D5-022/D4-038)은 "연결"이 과제이지 "재설계"가 과제가 아니다. enum 확장(B)은 legacy 화면·server guard 회귀 범위가 커서 이번 UI 재구성 범위와 분리해야 한다. mutation 연결 범위는 6종 버튼 전부 연결하되, 상차완료/하차완료는 dispatch 상하차 상태 갱신(IMPL-D5-058 자산 경로)으로, 운행 마감은 IMPL-D4-004 createSettlement 경로로 연결한다.

**이 결정에 막히는 항목**: NB-063, NB-046 (직접) / NB-029 상태 필터 칩 매핑, NB-064 필터 교차 카운트, NB-052 배정 시 자동 전환 검증, NB-035·NB-038 "배차중 한정" 파생 UI selector (파급) / WP-목록·WP-행확장·WP-배정패널·WP-전역.

### DEC-002 — 목록 UX/IA 재설계 gate 해제

- 통합 NB: **NB-037** (+NB-027, NB-029 동반)

**질문**: Phase 3가 "재설계하지 않는다"(DEC-F15-005)로 확정하고 "신규 목록 기획 확정 전까지" 조건부 보류한 목록 UX/IA 재설계(IMPL-D7-043)에 대해, nbbb1 목록 기획(NB-027/029/037 — 날짜·상태 건수 칩 + 7컬럼 IA + 행 확장 아코디언)을 그 "신규 목록 기획"으로 인정하고 gate를 여는가?

**배경**: Phase 3의 canonical 목록 자산은 legacy `broker-order-list-main.tsx` 계열이며 **무변경 재사용 계약**으로 보호된다(IMPL-D7-022/023, M2 판정 전제). 재설계는 DEC-F15-005로 기각·조건부 보류(IMPL-D7-043)됐고, 보류 해제 조건이 정확히 "신규 목록 기획 확정"이다. nbbb1 목록 사양은 데이터 소스(`GET /api/orders/with-dispatch`)는 재사용하고 표시 계층만 전면 재설계하는 구조라(M2 관찰 메모 2) gate의 재개 조건을 충족하는 첫 사례다.

**선택지**:

| 선택지 | 내용 | 영향 | 비용 |
|---|---|---|---|
| A | gate 해제 — nbbb1 기획을 신규 목록 기획으로 인정, 데이터 소스 재사용 + 표시 계층 재설계 | §2 12항목 착수 가능. legacy 목록은 무변경 유지(신규 route에서 병행) | 중간. 컬럼 IA·칩 카운트·아코디언 신규 |
| B | gate 유지 — 기존 목록 UI에 최소 수정(칩만 추가) | nbbb1 콕핏 구조(행 확장 아코디언·우측 패널 상주) 실현 불가 | 낮으나 nbbb1 핵심 UX 포기 |

**권고안**: **A**. 근거: IMPL-D7-043의 보류는 무기한 기각이 아니라 "신규 기획 확정 전까지"라는 조건부이고, nbbb1이 그 조건을 충족한다. 단 M2 관찰 메모 2의 전제 조건을 함께 결정에 포함할 것 — 착수 전 목록 조회 API 안정성 이슈(IMPL-D7-066 미확인, IMPL-D5-026/D8-029 smoke 중 500/0건, 수정 커밋 부재) 런타임 확인을 선행한다.

**이 결정에 막히는 항목**: NB-037, NB-027, NB-029 (직접) / §2 운행 내역 리스트 전체 11항목 + NB-064 (파급) / WP-목록.

### DEC-003 — 구간(pair) 집계 인프라: 서버 집계 vs 클라이언트 근사

- 통합 NB: **NB-055** (+NB-007 동일 데이터 소스)

**질문**: 운행 데이터 기반 주요 구간(상차+하차 pair) 빈도 랭킹에 필요한 신규 집계 백엔드를 만들 것인가? 서버측 병합/랭킹 API가 Phase 3에서 일관 보류(IMPL-D2-034)된 상태에서, 보류를 해제한 서버 집계로 갈지 클라이언트 근사 집계로 시작할지?

**배경**: Phase 3 주소 후보는 전부 side별 단일 주소(saved/recent) 기준이고 적용도 side별 독립이라(IMPL-D2-018, D2-004) "pair 동시 입력·pair 랭킹" 동작 자체가 미구현이다(M1 NB-007). 서버측 후보 병합/랭킹 API(`GET /addresses/location-candidates`)는 "client adapter 검증 후 2차 후보"(E1-37/E2-49/P09-26) + "신규 outbound 금지"(P16b-09)로 일관 보류됐고 재개 기록이 없다(IMPL-D2-034, M3 NB-055). M1 관찰 메모 2가 지적하듯 **구간 집계 쿼리 하나가 NB-007(구간 칩)·NB-022(운임 추천)·NB-055(추천 패널 주요구간)를 동시에 해결**한다.

**선택지**:

| 선택지 | 내용 | 영향 | 비용 |
|---|---|---|---|
| A | IMPL-D2-034 보류 해제 — 서버 집계 API 신설(구간 빈도 + 운임 통계 동시 반환) | 정확한 랭킹·페이지네이션 무관 전체 이력 집계. NB-007/022/055/065 공통 소스 확보. Phase 3 "신규 outbound 금지" 원칙의 첫 예외 | 높음. 백엔드 신규 개발 + 쿼리 성능 검토 |
| B | 클라이언트 근사 집계 — 기존 `/api/orders/with-dispatch` 응답을 클라이언트에서 pair 그룹핑 | 백엔드 무변경. 조회 페이지 범위 내 근사치라 랭킹 정확도 낮음(장기 이력 미반영) | 낮음. 단 "주요 구간"이라는 기능 가치 훼손 |
| C | 1차 B로 시작 + 서버 집계를 별도 백엔드 패키지로 승인(2단계) | 착수 지연 없음 + 정확도는 후속 개선 | 중간. 근사→정식 전환 시 정렬 결과 변동 |

**권고안**: **A**. 근거: 이 집계는 NB-007·NB-022·NB-055·NB-065 4개 기능의 공통 전제라 근사치(B)로 시작하면 4개 기능 전부가 낮은 품질로 출발하고, pair 빈도·운임 통계는 오더 전체 이력 집계라 클라이언트 근사와 결과 차이가 크다. IMPL-D2-034의 보류 사유("client adapter 검증 후 2차 후보")는 이미 client adapter가 검증 완료(IMPL-D2-011/013/014 완료)됐으므로 해제 조건이 성립한다. 단 백엔드 규모 산정 결과 이번 릴리스에 못 들어가면 C로 전환한다.

**이 결정에 막히는 항목**: NB-055 (직접) / NB-007 자주 쓰는 구간 칩, NB-022 운임 추천(데이터 소스), NB-065 입력 추천 체인 3단계 (파급) / WP-폼·WP-추천패널.

### DEC-004 — 운임 통계 추천 승인 + 집계 scope + AI 운임 지표

- 통합 NB: **NB-022** (운임 추천 버튼), **NB-065** (집계 scope), **NB-038** (AI 어드바이저의 구간 평균 운임 지표) — 셋 다 "과거 오더 운임 통계를 노출해도 되는가 + 어느 범위 데이터로"라는 같은 근본 결정.

**질문**: ① 구간별 최근/최빈 운임 추천(통계 기반)을 운임 산식 금지선(IMPL-D4-057/058)과 별개로 승인하는가? ② 집계 scope는 선택 화주 한정인가, 주선사 자사 오더 전체인가(cross-company 금지선 RSMP-BLOCK-004와의 경계)? ③ AI 어드바이저의 "구간 평균 운임(적정/유사/평균 이하)" 지표는 pricing source 확보를 선행할지, 지표를 축소(운임 제외)해 시작할지?

**배경**:
- 기준금액/자동 운임 계산은 **"pricing source 미확인, distance에서 임의 추론 금지"** 금지선으로 보류됐다(IMPL-D4-057, 31-closeout 07-03; IMPL-D4-058 동일 gate 종속). 단 nbbb1의 최근/최빈 운임은 **요율 산식이 아니라 과거 오더 집계**라 금지선과 성격이 다르다(M1 NB-022 — "기획서에 경계 명시" 권고 포함).
- cross-company recent 집계는 기각 확정 금지선이다(IMPL-D8-018, IMPL-D1-024 — RSMP-BLOCK-001/004). "주선사 자사 오더 전체" 집계는 자사 데이터이므로 이 금지선 위반이 아니나 경계 확인이 M5 NB-065에서 요구됐다.
- NB-038의 운임 지표는 "적정/평균 이하" **판정**을 포함해 단순 통계 노출보다 한 단계 더 나간다 — 판정 기준(임계값) 정의가 산식 영역에 접근한다(M2 NB-038).

**선택지**:

| 선택지 | 내용 | 영향 | 비용 |
|---|---|---|---|
| A | 통계 추천 승인(자사 오더 전체 scope) + AI 운임 지표는 "판정 없이 통계 표시만"으로 축소 | 최근/최빈 원클릭 입력(NB-022)·추천 체인(NB-065) 실현. "적정" 판정은 산식 gate 뒤로 유보 | 중간. DEC-003 집계 인프라에 운임 필드 추가 |
| B | 통계 추천 승인(선택 화주 한정 scope) | 금지선에서 가장 먼 보수적 scope. 화주별 오더 수가 적으면 최빈값 무의미 | 중간. 데이터 희소성으로 추천 품질 낮음 |
| C | pricing source 확보 선행 — 운임 관련 전부 보류 유지 | 금지선과 완전 무충돌. NB-022 미착수, NB-038 지표 3종 중 운임 제외, NB-065 체인 3단계 제외 | 없음(기능 포기) |

**권고안**: **A**. 근거: ① 과거 자사 오더의 실제 운임 통계는 IMPL-D4-057이 금지한 "distance 기반 임의 추론"이 아니라 실데이터 집계로 성격이 다르며, M1 NB-022도 이 구분을 전제로 경계 명시를 권고했다. ② scope는 자사 오더 전체가 타당 — RSMP-BLOCK-004의 금지 대상은 cross-company(타사 데이터) 집계이고 자사 전체 집계는 경계 안이다. 단 이 경계 해석을 결정문에 명문화한다. ③ "적정/평균 이하" 판정은 임계값 산식이 필요해 IMPL-D4-057 gate 소관이므로, AI 어드바이저 1차는 판정 없는 통계 표시(구간 평균·최근값)로 축소한다.

**이 결정에 막히는 항목**: NB-022, NB-065, NB-038(운임 지표부) (직접) / NB-007 구간 칩의 운임 노출부 (파급) / WP-폼·WP-목록(AI 어드바이저)·WP-추천패널. 전제: DEC-003.

### DEC-005 — 대화방 패키지 범위·채널

- 통합 NB: **NB-058** (/chats 전체 페이지), **NB-060** (메시지 송수신 채널), **NB-061** (보조 아이콘 2종) — §6 대화방의 범위·채널·부속 기능이라 하나의 패키지 결정.

**질문**: ① 대화방을 콕핏 패널 전용으로 축소하고 `전체 화면` 버튼(/chats 이동)을 제거하는가, /chats 전체 페이지까지 신규 기획하는가? ② 메시지 채널은 실시간 양방향 채팅 신규 구축인가, legacy LMS 문자 자산(IMPL-D5-067) 활용/병행인가? ③ 하단 보조 아이콘 2개(문서/즐겨찾기 — 목업에서도 동작 미정의 ⚠)는 첨부·템플릿으로 신규 정의하는가, 1차 제외하는가?

**배경**:
- **Phase 3에 대화방(채팅) 도메인 자산은 존재하지 않는다** — D1~D8 전수 검토 결론(M4 서두). 인접 자산은 3건뿐: IMPL-D5-067 legacy LMS 문자(작성 전송 + 이력, legacy 상세 시트 한정·Phase 3 콘솔 미반영·단방향 SMS), IMPL-D7-051~053 운영 메모(내부 기록용, 상대방 소통 아님), IMPL-D7-013 콘솔 store 패턴.
- /chats 페이지는 nbbb1 분석 자체도 범위 밖으로 뒀다(NB-001 비고 "배차 관리 외 페이지는 분석 범위 밖", M4 NB-058). M4의 판정도 NB-058은 유일한 제외후보다.
- 첨부 기능의 전제인 파일 storage 자산이 없다(IMPL-D8-025 — base64 UI만 존재, API/storage write 없음, M4 NB-061).
- 실시간 채팅 백엔드(메시지 저장·전달·읽음 처리)는 Phase 3 "backend 무변경" 원칙 밖의 **첫 신규 서버 도메인**이 될 규모다(M4 관찰 메모 3 — 별도 기획 패키지 규모 산정 필요).

**선택지**:

| 선택지 | 내용 | 영향 | 비용 |
|---|---|---|---|
| A | 패널 전용 + LMS 활용: `전체 화면` 버튼 제거, 송신은 LMS 문자(IMPL-D5-067) 재사용, 보조 아이콘 1차 제외 | 완료 자산 재사용으로 빠른 착수. 단방향(수신 표시 불가)이라 "대화방"이 아닌 "메시지 발송 + 이력" UX로 축소. LMS gap(약한 server validation, placeholder 버튼) 해소 필요 | 낮음~중간 |
| B | 실시간 양방향 채팅 신규 구축 + /chats 페이지 포함 | nbbb1 사양 완전 실현. 신규 서버 도메인(저장·실시간 전달·읽음)·상대방 클라이언트(차주 앱/화주 화면) 전제 — 별도 패키지 규모 | 매우 높음. 이번 범위 내 완결 불가 가능성 |
| C | 1차 A로 착수 + 실시간 채팅을 별도 기획 패키지로 분리 상신(2단계), /chats·보조 아이콘은 그 패키지에서 재판단 | 이번 범위는 완료 자산으로 완결, 대화방 진짜 요구는 규모 산정 후 결정 | 낮음(1차) + 별도 패키지 |

**권고안**: **C**. 근거: 대화방은 M4 전체가 "자산 없음" 도메인이고(M4 관찰 메모 1), 실시간 채팅은 상대방 수신 클라이언트까지 전제하는 신규 서버 도메인이라(M4 관찰 메모 3) 이번 UI 재구성 범위에 넣으면 전체 일정을 지배한다. LMS 발송+이력으로 "연락" 요구의 최소 가치를 먼저 제공하고, 상대 정보 바(NB-062)는 화주/차주 snapshot 재사용(IMPL-D1-001/D5-001)으로 함께 구현한다. NB-058 `전체 화면` 버튼과 NB-061 보조 아이콘은 1차 제외.

**이 결정에 막히는 항목**: NB-058, NB-060, NB-061 (직접) / NB-059 복귀 버튼, NB-062 상대 정보 바 (파급 — 패널 존재가 전제) / NB-039·NB-044의 `대화방` 버튼 / WP-대화방·WP-행확장(버튼).

### DEC-006 — 차주 추천·차주 데이터 모델 확장

- 통합 NB: **NB-048** (AI 최적 추천 차주), **NB-050** (차주 카드 필드), **NB-051** (운행중 차주 연계지 거리) — 셋 다 "차주 데이터 모델(위치·상태·이력·계좌) 공백"이 근본 원인.

**질문**: ① AI 최적 추천 차주(최대 3명 + 추천 사유 태그)를 위해 보류된 고급 배차 추천(IMPL-D5-025)·후보 자동 매칭(IMPL-D5-045)을 재개하는가, 1차는 전체 차주 목록만으로 가는가? ② 차주 카드의 계좌·현위치 거리·월 총운임·운행 상태 필드는 데이터 모델 확장을 선행하는가, 가용 데이터로 축소하는가? ③ 운행중 차주의 연계지 거리(실시간 위치 수집 인프라 전제)를 1차 범위에 포함하는가?

**배경**:
- 고급 배차 추천은 Epic 차원 보류다 — E1-35(비범위), E2-51(deferred roadmap), P13-22(F4C Non-Scope 기각, "별도 domain logic 필요")(IMPL-D5-025). CargoMan 후보↔내부 DB 자동 매칭도 "Phase 3 완료 후 별도 기획"(P02-19~21, 2026-07-07 결정)(IMPL-D5-045). 추천 사유 태그에 필요한 차주 위치·상태·이력 분석 데이터 모델 자체가 부재하고, nbbb1 원문도 추천 알고리즘 미기록(⚠)이다(M3 NB-048).
- 차주 카드 코어(차주명/연락처/차량번호/톤수/차종)와 최근 배차 metadata·소속 chip은 완료 자산이다(IMPL-D5-002/005/001). 계좌 필드는 IMPL-D5-004(차주 등록 확장)가 **계획만** 상태(P12-19, 실현 기록 없음)이고, 현위치 거리·월 총운임·운행 상태는 데이터 소스 전무다(M3 NB-050).
- 거리 계산 서버 chain(IMPL-D2-050, `POST /api/distance/calculate`)은 완료 자산이라 좌표 2점만 있으면 재사용 가능하나, 입력이 되는 차주 위치/하차 예정지 소스가 없다 — 차주 앱 위치 수집 등 인프라 전제(M3 NB-051).

**선택지**:

| 선택지 | 내용 | 영향 | 비용 |
|---|---|---|---|
| A | 1차 축소: 전체 차주 목록(IMPL-D5-002/003 재사용) + 카드는 가용 필드(이름/차량번호/스펙/최근 배차/소속 chip)만. 추천·위치·월 총운임·연계지 거리 제외 | 배정 패널 즉시 착수 가능. nbbb1 §4의 차별 가치(추천) 유보 | 낮음 |
| B | A + 차주 데이터 모델 확장 1단계(계좌: IMPL-D5-004 실현 + 운행 상태 enum 신설)를 병행 | 카드 완성도 상승, 상태 기반 "가용 차주" 필터(NB-049) 가능해짐 | 중간. 차주 스키마 확장 |
| C | 추천 패키지 신규 착수(IMPL-D5-025/045 재개 + 위치 수집 인프라 기획 포함) | nbbb1 사양 완전 실현. Epic 보류 2건 해제 + 차주 앱 연동 인프라라 별도 로드맵 규모 | 매우 높음 |

**권고안**: **B**. 근거: 배정 실행 계층(assignDriver·상태 동기화·수배 경로 enum)은 완료 자산이 받쳐주므로(M3 관찰 메모 1) 목록·배정은 지금 착수 가능하다. 계좌와 운행 상태는 위치 인프라와 달리 스키마 확장만으로 가능하고 IMPL-D5-004에 이미 계획 근거가 있다. 추천(NB-048)과 실시간 위치(NB-051)는 IMPL-D5-025/045의 보류 사유("별도 domain logic", "별도 기획")가 여전히 유효하므로 별도 패키지로 분리 상신한다.

**이 결정에 막히는 항목**: NB-048, NB-050, NB-051 (직접) / NB-049의 "배정 가능(가용)" 필터, NB-035·NB-038의 대기 차주 지표(동일 상태 모델 의존) (파급) / WP-배정패널.

### DEC-007 — 경유지(최대 3개) 보류 해제

## 4. 결정 진행 방법 제안

1. **1순위 gate 결정(DEC-001, DEC-002, DEC-003)** 을 먼저 확정한다 — 이 3건이 풀리면 WP-목록·WP-행확장·WP-폼·WP-추천패널의 착수 차단이 해소된다.
2. DEC-004~007은 scope 결정이므로 권고안 기준으로 가안 확정 후 work package 기획에 반영하고, 이견이 있는 항목만 재론한다.
3. DEC-008~012는 항목이 독립적이라 개별 승인으로 처리 가능하다.
4. 별도 패키지로 분리 상신되는 항목(권고안 채택 시): 실시간 대화방(DEC-005), 차주 추천·위치 인프라(DEC-006), 카드 정산 수단(DEC-009), nbbb1 모바일 전용 규칙(DEC-011).
