## 1. 개요

### 1.1 기준 축과 교차 검증 결과

본 인벤토리의 기준 출처는 **01-기능분석-이벤트정리.md**다. 02(유저플로우)·03(와이어프레임)을 교차 검증한 결과, 01에 없는 신규 기능은 발견되지 않았다 — 03의 주석 40개는 전부 01의 이벤트와 1:1로 대응한다. 따라서 전 항목의 기준 출처는 01이며, 02·03은 보강 출처로 병기한다.

### 1.2 화면 구조 요약 — 콕핏형 2패널

`/dispatches`는 페이지 이동 없이 좌/우 패널의 모드 전환만으로 등록 → 배정 → 관제 → 소통 전 과업을 처리하는 콕핏(cockpit)형 단일 화면이다 (01#1).

| 패널 | 모드 | 해당 섹션 |
|---|---|---|
| 좌측 (약 600px 고정) | 모드 1: 운행 등록 폼 (기본) | 섹션 3 |
| | 모드 2: 차량/차주 배정 패널 | 섹션 6 |
| | 모드 3: 대화방 패널 | 섹션 8 |
| 우측 (유동 폭) | 모드 1: 운행 내역 리스트 (기본) | 섹션 4 |
| | 모드 2: 주요 상하차지 추천 목록 패널 | 섹션 7 |
| | (부가 상태: 전체 접힘 — 플로팅 탭으로 축소) | NB-032 |

즉 **좌측 3모드 × 우측 2모드**(+우측 접힘 상태) 조합이 화면 전체의 상태 공간이며, 모드 간 전환 트리거는 각 항목의 상세에 기술한다.

### 1.3 문서 구성

- 섹션 2~9: 화면/영역별 기능 표 (NB-001 ~ NB-065, 65항목)
- 섹션 10: 불명확 항목 11건 — 구현 기획 시 확정해야 할 질문 목록
- 섹션 11: as-is 한계 — 목업의 미비점으로, 기능 항목이 아닌 구현 시 개선 포인트

## 11. as-is 한계 (01#6 — 구현 시 개선 포인트, 기능 항목 아님)

아래는 분석 대상 목업의 한계로, 목표 기능이 아니라 구현 시 개선해야 할 전제 조건이다.

- **영속화 없음**: 새로고침 시 등록·수정·상태 변경이 모두 소실된다.
- **페이지네이션/가상 스크롤 없음**: 50건 전체를 한 번에 렌더링한다 (NB-037).
- **피드백 일관성 부족**: 운임 수정에는 토스트가 있으나(NB-043), 선택 배정(NB-052)·배차 등록(NB-025)에는 성공 토스트가 없다.
- **alert() 기반 검증**: 필수값 누락 시 인라인 필드 에러 없이 alert로만 안내한다 (NB-025).
- **데이터 공백**: 목업 데이터에 하차완료 상태가 0건이라 해당 상태의 화면 검증이 불가하다.
