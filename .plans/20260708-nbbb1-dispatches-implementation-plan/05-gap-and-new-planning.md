## 1. GAP-001 — 구간(상차→하차 pair) 단위 이력 집계 데이터 소스 부재

- **대상 NB ID**: NB-007 (자주 쓰는 구간 칩), NB-022 (운임 추천 버튼), NB-055 (운행데이터 기반 주요구간 순위 1~5)

### 무엇이 없는가

Phase 3 주소 후보는 전부 side별 단일 주소(saved `/api/addresses` IMPL-D2-014, recent `/api/addresses/recent` IMPL-D2-013) 기준이고, 적용도 side별 독립(IMPL-D2-018 apply, IMPL-D2-004 side 독립성)이다. "상차+하차 pair를 하나의 구간으로 집계·랭킹"하는 데이터 소스와 "pair 동시 입력" 동작이 모두 없다 (M1 NB-007 근거). 구간별 운임 통계(최근/최빈) 소스·API도 없다 (M1 NB-022 근거). 서버측 후보 병합/랭킹 API(`GET /addresses/location-candidates`)는 Phase 3에서 일관 보류 상태다 (IMPL-D2-034 — E1-37/E2-49/P09-26 "client adapter 검증 후 2차 후보", P16b-09 신규 outbound 금지, 재개 기록 없음).

### 신규 기획안

- **기능 요구사항**
  1. 선택 거래처 기준 주요 구간(상차→하차 pair) 빈도 랭킹 Top 5 산출 (NB-055).
  2. 구간 칩 클릭 시 상차지+하차지 동시 draft 적용 (NB-007) — 기존 side별 후보→draft 적용 파이프(IMPL-D2-018)를 pair 동시 apply로 확장.
  3. 구간별 최근 운임/최빈 운임 산출 및 원클릭 입력 (NB-022) — 요율 "산식"이 아닌 자사 과거 오더의 통계 집계임을 기획서에 명시해 운임 산식 금지선(IMPL-D4-057 "pricing source 미확인, distance에서 임의 추론 금지"; IMPL-D4-058 동일 gate 종속)과 경계를 구분한다 (M1 NB-022 비고).
- **데이터 소스/백엔드**: 구간 집계 쿼리 1종 신규 — 자사 오더 이력에서 (거래처, 상차 주소 정규화 키, 하차 주소 정규화 키)로 group by 하여 빈도 순위·최근 운임·최빈 운임을 반환. 상세는 §9 INFRA-1 참조. 집계 scope는 cross-company recent 집계 기각(IMPL-D8-018, IMPL-D1-024 — RSMP-BLOCK-001/004 금지선)을 넘지 않도록 "선택 화주 한정 vs 주선사 자사 오더 전체" 중 결정해야 한다 (M5 NB-065 결정 대기).
- **UI 동작**: ① 등록 폼 상단 인라인 구간 칩 row(클릭 → 상·하차 동시 입력 + 운임 추천 버튼 노출), ② 우측 추천 패널의 "주요구간" 섹션(순위 1~5, `선택` → 동시 적용 — 기존 선택/적용 분리 패턴 IMPL-D2-018 확장), ③ 운임 추천 버튼 클릭 → 최근/최빈 값 중 택일해 운임 필드 입력.

### 선행 조건 (사용자 결정 대기)

1. **NB-022 결정** (M1 userDecision): 통계 기반 운임 추천을 운임 산식 gate(IMPL-D4-057/058 보류)와 별개로 승인해 구현할 것인지.
2. **NB-055 결정** (M3 userDecision #4): 서버측 집계로 구현(IMPL-D2-034 보류 해제 필요)할지, client 측 근사 집계로 시작할지.
3. **NB-065 결정** (M5 userDecision): 집계 scope — 선택 화주 한정 vs 주선사 자사 오더 전체.

### 난이도: **중**

집계 쿼리 1종 + UI 확장으로 수렴하며(M1 관찰 메모 2: "구간 집계 쿼리 하나가 NB-007·NB-022·NB-055를 동시에 해결"), 주소→draft 적용 파이프(IMPL-D2-018)와 칩 UI 패턴은 기존 자산 확장이다. 단 서버 집계 채택 시 backend 신규 API가 필요하다(§10 참조).

## 2. GAP-002 — 경유지(waypoint) 기능: Phase 3 의도적 보류와의 충돌

- **대상 NB ID**: NB-014 (경유지 추가/삭제, 최대 3개)

### 무엇이 없는가

경유지 UI는 Phase 3에서 명시 보류다 — IMPL-D2-008: "F3 Out of Scope, 1차는 상차/하차 2점만, 후순위 route package. 이후 어떤 closeout에도 재개 기록 없음". route draft·주소 dialog가 상차/하차 2점 전제로 설계돼 있어, 경유지 N점 지원은 draft 구조·submit payload·거리 계산 route key까지 확장이 필요하다 (M1 NB-014 근거).

### 신규 기획안

- **기능 요구사항**: 경유지 최대 3개 추가/삭제, 카카오 주소 검색 연동, 등록 시 저장 및 상세 표시. M2 NB-039(행 확장 카드 헤더 `+ 경유지 추가` 버튼 — 수정후이식 판정이나 IMPL-D2-008 충돌 참고 표기)도 이 package에 종속된다.
- **데이터 소스/백엔드**: ① route draft를 2점 고정 → N점 배열 구조로 확장, ② submit payload와 오더 저장 모델에 경유지 저장 구조 신설(오더 스키마 확장 — backend 변경), ③ 거리 계산 route key를 경유지 포함 다점 경로로 확장(서버 chain IMPL-D2-050 `POST /api/distance/calculate` → distance_cache → Kakao directions 의 입력 계약 확장 검토).
- **UI 동작**: 등록 폼 상·하차 사이 경유지 행 추가/삭제(최대 3), 각 경유지에 기존 주소 검색 dialog(IMPL-D2-010, Kakao REST IMPL-D2-012) 재사용, 행 확장 상세 카드에 경유지 표시 + `+ 경유지 추가`.

### 선행 조건 (사용자 결정 대기)

- **NB-014 결정** (M1 userDecision, M2 NB-039와 공통): IMPL-D2-008 보류를 해제하고 route draft/payload/거리 계산 확장을 포함한 "경유지 package"를 기획할 것인지.

### 난이도: **상**

단일 기능이지만 route 도메인의 2점 전제(draft 구조, 주소 dialog, 거리 계산, submit payload, 저장 스키마)를 관통하는 구조 확장이며, Phase 3가 의도적으로 "route package" 규모로 미룬 작업이다 (IMPL-D2-008 보류 사유).

## 3. GAP-003 — 화물실중량: 저장 컬럼 부재로 보류 확정된 필드의 표면화

- **대상 NB ID**: NB-018 (화물실중량 입력)

### 무엇이 없는가

draft 필드(`actualWeightTon`/`actualWeightSource`)는 존재하나 **화면 숨김 + submit payload 제외가 테스트로 고정된 계약**이다 (IMPL-D3-010 완료 — "숨김 유지+draft 보존+submit 제외 자체가 구현된 결정 사항"). 노출·payload write는 사용자 보류 확정 상태다 (IMPL-D3-011 보류 — DR-CARGO-META-001, 32-realignment 2026-07-03; 근본 원인 = 오더 저장 컬럼 부재, P23-10). nbbb1 원본도 단위·검증 규칙이 미기록(⚠)이다 (M1 NB-018).

### 신규 기획안

- **기능 요구사항**: 등록 폼 화물 정보 섹션에 실중량 입력 필드 노출, 단위(톤 전제) 및 검증 규칙(범위·소수 자리) 신규 정의, 오더 저장·상세 표시.
- **데이터 소스/백엔드**: 오더 테이블에 실중량 저장 컬럼 신설(backend 변경 — IMPL-D3-011 보류의 근본 원인 해소). draft·화면 배선은 IMPL-D3-010 자산 위에서 노출 전환 + submit payload 포함으로 변경(고정된 테스트 계약 갱신 필요).
- **UI 동작**: 숫자 입력 필드 1개 + 단위 표기. 기존 화물 섹션 레이아웃에 삽입.

### 선행 조건 (사용자 결정 대기)

- **NB-018 결정** (M1 userDecision): IMPL-D3-011 보류(DR-CARGO-META-001) 해제 + DB 컬럼/단위·검증 규칙 결정.

### 난이도: **하**

M1 판정 그대로 — "보류 해제 시 D3-010 draft 자산 위에서 노출만 켜면 되는 구조라 착수 비용은 낮음". 단 컬럼 신설이 backend 변경이라는 점만 유의(§10).

## 4. GAP-004 — 정산 예정일: 데이터 모델 자체에 없는 필드

- **대상 NB ID**: NB-020 (정산 예정일 퀵버튼 — 미지정/바로/15일/당월말/45일/익월말)

### 무엇이 없는가

정산 예정일(수금 예정일) 필드가 Phase 3 정산 draft/payload 어디에도 없다 — D4 전 항목(A~F절) 확인 결과 결제방법/금액/수수료/메모만 존재한다 (M1 NB-020 근거). 인접 개념인 계산서발행일 등 증빙 read는 별개 필드이며 그 자체도 보류다 (IMPL-D4-053 보류 — "P3에서 만들지 않음, next phase backlog", P25-09). nbbb1 원본도 퀵버튼 산식을 "부근" 표현으로만 남겨 미확정(⚠)이다. M2 NB-039(행 확장 상세 카드)에서도 '정산 예정일' 표시 필드의 데이터 모델 확인/신설이 필요하다고 재확인됐다.

### 신규 기획안

- **기능 요구사항**: ① 정산 예정일 필드 신설(등록 폼 입력 + 행 확장 상세 표시), ② 퀵버튼 6종의 날짜 산식 확정 — 바로/15일/당월말/45일/익월말 각각의 기준일(등록일 vs 하차일)과 말일 처리 규칙 정의, ③ 미지정 허용(nullable).
- **데이터 소스/백엔드**: 오더 또는 정산 스키마에 날짜 필드 1개 신설(backend 변경). 저장 위치(오더 vs 정산 엔티티)는 정산 도메인과 정렬해 결정.
- **UI 동작**: date preset 퀵버튼 UI는 기존 quick datetime preset 패턴(IMPL-D2-040 완료 — 지금/당일/내일 → 실제 date/time 확정, 회귀 테스트 존재)을 이식. 직접 지정은 shadcn Calendar(IMPL-D2-038) 재사용.

### 선행 조건 (사용자 결정 대기)

- 퀵버튼 산식(기준일·말일 규칙) 확정 — nbbb1 원본 미확정(⚠)이므로 기획 단계에서 신규 정의 필요.
- 저장 위치(오더 vs 정산 엔티티) 결정.

### 난이도: **중**

UI 패턴(D2-040/038)은 기존 자산 이식으로 충분하나, 필드·스키마 신설과 산식 신규 정의가 필요하다.

## 5. GAP-005 — AI 기능군: 산출 로직·데이터 모델 전무 + 보류/금지선 다중 충돌

- **대상 NB ID**: NB-035 (AI 배차 난이도 신호등 인디케이터), NB-038 (AI 실시간 배차 분석 어드바이저 — 3단계 배너 + 지표 3종), NB-048 (AI 최적 추천 차주 목록 — 최대 3명 + 추천 사유 태그)

### 무엇이 없는가

- 배차 난이도 산출·표시 자산은 8개 도메인 어디에도 없다. Phase 3의 유일한 AI 계열 자산(IMPL-D8-024 F6 AI Apply)은 문서 추출→폼 반영 기능으로 무관하다 (M2 NB-035 근거).
- 어드바이저 지표 중 '구간 평균 운임(적정/유사/평균 이하 판정)'은 기준금액/운임 산식 source 미확인으로 명시 보류·산식 임의 생성 금지선(IMPL-D4-057, IMPL-D4-058)에 걸린 영역이다. 대기 차주/경쟁 화물 지표도 데이터 소스가 존재하지 않는다 (M2 NB-038 근거).
- AI 추천 차주는 대응 자산이 전부 보류다 — 고급 배차 추천/자동 매칭은 Epic 차원 보류(IMPL-D5-025 — E1-35 비범위, E2-51 deferred roadmap, P13-22 "별도 domain logic 필요"), CargoMan 후보↔내부 DB 자동 매칭도 "Phase 3 완료 후 별도 기획"(IMPL-D5-045 — P02-19~21, 2026-07-07 결정). 추천 사유 태그(동종화물 유경험/상차지 인접/현재 대기중)에 필요한 차주 위치·상태·이력 분석 데이터 모델 자체가 부재하다 (M3 NB-048 근거).
- 세 항목 모두 nbbb1 원본에도 산출 로직이 미기록(⚠)이다 (M2 관찰 메모 3, M3 NB-048 비고).

### 신규 기획안

- **기능 요구사항**

## 6. GAP-006 — 차주 실시간 위치·운행 상태 인프라 부재

- **대상 NB ID**: NB-051 (운행중 차주 "연계지 거리" 대체 로직)

### 무엇이 없는가

차주 현위치·운행 상태·하차 예정지 tracking 자산이 Phase 3에 전무하다. 거리 계산 서버 chain(IMPL-D2-050 완료: `POST /api/distance/calculate` → distance_cache → Kakao directions)은 좌표 2점만 주어지면 재사용 가능하나, 입력이 되는 차주 위치/하차 예정지 데이터 소스 자체가 없다 — 차주 앱 위치 수집 등 인프라가 전제된다 (M3 NB-051 근거). 같은 공백이 M3 수정후이식 항목에도 걸쳐 있다: NB-049 "배정 가능(가용)" 필터·NB-050 현위치 거리/현재 상태 필드 모두 "차주 상태(대기중/운행중) 모델이 없어 불가" (M3 NB-049/050 근거).

### 신규 기획안

- **기능 요구사항**: 운행중 차주는 현위치 대신 "현재 운행의 하차 예정지(연계지) → 대상 오더 상차지" 거리를 표시. 대기중 차주는 최근 위치(또는 위치 미상 표기) 기준.
- **데이터 소스/백엔드**: ① 차주 운행 상태(대기중/운행중) 모델 — 진행 중 배차 존재 여부로 파생 산출하는 1차 근사안과, 차주 앱 실시간 위치 수집 인프라를 전제로 한 완성안의 2단계 제안(§9 INFRA-2). ② 연계지 좌표는 해당 차주의 진행 중 오더 하차지 좌표(기존 오더 데이터)로 조달 가능 — 이 경로만이라면 신규 수집 인프라 없이 파생 계산으로 시작할 수 있다. ③ 거리 계산은 IMPL-D2-050 재사용 (client 직접 Kakao 호출 금지 계약 IMPL-D2-055 준수).
- **UI 동작**: 차주 카드(NB-050) 거리 필드에 "연계지 기준" 라벨 병기, 운행중/대기중 상태 뱃지.

### 선행 조건 (사용자 결정 대기)

- **NB-051 결정** (M3 userDecision #3): 차주 실시간 위치·운행 상태 수집 인프라(차주 앱 연동 등)가 전제 — 1차 범위 포함 여부. (파생 근사안 채택 시에도 상태 모델 정의는 필요.)

### 난이도: **상**

완성형은 차주 앱 위치 수집이라는 외부 인프라를 전제하며(M3 근거), 근사안조차 차주 상태 모델 신설(INFRA-2)이 선행돼야 한다. 거리 계산 chain 재사용은 가능하나 입력 데이터가 전부 신규다.

## 7. GAP-007 — 대화방 도메인 신설 (Phase 3 최초의 신규 서버 도메인)

- **대상 NB ID**: NB-059 (`운행등록으로 복귀` 모드 전환), NB-060 (메시지 입력·전송 + 버블 UI), NB-061 (하단 보조 아이콘 2개 — 문서/즐겨찾기), NB-062 (상대 정보 바 + 시스템 인사 메시지)

### 무엇이 없는가

M4 전수 검토 결론: **"Phase 3에 대화방(채팅) 도메인 자산은 존재하지 않는다."** 인접 자산은 3건뿐 — ① IMPL-D5-067 Legacy LMS 문자(최근 이력/작성 전송/전체 이력, legacy 상세 시트 한정·Phase 3 콘솔 미반영, server validation이 client보다 약함·driver card 메시지 버튼 placeholder toast gap 잔존), ② IMPL-D7-051~053 운영 메모(내부 기록 — 상대방 소통 아님), ③ IMPL-D7-013 콘솔 store(panelOpen/panelMode 패턴). 메시지 저장·실시간 전달·읽음 처리 백엔드는 완전 신규다 (M4 서문·관찰 메모 3).

### 신규 기획안

- **기능 요구사항**
  1. **채널 결정이 최우선** (NB-060 전제 결정, M4): 실시간 양방향 대화방 신규 구축 vs 기존 LMS(IMPL-D5-067 완료 자산) 활용/병행. LMS 활용 시 Phase 3 콘솔 미반영 + gap(약한 server validation, placeholder 버튼) 해소가 별도 과제.
  2. 대화방 패널(NB-060): 메시지 입력·전송, 내 메시지 파랑·우측 정렬 즉시 추가, 타임스탬프. 상대 유형 2종(거래처 — NB-039 진입, 차주 — NB-044 진입) 모두 커버하는 데이터 계약 (M4 NB-062 비고).
  3. 상대 정보 바(NB-062): `{거래처명} | 거래처 | {전화번호}` — 데이터 원천은 기존 snapshot 재사용 (거래처: IMPL-D1-001 화주 정보 섹션, 차주: IMPL-D5-001 차주 정보 섹션, 모두 완료). 시스템 인사 메시지 자동 표시 규칙 신규 정의.
  4. 모드 복귀(NB-059): 좌측 패널 3모드(운행등록↔배정↔대화방) 전환 — IMPL-D7-013 콘솔 store 패턴을 모드 enum에 chat 추가로 확장 (M4 권장 방향). NB-003(콕핏 레이아웃, M5 수정후이식)의 산출에 종속.
  5. 보조 아이콘(NB-061): nbbb1 목업에서도 실동작 미구현·목표 동작 미정의(⚠, 01#2-F/F4). 첨부라면 서버 저장 자산도 없음(IMPL-D8-025 — base64 UI만, API/storage write 없음). **1차 범위 제외를 기본안으로 권고**하고 스펙 정의를 별도 결정으로 남긴다.
- **데이터 소스/백엔드**: 실시간 채팅 채택 시 — 대화방/메시지 엔티티, 저장·전달(polling 또는 실시간 push), 상대 식별(거래처 담당자·차주), 읽음 처리. M4 관찰 메모 3 그대로 "Phase 3의 backend 무변경 원칙 밖의 첫 신규 서버 도메인 — 별도 기획 패키지 규모로 산정 필요".
- **UI 동작**: 좌측 패널 모드 3(대화방) — 헤더(상대 정보 바 + 복귀 버튼), 메시지 버블 리스트, 하단 입력줄 + 전송. route-local design scope(IMPL-D8-002) 내에서 스타일링.

### 선행 조건 (사용자 결정 대기)

1. **NB-060 채널 결정** (M4): 실시간 대화방 신규 구축 vs LMS 활용/병행.
2. **NB-058 범위 결정** (M4, 제외후보): `/chats` 전체 페이지를 신규 기획 범위에 포함할지, 대화방을 콕핏 패널 전용으로 축소할지 — 패널 전용 축소 시 `전체 화면` 버튼 제거.
3. **NB-061 스펙 결정** (M4): 첨부/템플릿 기능을 신규 정의할지 1차 제외할지.

### 난이도: **상** (17건 중 최대 규모)

섹션 전체가 "자산 없음" 도메인이며(M4 관찰 메모 1 — 기각 기록조차 없이 애초 범위 밖), 재사용 가능한 것은 UI 패턴이 아니라 주변 계약(store 패턴 D7-013, 연락처 snapshot D1-001/D5-001)뿐이다. 실시간 채팅 채택 시 메시지 저장·전달·읽음 처리라는 신규 서버 도메인 구축이 필요해 backend 무변경 원칙을 가장 크게 넘어선다.

## 8. GAP-008 — 앱 전역 화면 골격 신규 (사이드바·프로필·패널 접힘)

- **대상 NB ID**: NB-001 (사이드바 내비게이션 70px 7메뉴), NB-002 (사용자 프로필 아바타), NB-032 (우측 패널 접기/펼치기 + 플로팅 세로 탭)

- **데이터 소스/백엔드**: 없음 — 3건 모두 frontend 전용. 프로필 표시 데이터는 기존 auth context 재사용.

### 선행 조건 (사용자 결정 대기)

1. **NB-001 결정** (M5 userDecision): 사이드바 7메뉴를 이번 범위에 포함할지, 기존 mm-broker 내비게이션 및 운영 route 승격 보류(IMPL-D7-044)와의 관계.

frontend 전용이라 기술 리스크는 낮으나(NB-002·NB-032 개별로는 하 수준), 사이드바는 운영 IA 미확정(운영 route 승격 보류)이라는 조직 결정에 걸려 있고, 패널 접힘은 콕핏 store 2축 확장(M5 관찰 메모 2 — drawer 전제 계약 재검증 필수)과 묶여 있다.

여러 gap이 같은 기반을 공유한다. gap별 개별 구현 대신 아래 인프라를 선행 확보하면 중복이 제거된다.

### INFRA-1 — 구간(pair) 집계 쿼리 1종

| 항목 | 내용 |
|---|---|
| 수혜 항목 | GAP-001 전체(NB-007 구간 칩, NB-022 운임 추천, NB-055 주요구간 순위), GAP-005의 구간 평균 운임 지표(NB-038), M5 NB-065 입력 추천 체인(수정후이식)의 신규 요소 |

## 10. Phase 3 "backend 무변경 원칙" 초과 항목 (명시 표기)

Phase 3는 기존 서버 재사용을 원칙으로 했다(M4 관찰 메모 3 — "F7 memo, CargoMan 연동 모두 기존 서버 재사용"). 아래는 이 원칙을 넘어서는 gap이다.

| GAP | 초과 내용 | 회피/축소 옵션 |
|---|---|---|
| GAP-001 | 구간 집계 API 신설 (서버측 집계 채택 시 — IMPL-D2-034 보류 해제 필요) | client 측 근사 집계로 시작 가능 (M3 userDecision #4의 대안) — 단 집계 품질·성능 제약 |
| GAP-002 | 경유지 저장을 위한 오더/route 스키마 확장 + submit payload 확장 | 없음 — 저장하지 않는 경유지는 기능 목적(상세 표시·거리 계산) 불충족 |
| GAP-003 | 오더 테이블 실중량 저장 컬럼 신설 (IMPL-D3-011 보류의 근본 원인) | 없음 — 컬럼 부재가 보류 사유 그 자체 |
| GAP-004 | 정산 예정일 필드의 오더/정산 스키마 신설 | 없음 — 필드 자체가 부재 |
| GAP-005 | 지표 집계(대기 차주·경쟁 화물·구간 운임) 및 추천 점수 산출의 서버 구현 채택 시 | 경쟁 화물·이력 기반 사유는 기존 오더 데이터의 client 집계로 근사 시작 가능 |
| GAP-006 | 차주 위치 수집 인프라(차주 앱 연동) — 완성형 채택 시 | 1단계 파생 근사(진행 중 오더 하차지 좌표)는 기존 데이터로 가능 (§9 INFRA-2) |
| GAP-007 | 채팅 백엔드(대화방/메시지 엔티티, 전달, 읽음 처리) — **초과 폭 최대** | LMS(IMPL-D5-067) 활용/병행 채택 시 신설 회피·축소 가능 (NB-060 채널 결정에 종속) |
| GAP-008 | 해당 없음 — frontend 전용 | — |

## 11. 요약

- gap 그룹 8개 (GAP-001~GAP-008) / 신규기획 17건 전수 배정.
- 난이도 분포: 상 4건(GAP-002 경유지, GAP-005 AI 기능군, GAP-006 차주 위치·상태, GAP-007 대화방), 중 3건(GAP-001 구간 집계, GAP-004 정산 예정일, GAP-008 화면 골격), 하 1건(GAP-003 실중량).
- 최대 규모는 **GAP-007 대화방 도메인** — Phase 3에 자산이 전무한 유일한 도메인이자 backend 무변경 원칙을 넘어서는 첫 신규 서버 도메인 후보(M4 관찰 메모 3).
- 공통 인프라 4종(구간 집계 쿼리, 차주 상태·위치 모델, 채팅 백엔드, 패널 store 2축 확장) 선행 확보 시 gap 8개 중 6개가 수혜.
- 착수 전 사용자 결정 대기: GAP별 선행 조건 절에 명시 — 특히 Phase 3 보류·기각과 충돌하는 결정(NB-014 경유지, NB-018 실중량, NB-022 운임 추천, NB-038 지표 축소, NB-048 추천 착수, NB-055 집계 방식, NB-060 채널, NB-001 사이드바 범위)이 8건이다.
