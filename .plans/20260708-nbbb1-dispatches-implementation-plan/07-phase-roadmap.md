# 07. Phase 로드맵 — nbbb1 /dispatches 구현 순서

- 작성일: 2026-07-09
- 목적: 06 문서의 work package 25개(WP-001~WP-025)를 의존 관계·사용자 결정(DEC) 종속에 따라 Phase로 묶고, Phase 내 병렬 실행 축과 Phase별 마일스톤(사용자 확인 가능 동작 상태)을 정의한다.
- 입력 소스:
  - `06-work-packages.md` (WP 정의·선행 조건·DEC/GAP 참조)
  - `_working/mapping/M1.md` ~ `M5.md` (분류·의존 근거)

  4. Phase 내 WP는 의존 사슬이 없는 것끼리 병렬 축으로 명시한다.

## 2. Phase 상세

### Phase 0 — 결정 수렴·콕핏 기반

| 작업 | 내용 | 병렬 |
|---|---|---|
| DEC 결정 세션 | 06 문서 §2의 14개 DEC 그룹 일괄 상신·결정 (04 문서 확정). 특히 Phase 2를 여는 DEC-목록재설계-게이트·DEC-상태전환-범위와, Phase 3 기획 선행이 필요한 DEC-AI지표·DEC-차주추천은 이 시점 결정이 필수 | A축 |
| WP-001 콕핏 2패널 셸·store 2축 확장 | 결정 불요. 전 화면 WP의 공통 기반 (NB-003, NB-032) | B축 |
| GAP-목록API-안정성 런타임 확인 | IMPL-D7-066/D5-026/D8-029의 500/0건 이슈 재현·수정 — Phase 2 목록 축(WP-011/012)의 전제 (M2 관찰 메모 2) | C축 |

- 병렬 가능: A·B·C 3축 완전 병렬 (상호 의존 없음).
- **마일스톤**: 페이지 이동 없이 좌측 고정 패널(등록 폼 자리) + 우측 기존 목록이 병렬 표시되고, 패널 모드 전환·접힘·dirty guard가 동작하는 콕핏 셸을 사용자가 확인할 수 있다.

### Phase 1 — 등록 코어 이식 (결정 불요)

선행: WP-001. DEC가 전부 미결이어도 착수 가능한 WP만 배치.

| 병렬 축 | WP (실행 순서) | 비고 |
|---|---|---|
| A. 등록 폼 | WP-003(거래처) → WP-004(주소) → WP-005(일시) ∥ WP-006(차량·화물) → WP-008(제출·초기화) | WP-004는 WP-003의 화주 컨텍스트 종속. WP-005/006은 상호 독립 — WP-004와도 병렬 가능. WP-008은 폼 입력부(003~006) 완성 후. GAP-담당자-서버검증·GAP-recent-invalidate 해소를 WP-008에 포함 |
| B. 추천 패널 | WP-018(주요 상하차지 추천 패널) | WP-003의 진입점(NB-006/008) 완성 후 착수 — A축 초반과 겹쳐 병렬 진행 가능 |

- 정산 방법(WP-007)은 DEC-정산-4옵션 종속이라 Phase 2로 미룬다. Phase 1 폼은 기존 2옵션 selector(IMPL-D4-009)를 임시 배치해 제출 파이프를 막지 않는다.
- **마일스톤**: 좌측 상시 폼에서 거래처 칩→주소 칩/검색→일시 퀵버튼→차량·품목 칩으로 오더를 입력하고 `배차 등록`으로 저장하면 우측 목록에 반영·폼이 초기화되는 등록 코어 한 바퀴를 사용자가 확인할 수 있다.

### Phase 2 — 목록·상세·배정 (DEC 개방 후)

선행: Phase 0의 해당 DEC 결정 + GAP-목록API-안정성 해소.

| 병렬 축 | WP (실행 순서) | 여는 결정 |
|---|---|---|
| A. 목록 재설계 | WP-012(테이블·검색) ∥ WP-011(필터 칩·교차 카운트) → WP-013(행 아코디언·상세 카드) → WP-014(운임 인라인 편집) ∥ WP-015(상태 모델·전환 버튼) | WP-011/012: DEC-목록재설계-게이트. WP-015: DEC-상태전환-범위 (건수 재계산 연동으로 WP-011에도 종속) |
| B. 배정 | WP-017(배정 패널 UI) → WP-016(차주 배정 실행) | WP-017: DEC-차주카드-데이터. WP-016: DEC-수동배정-원칙. WP-017의 진입 트리거(NB-045)가 WP-013 소관이라 A축 WP-013과 접점 — 트리거 wiring만 WP-013 완료 대기 |
| C. 폼 확장 | WP-007(정산 토글) ∥ WP-009(실중량·정산 예정일) | WP-007: DEC-정산-4옵션. WP-009: DEC-화물실중량 + 정산 예정일 스키마 결정 |
| D. 전역 셸 | WP-002(사이드바·프로필) | DEC-사이드바-범위 |

- 병렬 가능: A~D 4축 병렬. A축 내부는 WP-011∥WP-012 병렬 후 WP-013이 합류 지점, WP-014∥WP-015 병렬.
- WP-013의 상세 카드 중 정산 예정일 표시는 C축 WP-009 완료 후 활성(그 전에는 미표시), `+ 경유지 추가`·`대화방` 버튼은 각각 Phase 3 WP-010·Phase 4 WP-023 완료 전까지 비활성 placeholder (06 문서 WP-013 선행 조건).
- **마일스톤**: 우측 목록에서 날짜×상태 칩(건수 교차 재계산)으로 필터하고, 행을 클릭해 아코디언에서 상세 확인→운임 스테퍼 수정(목록 즉시 갱신)→차량배정 패널에서 차주 선택 배정(배차중→배차완료)→상태 버튼 전환까지, 등록 이후 전 과업이 한 화면에서 도는 것을 사용자가 확인할 수 있다.

### Phase 3 — 집계·지능 기능

선행: Phase 1 완료(폼·추천 패널 컨테이너), Phase 2의 WP-012/013/015(AI 표시 위치·상태 selector).

| 병렬 축 | WP (실행 순서) | 여는 결정 |

### Phase 4 — 신규 도메인·반응형

선행: Phase 2 완료(진입점·재배열 대상 코어).

## 5. 리스크 메모

1. **Phase 2 병목**: DEC-목록재설계-게이트와 GAP-목록API-안정성이 동시에 걸린 목록 축(WP-011/012)이 Phase 2의 임계 경로다. Phase 0에서 이 둘을 최우선 처리하지 않으면 아코디언(WP-013) 이후 전 사슬이 밀린다.
2. **WP-001 재작업 리스크**: 콕핏 전환은 drawer 전제 계약(IMPL-D7-016/018/019, IMPL-D6-006 wizard 상태)의 재검증이 필수(M5 관찰 메모 2) — Phase 0에서 store 축 확장을 끝내지 못하면 Phase 1 폼 WP 전체가 임시 컨테이너 위에서 작업하게 되어 재작업이 발생한다.
3. **Phase 3 데이터 소스 수렴**: 구간(pair) 집계 하나가 NB-007/022/055를 동시에 해결(M1 관찰 메모 2)하므로 WP-019의 집계 설계 품질이 Phase 3 전체의 상한이다. DEC-구간랭킹-집계방식에서 서버/client 방식을 정할 때 WP-020 운임 통계까지 담을 수 있는 구조로 결정해야 이중 작업을 피한다.
4. **금지선 준수**: 운임 관련 신규 기능(WP-020/021)은 운임 산식 임의 생성 금지선(IMPL-D4-057/058)과 cross-company 집계 금지선(IMPL-D8-018, RSMP-BLOCK-004)의 경계 안에서만 설계한다 — 각 WP 기획서에 경계 명시를 완료 기준으로 포함했다(06 문서).
