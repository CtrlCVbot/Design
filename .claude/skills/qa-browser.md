# QA Browser Testing Skill

## 적용 조건

- `/qa` 명령 실행 시
- MCP 브라우저로 탐색적 테스트 수행 시

---

## 0단계: 영향 분석 및 지식 로드 (필수)

QA 시작 전 반드시 수행:

### 0-1. 기존 지식 로드

```
.claude/knowledge/feature-impact-map.md
.claude/knowledge/qa-edge-cases.md
```

### 0-2. 변경 파일 기반 영향 분석

1. 변경된 파일 패턴 확인
2. `feature-impact-map.md`에서 해당 기능 영역 찾기
3. "회귀 테스트 필요 페이지" 추출

```
예시:
- 변경 파일: server/logishm-integration/**
- → "화물맨 연동" 섹션 찾기
- → 회귀 테스트: 화물현황 상세 (공유 상태 표시 확인)
```

### 0-3. 엣지케이스 로드

`qa-edge-cases.md`에서 테스트 대상 페이지의 알려진 엣지케이스 확인:
- `open` 상태: 재현 테스트 수행
- `fixed` 상태: 회귀 테스트 수행

### 0-4. 통합 정합성 검증

`.claude/knowledge/integration-coherence-checklist.md` 읽고 경계면 교차 비교 수행:
- API 응답 shape ↔ 프론트 훅 타입 일치 여부
- 라우팅 경로 ↔ href/router.push 매핑
- 도메인 상태 전이 ↔ 실제 status 업데이트 코드

### 0-4. 영향 분석 결과 출력

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🎯 영향 분석 결과
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

변경된 기능 영역: [기능명]

📍 회귀 테스트 필요 페이지:
  1. /path - 확인 포인트
  2. /path - 확인 포인트

📋 관련 엣지케이스:
  - [EDGE-XXX] 설명...
  - [EDGE-YYY] 설명...
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 테스트 관점 (필수)

| 관점 | 테스트 예시 |
|------|-------------|
| **경계값** | 0, null, undefined, 빈 문자열, 최대 길이, 음수 |
| **상태 전이 오류** | 이미 처리된 건 다시 처리, 잘못된 순서로 진행 |
| **권한/역할 불일치** | 다른 사용자 데이터 접근, 권한 없는 액션 시도 |
| **동시성** | 빠른 중복 클릭, 동시 요청, 탭 여러 개 |
| **실패 후 복구** | 에러 발생 후 재시도, 새로고침 후 상태 |
| **UI/서버 상태 불일치** | 화면은 성공인데 서버는 실패, 낙관적 업데이트 오류 |

## 수행 규칙

### 1. 속도 최적화 (필수)

**1-1. `browser_fill_form` 활용 - 여러 필드 한 번에:**

```
❌ 느림: type → type → type (3번 호출)
✅ 빠름: browser_fill_form으로 1번 호출
```

**1-2. `browser_run_code` 활용 - 복잡한 작업 한 번에:**

```javascript
// 여러 액션을 한 번의 호출로 실행
async (page) => {
  await page.fill('#email', 'test@test.com');
  await page.fill('#password', '1234');
  await page.click('button[type="submit"]');
  await page.waitForURL('**/dashboard');
}
```

**1-3. snapshot 호출 최소화:**

```
❌ 느림: click → snapshot → click → snapshot → click → snapshot
✅ 빠름: click → click → click → snapshot (결과만 확인)
```

**1-4. 명시적 대기 사용:**

```
❌ 느림: waitForTimeout(3000) - 고정 대기
✅ 빠름: waitForResponse('**/api/orders') - 조건 대기
✅ 빠름: waitFor({ text: '로딩 완료' }) - 텍스트 대기
```

### 2. 스크린샷 촬영 금지

```
토큰 절약을 위해 스크린샷은 촬영하지 않는다.
browser_snapshot으로 접근성 트리만 확인한다.
```

### 2. 발견 사항 보고 형식

각 발견 사항마다 반드시 포함:

```
🐛 [버그/엣지케이스 제목]

📍 재현 경로:
  1. xxx 페이지 이동
  2. xxx 클릭
  3. xxx 입력

🔍 현상:
  - 실제 동작: ...
  - 기대 동작: ...

⚠️ 왜 사람이 놓치기 쉬운가:
  - ...

💥 실제 발생 시 장애:
  - ...

🤖 자동화 가능 여부:
  - [ ] E2E 테스트로 커버 가능
  - [ ] Unit/Integration으로 커버 가능
  - [ ] 수동 QA 필요
```

### 3. 테스트 우선순위

1. **변경된 기능** - 이번 커밋에서 수정된 부분
2. **핵심 비즈니스 로직** - 돈, 정산, 상태 전이
3. **사용자 입력 처리** - 필터, 폼, 검색
4. **에러 처리** - 네트워크 실패, 잘못된 입력

## 테스트 시나리오 예시

### 경계값 테스트

```
- 필터에 빈 문자열 입력
- 검색어에 특수문자 입력: '; DROP TABLE; --
- 금액에 0, 음수, 매우 큰 숫자 입력
- 날짜 필터에 미래 날짜, 과거 날짜
```

### 상태 전이 테스트

```
- 이미 정산 완료된 건 다시 정산 시도
- 취소된 주문에서 정산 시도
- 진행 중인 작업 중복 실행
```

### 동시성 테스트

```
- 저장 버튼 빠르게 2번 클릭
- 같은 페이지 탭 2개 열고 동시 작업
- 필터 빠르게 여러 번 변경 (debounce 테스트)
```

### 실패 후 복구 테스트

```
- API 에러 발생 후 재시도
- 네트워크 끊김 후 복구
- 에러 토스트 닫고 다시 시도
```

```
  - 필터 적용 → 목록 갱신
  - 페이지네이션 동작
  - ...

🐛 발견된 이슈 (N건):

[1] 빈 문자열 필터 시 전체 조회되지 않음
    📍 재현: 거래처 필터 → 빈 문자열 입력 → 적용

    🤖 자동화: E2E 테스트 추가 권장

[2] 빠른 더블클릭 시 중복 요청
    📍 재현: 정산 버튼 빠르게 2번 클릭
    ⚠️ 놓치기 쉬운 이유: 테스터는 천천히 클릭
    💥 장애: 중복 정산 발생 가능
    🤖 자동화: 프론트 디바운스 + 백엔드 멱등성 체크

💡 E2E 스크립트 제안:
  - edge-case: 빈 필터 테스트 추가
  - edge-case: 중복 클릭 방지 테스트 추가
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
