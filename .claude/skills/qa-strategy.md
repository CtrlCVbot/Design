# QA Strategy Skill

## 🚨 BLOCKING: 컨텍스트 파일 먼저 읽을 것

```
이 스킬 적용 전 반드시 읽어야 함:
→ dev/active/qa-strategy/context.md

이 파일을 읽지 않고 QA 관련 작업 진행 금지.
결정 배경, 상세 전략, 코드 예시가 모두 해당 파일에 있음.
```

## 적용 조건

- QA, 테스트 전략 논의 시
- 체크리스트 관련 작업 시
- E2E 테스트 작성 시

> **테스트 코드 작성 규칙**은 `.claude/skills/testing.md` 참조

---

## 핵심 원칙

```
개발자 (+ AI): 기능이 명세대로 동작하는가? → 자동화 테스트
비전문가 QA: 사용자 입장에서 이상한가? → 탐색적 테스팅 (5-7개)
```

---

## 테스트 구조

```
CI (PR마다 실행):
├── 백엔드: Unit + Integration
├── 프론트: Unit + Integration
└── E2E: 핵심 플로우 3-5개

별도 (Nightly/배포 전 로컬에서):
└── AI 탐색적 테스트 → 발견 시 E2E에 추가

수동 QA:
└── 5-7개 질문 (UX, 직관성)
```

---

## E2E 테스트

### 대상 (3-5개)
1. 로그인 → 주문 목록 조회
2. 주문 상세 → 정산 처리
3. 필터 적용 → 결과 확인
4. 에러 발생 시 메시지 표시
5. (선택) 모바일 반응형

### 참고 예시
- `e2e/settlement-flow.spec.ts`

---

## PR 체크 원칙

```
PR 체크 = 결정적 (같은 코드 → 항상 같은 결과)
AI 창의적 탐색 = 비결정적 → PR과 분리
```

- PR: Unit + Integration + E2E 스크립트
- AI 탐색: Nightly 또는 배포 전 (PR 블로킹 X)

---

## QA 체크리스트

### 자동화 대상 (테스트 코드)

- 로직 검증, 계산, API 연동, 필터, 상태 관리

### 수동 QA (5-7개)

```
□ 처음 쓰는 사람 입장에서 헷갈리는 거 있나요?
□ 에러 메시지 이해되나요?
□ "어? 이거 왜 이래?" 싶은 거 있나요?
□ 불편한 점 있나요?
□ 모바일에서 문제 있나요?
```

---

## AI 역할

| 역할 | 타이밍 |
|-----|-------|
| 테스트 코드 작성 | 개발 시 |
| 탐색적 테스트 | Nightly/배포 전 |
| 발견 → 스크립트화 | 발견 후 |

---

## 실행 로드맵

- Phase 1: 프론트 Unit/Integration + CI
- Phase 2: E2E 3-5개 + CI
- Phase 3: AI 탐색 + QA 가이드 변경
