# QA 도메인 지식 저장소

> QA 세션에서 축적된 지식을 관리하는 디렉토리

## 목적

1. **회귀 테스트 자동 판단**: 변경된 파일 → 관련 페이지 자동 추출
2. **엣지케이스 축적**: 세션마다 발견한 케이스 축적으로 꼼꼼한 QA 수행

---

## 파일 구조

```
.claude/knowledge/
├── feature-impact-map.md                  # 기능 간 영향 관계 맵 (회귀 테스트용)
├── qa-edge-cases.md                       # 발견된 엣지케이스 목록
├── integration-coherence-checklist.md     # 통합 정합성 검증 체크리스트 (경계면 교차 비교)
└── README.md                              # 이 파일
```

---

## 사용 방법

### QA 시작 시 (0단계)

1. 변경된 파일 확인
2. `feature-impact-map.md`에서 해당 기능 영역 찾기
3. "회귀 테스트 필요 페이지" 확인
4. `qa-edge-cases.md`에서 해당 페이지의 알려진 엣지케이스 확인
5. `integration-coherence-checklist.md`로 경계면 교차 비교 수행

### QA 종료 시 (6단계 - 인계)

1. 새로 발견한 영향 관계 → `feature-impact-map.md`에 추가
2. 새로 발견한 엣지케이스 → `qa-edge-cases.md`에 추가

---

## 축적 기준

### 저장해야 할 것

- **기능 간 영향 관계**: "A 건드리니 B가 깨졌다"
- **재현 가능한 엣지케이스**: 동시성, 경계값, 상태 전이 등
- **비즈니스 규칙**: 상태 변경 조건, 권한 규칙 등

### 저장하지 말 것

- **일시적 버그**: 환경 특정, 재현 불가
- **테스트 데이터 특정**: 특정 ID에서만 발생
- **추측/가설**: 검증되지 않은 내용

---

## 지식 생명주기

### 엣지케이스 상태

- `open`: 아직 수정되지 않음 → 재현 테스트 필요
- `fixed`: 수정됨 → 회귀 테스트 필요
- `wontfix`: 수정하지 않기로 결정됨

### 업데이트 시점

- QA 세션 종료 시 인계 단계에서 업데이트
- 버그 수정 후 상태 변경 (open → fixed)
- 더 이상 재현 안 되면 상태 변경 또는 삭제

## 기여 가이드

### 엣지케이스 추가 템플릿

```markdown
#### [EDGE-XXX] 제목

- **발견일**: YYYY-MM-DD
- **심각도**: critical | high | medium | low
- **상태**: open
- **재현**:
  1. 단계 1
  2. 단계 2
- **현상**: 관찰된 동작
- **확인점**: 기대 동작
```

### 영향 관계 추가 템플릿

`feature-impact-map.md`의 해당 기능 영역에 추가:

```markdown
### 과거 회귀 이슈

- YYYY-MM-DD: [변경 내용] → [영향받은 페이지와 현상]
```
