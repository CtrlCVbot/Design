# /fix-pipeline - 버그 수정 전체 파이프라인

Notion 버그 리포트 확인 → 원인 분석 → 테스트 작성 → 수정 → SOLID 순차 리뷰 → 검증 → QA → 알림까지 자동화합니다.

---

## 사용법

```
/fix-pipeline https://www.notion.so/logishm/... fix-name
```

- 첫 번째 인자: Notion 버그 리포트 링크 (MCP 브라우저로 열림)
- 두 번째 인자: 수정 이름 (브랜치/worktree 이름에 사용, kebab-case)

---

## 상세 단계

### STEP 0: Git Worktree 생성 + Notion 버그 리포트 확인

**0-1. Git Worktree 생성:**

```bash
# 1. 변수 설정
FIX_NAME="{두 번째 인자}"
BRANCH_NAME="fix/$FIX_NAME"
WORKTREE_PATH="C:/Users/USER/Desktop/clone/mm-broker-worktrees/fix-$FIX_NAME"

# 2. Worktree 생성 (main 기준 새 브랜치)
git worktree add -b "$BRANCH_NAME" "$WORKTREE_PATH" main

# 3. 이후 모든 작업은 worktree 경로에서 수행
cd "$WORKTREE_PATH"
```

**주의사항:**
- 이미 해당 worktree가 있으면 `cd`만 수행
- 이미 해당 브랜치가 있으면 `git worktree add "$WORKTREE_PATH" "$BRANCH_NAME"`
- **STEP 1~7의 모든 파일 읽기/쓰기/빌드/테스트는 worktree 경로 기준**
- `dev/active/fix-{name}/` 산출물도 worktree 내에 생성

**0-2. Notion 버그 리포트 확인:**

1. MCP 브라우저로 Notion 링크 열기
2. 로그인 필요 시 `<NOTION_ACCOUNT>` 계정 사용
3. 버그 리포트 내용을 접근성 스냅샷으로 파악
4. 버그 요약 정리:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🐛 버그 리포트 요약
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📌 제목: {버그 제목}
📍 발생 위치: {페이지/기능}
🔄 재현 시나리오:
  1. ...
  2. ...
  3. ...
⚡ 기대 동작: ...
❌ 실제 동작: ...
📎 참고: {추가 정보}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

5. `dev/active/fix-{name}/bug-report.md`에 저장

**완료 조건**: 버그 재현 시나리오 명확히 정리됨

### STEP 1: 원인 분석

Notion에서 파악한 버그 정보를 바탕으로 코드를 탐색합니다.

**1-1. 관련 코드 탐색:**

- 버그 발생 페이지의 API Route 파일 찾기
- Route → Service → Repository → Domain 추적
- 관련 프론트엔드 컴포넌트 확인

**1-2. 근본 원인 파악:**

- 버그의 직접 원인 (어떤 코드가 잘못됐는지)
- 근본 원인 (왜 그 코드가 잘못됐는지)
- 영향 범위 (같은 패턴이 다른 곳에도 있는지)

**1-3. 분석 결과 기록:**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔍 원인 분석 결과
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📁 관련 파일:
  - {file1} (직접 원인)
  - {file2} (영향 범위)

🎯 직접 원인: ...
🌱 근본 원인: ...
💥 영향 범위: ...

📋 수정 계획:
  1. ...
  2. ...
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

`dev/active/fix-{name}/analysis.md`에 저장

**완료 조건**: 근본 원인 파악 + 수정 계획 수립

### STEP 2: 버그 재현 테스트 작성

수정 전에 **실패하는 테스트**를 먼저 작성합니다.

**테스트 작성 우선순위:**

1. **Domain 단위 테스트** - 비즈니스 규칙 위반이 원인인 경우
2. **Service 단위 테스트** - 오케스트레이션 로직이 원인인 경우
3. **Route 통합 테스트** - API 레벨 동작이 원인인 경우

> 테스트 작성 시 `.claude/skills/testing.md` 참조

**실행 및 확인:**

```bash
# 재현 테스트 실행 - 실패해야 정상 (Red)
pnpm test -- --testPathPattern "{test-file}"
```

**완료 조건**: 버그를 재현하는 테스트가 **실패** 상태

### STEP 3: 수정 코드 작성

테스트를 통과시키는 최소한의 수정을 합니다.

**수정 원칙:**

- **최소 수정**: 버그를 고치는 데 필요한 변경만
- **사이드이펙트 방지**: 기존 동작을 깨뜨리지 않는지 확인
- **레이어 규칙 준수**: Route/Service/Repository/Domain 경계 유지

**실행 및 확인:**

```bash
# 1. 재현 테스트 통과 확인 (Green)
pnpm test -- --testPathPattern "{test-file}"

# 2. 관련 테스트 전체 통과 확인
pnpm test -- --testPathPattern "{related-pattern}"

# 3. 빌드 확인
pnpm build
```

**완료 조건**: 재현 테스트 통과 (Green) + 관련 테스트 통과 + 빌드 성공

### STEP 4: SOLID 순차 리뷰 + 권장사항 적용

5개 SOLID 원칙을 **순차적으로** 리뷰하고, 각 리뷰의 권장사항을 즉시 반영합니다.

**순서:**

```
SRP → OCP → LSP → ISP → DIP
```

**각 원칙별 절차 (5회 반복):**

1. 해당 원칙의 리뷰 커맨드 실행 (`/srp-review` → `/ocp-review` → `/lsp-review` → `/isp-review` → `/dip-review`)
2. ❌ 위반 사항 즉시 수정
3. ⚠️ 개선 사항 판단 후 적용
4. 다음 원칙으로 진행

**중요:**
- 각 원칙 리뷰는 **이전 원칙의 수정이 반영된 상태**에서 수행
- 5개 원칙 모두 완료 후 다음 STEP 진행
- 빌드/테스트는 이 단계에서 수행하지 않음 (최종 검증 단계에서 일괄 수행)

**완료 조건**: 5개 원칙 리뷰 완료 + 권장사항 반영 완료

### STEP 5: /verify-all + 권장사항 적용

`.claude/commands/verify-all.md`를 **그대로** 수행. ❌ 위반은 즉시 수정, ⚠️ 개선은 판단 후 적용. 수정 후 재빌드 + 재테스트 필수 (최대 3회).

### STEP 6: /qa 수정 확인 + 권장사항 적용

`.claude/commands/qa.md`를 **그대로** 수행하되, 다음에 집중:

1. **수정 확인**: Notion 재현 시나리오를 브라우저에서 직접 수행
2. **사이드이펙트**: 수정이 주변 기능을 깨뜨리지 않았는지 확인
3. **엣지케이스**: 유사 상황에서 같은 버그가 재발하지 않는지

버그 발견 시 즉시 수정 + 재QA (최대 3회). E2E 스크립트/knowledge 인계는 QA 프로세스 내에서 처리.

### STEP 7: 최종 검증 + 알림

**7-1. 최종 검증 수행:**

```bash
# 빌드 확인
pnpm build

# 전체 테스트 실행
pnpm test
```

**7-2. 파이프라인 결과 리포트 출력:**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🏁 FIX PIPELINE COMPLETE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🐛 버그: {버그 제목}
📎 Notion: {링크}
🌿 브랜치: fix/{name}
📂 Worktree: mm-broker-worktrees/fix-{name}

📊 단계별 결과:
  STEP 0 버그파악:    ✅/❌
  STEP 1 원인분석:    ✅/❌
  STEP 2 재현테스트:  ✅/❌
  STEP 3 수정:       ✅/❌
  STEP 4 SOLID 리뷰: ✅/❌ (SRP N건, OCP N건, LSP N건, ISP N건, DIP N건 적용)
  STEP 5 verify-all: ✅/❌ (권장 N건 적용)
  STEP 6 QA:         ✅/❌ (권장 N건 적용)
  STEP 7 최종검증:   ✅/❌

🎯 근본 원인: {한 줄 요약}
📁 수정 파일:
  - {file1}
  - {file2}

📁 산출물:
  - dev/active/fix-{name}/bug-report.md
  - dev/active/fix-{name}/analysis.md
  - dev/active/fix-{name}/pipeline-report.md

💡 다음 단계:
  - worktree에서 /commit 으로 커밋 생성
  - main에서 git merge fix/{name} 또는 PR 생성
  - 완료 후: git worktree remove mm-broker-worktrees/fix-{name}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**7-3. 파이프라인 리포트 저장:**

`dev/active/fix-{name}/pipeline-report.md`에 위 결과를 저장

## 중단 정책

각 STEP에서 치명적 실패 시:

| 실패 유형 | 행동 |
|-----------|------|
| Notion 접근 실패 | 사용자에게 URL 확인 요청 |
| 원인 파악 불가 (30분 이상) | 파이프라인 중단 + 현재까지 분석 결과 리포트 + 알림 |
| 빌드 실패 (3회 재시도 후) | 파이프라인 중단 + 현재까지 결과 리포트 + 알림 |
| 테스트 실패 (3회 재시도 후) | 파이프라인 중단 + 현재까지 결과 리포트 + 알림 |
| QA 크리티컬 버그 (3회 수정 후) | 파이프라인 중단 + 이슈 기록 + 알림 |

중단 시에도 **반드시 알림**을 보냅니다.
